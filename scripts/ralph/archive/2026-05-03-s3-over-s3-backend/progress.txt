  three-shape semantics ('' / 'null' / <uuid>) — US-007 of THIS cycle

- meta.Store interface is the contract; THREE backends update in
  lockstep: memory + cassandra + tikv. internal/meta/storetest/contract.go
  is the parity oracle — every meta.Store change must keep the
  contract suite green against all three.
- Cassandra LWT lesson (CLAUDE.md "Cassandra gotchas"): any UPDATE on
  a row that may be read with quorum AFTER a previous LWT must itself
  be LWT (IF EXISTS). The TiKV equivalent is "any update needing
  read-after-write coherence must use a pessimistic txn, not plain
  Put". US-007 (versioning Suspend → null overwrite) and US-009
  (multipart Resend-first-finishes-last) both touch this — wire LWT /
  pessimistic-txn paths in lockstep.
- Per-part composite-checksum response shape (US-003) follows AWS
  formula: concatenate raw part-digests (binary, not hex), SHA-256
  the concatenation, base64-encode. The composite is reported in
  CompleteMultipartUploadResult AND echoed on HEAD with
  ChecksumMode=ENABLED.
- Per-part offset tracking (US-001 PartChunks) is populated at
  CompleteMultipartUpload time from multipart_uploads.parts rows.
  Per-part checksum (added in US-004) MUST be stored at UploadPart
  time, not recomputed at Complete — the body bytes have already
  streamed past.
- BackendETag column on multipart_uploads.parts already exists from
  s3-over-s3 cycle US-010 (s3-backend-side ETag). Don't conflict
  with US-004's `checksum_value` column — they coexist (ETag is
  always set; ChecksumValue is set only when client sent
  x-amz-checksum-algorithm).
- ?partNumber=N GET (US-002) is a NEW HTTP route shape:
  internal/s3api/server.go currently routes ?partNumber=N + uploadId
  to UploadPart (in-progress multipart). Without uploadId,
  ?partNumber=N must dispatch to a NEW handler in object_get.go that
  uses the manifest's PartChunks to serve part-relative bytes.
