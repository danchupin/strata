## Codebase Patterns
- `meta.Store` interface lives in `internal/meta/store.go`. Adding methods requires implementations on all THREE backends (memory, cassandra, tikv) + a `case<Name>` in `internal/meta/storetest/contract.go`. Well-established pattern from bucket-quotas-billing cycle (BucketQuota CRUD across 3 backends).
- Cassandra schema migrations additive only: `tableDDL` (idempotent `CREATE TABLE IF NOT EXISTS`) and `alterStatements` (idempotent `ALTER TABLE ADD column`). Never destructive. New `cluster_registry` table goes in `tableDDL`.
- Cassandra LWT (`IF EXISTS` / `IF NOT EXISTS`) REQUIRED for read-after-write coherence on rows queried with quorum after a previous LWT write. `PutCluster` UPDATE on existing rows MUST use LWT `IF version = ?` for CAS-on-Version. Plain UPDATE after LWT INSERT leaves Paxos state stale.
- TiKV pessimistic txn with EARLY-RETURN paths (e.g. CAS-reject `applied=false, nil`) MUST call `txn.Rollback()` explicitly. `defer rollbackOnError(txn, &err)` only fires when `err != nil`. Leaked LockKeys lease deadlocks in-process memBackend forever.
- TiKV key encoding: variable-length string segments (cluster id) use FoundationDB-style byte-stuffing (`0x00 → 0x00 0xFF`, terminator `0x00 0x00`) per `internal/meta/tikv/keys.md`. Schema-additive.
- Admin API handlers under `internal/adminapi/`. New file `storage_clusters.go` follows patterns in `storage.go`. Audit override via `s3api.SetAuditOverride(ctx, action, resource, bucket, principal)` with `action=\"admin:<Verb>\"` (e.g. `admin:CreateCluster`), `resource=\"cluster:<id>\"`. Required for every state-changing admin handler per CLAUDE.md.
- OpenAPI contract `internal/adminapi/openapi.yaml` is the canonical Admin-API source. New admin endpoints MUST update it + handler.
- RADOS backend at `internal/data/rados/backend.go`. `Backend.clusters map[string]ClusterSpec` (line 22) is the in-memory cluster catalogue. `Backend.conns map[string]*goceph.Conn` (line 41) + `Backend.ioctxes map[string]*goceph.IOContext` (line 42) are the lazy-dial connection pool. `connFor(ctx, id)` (line 111) lazy-dials under `b.mu`. `ioctx(ctx, cluster, pool, ns)` (line 149) handles per-pool ioctx caching.
- `internal/serverapp/serverapp.go::buildDataBackend` (line 344) currently passes `cfg.RADOS.Clusters` (parsed from `STRATA_RADOS_CLUSTERS` env) into `datarados.New`. This cycle adds `Meta meta.Store` field to `datarados.Config`; serverapp wires after `buildMetaStore`.
- S3 backend (`internal/data/s3/backend.go`) is SINGLE-bucket per Backend instance. No `STRATA_S3_BACKENDS` multi-backend env exists. This cycle ships ONLY the registry surface (cluster_registry table + admin API + RADOS watcher); S3-backend per-instance lifecycle is a separate P2 follow-up filed in US-007 close-flip.
- Cluster catalogue spec is opaque JSON `[]byte` in `ClusterRegistryEntry.Spec` — registry doesn't validate spec internals; consuming backend (rados.Backend / s3.Backend) decodes. RADOS decodes via `json.Unmarshal(e.Spec, &ClusterSpec{})`.
- Watcher pattern: every gateway replica polls `meta.ListClusters` every 30s (no leader needed — diff is idempotent; meta.Store is consensus source). On poll: compute set-diff against `Backend.clusters` map. Added → merge under write lock; `connFor` lazy-dials on next traffic. Removed → close cached conn + ioctxes under lock, remove from map. Updated (same id, bumped Version) → same as add (replace + re-dial on next traffic).
- `STRATA_CLUSTER_REGISTRY_INTERVAL` env knob (default 30s, range [5s, 5m]) read at `RegistryWatcher` construction. Constructor-time env reads, not per-call.
- `data.Backend` interface gains `Close(ctx) error`. Idempotent. Memory Close = noop; S3 Close drops client; RADOS Close drains pool + stops watcher.
- Goroutine leak detection: `runtime.NumGoroutine()` baseline + delta within tolerance (pattern from US-002 prefetch reader test in parallel-chunks cycle).
- Race detector: `go test -race ./internal/data/rados/...` MUST stay clean. Watcher uses `b.mu` for cluster map mutations.

Gateway with empty registry starts cleanly but cannot serve RADOS traffic until operator POSTs at least one entry via admin API. - **Learnings for future iterations:**
  - Import cycle: `internal/meta` imports `internal/data` (for `data.Manifest`), so `internal/data/rados` CANNOT import `internal/meta`. The watcher operates on a minimal `CatalogEntry` projection + `CatalogReader` interface; an adapter at the consumer (`internal/serverapp`) bridges `meta.Store.ListClusters` to that interface. Same pattern when future data-layer features need meta state — define the projection in the data side, adapter in serverapp.
  - Watcher goroutine race: a tempting `defer close(w.done)` inside the loop while Stop later sets `w.done = nil` races even under a mutex — the channel field is read off-lock by the goroutine's deferred close. Fix: pass the captured `done chan struct{}` into `loop` as a parameter so the goroutine never reads the struct field after Start. Same pattern applies to any cancel/done helper struct with a Stop() that nils fields.
  - `defer rollbackOnError`-style helpers don't apply here, but the symmetrical lesson does: `Backend.Close` must call `watcher.Stop()` BEFORE acquiring `b.mu`. If Close took `b.mu` first, an in-flight `ReconcileClusters` (which holds `b.mu` mid-tick) would deadlock against Stop's wait-for-loop-to-exit step.
  - `Close(ctx) error` signature change scope: only ~25 callsites in the tree, mostly test cleanup `_ = b.Close()`. The interface bump is mechanical but worth the clarity — shutdown context lets Close honour gateway shutdown deadlines (currently the rados Close synchronously calls into librados, but the contract now allows future async drain).
  - `s3.Backend.Close` now nils `b.client`; subsequent method calls on a closed s3 backend NPE rather than silently no-op. Acceptable because the only caller is `defer dataBackend.Close(...)` at gateway shutdown; tests don't reuse a Close'd backend.
  - The legacy `cfg.Clusters` / `cfg.ConfigFile` path is preserved as a soft-fallback when `cfg.Catalog == nil` — US-006 retires it entirely. Until then, the cycle's mixed-mode shape lets the s3-tests harness keep running unchanged.
  - `runtime.NumGoroutine()` leak-checks need a tolerance: parallel tests + race-detector helpers spawn background goroutines outside the watcher. The test uses `baseline+1` as the upper bound and polls up to 2s for the count to deflate. More precise leak detection (e.g. `gleak`) would be heavier than warranted.
