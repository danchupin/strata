- **Learnings for future iterations:**
  - The `RingBuffer.lru` is a `container/list.List` with `PushFront` on insert + `MoveToFront` on touch; `Front()` is therefore the newest entry, `Back()` is the eviction candidate. Walking `el = lru.Front(); el.Next()` yields LRU order (newest first) without sorting â€” directly matches "started_at desc" semantics. Don't reach for sort.Sort.
  - `TraceSummary.Status` aggregation: server picks `Error` if ANY span errored, otherwise the root span's status (parentless span). Without aggregation, an HTTP root span with `status=Unset` masks downstream failures â€” the client side would need to fan out and read every span. Compute on the server where the data already lives.
  - TanStack Query's `refetchInterval` is automatically paused when `document.visibilityState !== 'visible'` (because `refetchIntervalInBackground` defaults to false). No need to wire a `visibilitychange` listener. Same pattern applies to all polling panels (Storage, Multipart, AuditLog).
  - Audit stamp lives on the override (`stampAuditOverride(r, "admin:GetDiagnosticsTraces", "diagnostics", "-")`) â€” bucket is "-" because `diagnostics` isn't bucket-scoped. The audit middleware sits OUTSIDE this handler; the override is read AFTER `next.ServeHTTP` returns. Safe to stamp before doing the work; the middleware reads the latest value.
  - OpenAPI spec for diagnostics endpoints is sparse (only the new `/diagnostics/traces` path; the existing `/diagnostics/trace/{id}`, `/diagnostics/slow-queries`, `/diagnostics/hot-buckets` etc. are NOT in the spec). When adding a new diagnostics endpoint, document only that one â€” a sweep to backfill the others is a separate cleanup; keep stories focused.
  - `stampAuditOverride` (4-arg helper) vs `s3api.SetAuditOverride` (5-arg, owner explicit): the 4-arg helper resolves owner from the request's auth context â€” preferred for admin endpoints where the caller is the operator session subject. Reach for the 5-arg form only when stamping an audit row from a code path that doesn't have the owner in the request context (e.g. background-job spawn).
