- `writeChunk` (`backend.go:221`) is the single-OID RADOS write helper. It wraps `goceph.CreateWriteOp().WriteFull().Operate()`. Wrap this call site in the worker goroutine; the rest stays the same.
- `ObserveOp(ctx, b.logger, b.metrics, b.tracer, spec.Pool, op, oid, start, err)` is the per-op observability hook. Fires AFTER the RADOS op returns. Must continue to fire from each worker so per-chunk duration metrics + tracer spans stay accurate (start time captured per chunk inside the worker, not in the dispatcher).
- ETag is a running MD5 over chunk bytes IN SOURCE ORDER (`hash.Write(chunk)` in the current sequential loop). Parallel impl MUST preserve this: a single owner goroutine reads the input stream chunk-by-chunk in order, feeds the hasher in order, dispatches `(idx, []byte)` work to workers, and seals `m.ETag = hex.EncodeToString(hash.Sum(nil))` only after all workers report done. NOT a per-worker partial-hash — that breaks the byte-stream-MD5 invariant.
- Manifest chunk order MUST equal source byte order. Workers can complete out of order; assemble via `m.Chunks = make([]data.ChunkRef, totalChunks)` and `m.Chunks[idx] = ...` from each worker — pre-size by counting reads, OR use an ordered append into a `[]data.ChunkRef` keyed by index with a single seal pass after `wg.Wait()`.
- Error semantics: on first worker error, cancel the context to stop in-flight workers, drain the pool, call `cleanupManifest` over chunks that DID land in RADOS. The current sequential code does `cleanupManifest(m.Chunks)` on error — preserve that exactly. Use `errgroup.WithContext` from `golang.org/x/sync` (already a transitive dep).
- Reader cancel: respect `ctx.Err()` at chunk-read boundary (already done in current code) AND at worker dispatch (don't block on a backed-up worker pool channel when the client gave up). Use `select { case workCh <- work: case <-ctx.Done(): return ctx.Err() }`.
- `STRATA_RADOS_PUT_CONCURRENCY` env knob (range [1, 256], default 32). Read in `rados.New` / `rados.Backend` constructor — NOT per-call. Follows the project's "worker envs read at Build time" convention (similar to `STRATA_GC_CONCURRENCY` per CLAUDE.md).

## Sun May 10 23:55:00 MSK 2026 - US-002
- What was implemented: bounded-depth prefetch reader for the RADOS data backend's GetChunks. `internal/data/rados/prefetch.go` defines `newPrefetchReader(ctx, m, offset, length, depth, getOne)` returning a `*prefetchReader` (io.ReadCloser). The reader (a) selects only the chunks overlapping `[offset, offset+length)`, (b) launches a dispatcher goroutine that gates fetcher launches on a sem of capacity `depth`, (c) per-chunk fetcher writes its result into a per-chunk size-1 future channel, (d) `Read` consumes futures in source order and releases sem tokens after each chunk so dispatcher can start the next prefetch. Memory inflight is bounded by depth × chunk_size: the sem holds the slot until the reader has consumed the body. `Close` cancels the reader-scoped context (so in-flight `getOne` calls abort if they respect ctx), drains any unread futures non-blockingly, and waits for dispatcher + fetchers via WaitGroup. Range-GET semantics: the dispatcher hands `(off, length)` to `getOne` so the first chunk reads from in-chunk `firstOff` and the last chunk truncates to the trailing tail — bandwidth-equivalent to the old single-chunk-at-a-time radosReader. The legacy `radosReader` type is retired; `backend.go::GetChunks` now builds a `chunkGetFn` closure (`b.ioctx + ioctx.Read + ObserveOp`) and returns `newPrefetchReader(ctx, m, offset, length, b.getPrefetch, getOne)`. Knob `STRATA_RADOS_GET_PREFETCH` [1, 64] default 4 read once at Backend constructor (`Backend.getPrefetch` field).

- **Learnings for future iterations:**
  - Patterns:
    - Symmetric to PutChunks (US-001): a tag-free scheduler file (`prefetch.go`) lets us unit-test the bounded-pool coordination logic on macOS with no librados, while the ceph-tagged `backend.go` binds the storage callback. The `chunkGetFn` callback signature includes per-chunk `(off, length)` so the storage layer can short-read the first/last chunks of a range — bandwidth parity with the legacy single-chunk-at-a-time reader.
    - For bounded-concurrency producer/consumer with strict result ordering, prefer a per-result size-1 future channel + a sem of capacity N rather than a reorder buffer. The reader pulls future[cur], advances, and releases the sem token AFTER consuming — this naturally bounds memory to N (in-flight + buffered) without a separate "memory budget" accounting layer.
    - `sync.WaitGroup.Go(func)` (Go 1.25) is preferred over manual `Add(1) + go func() { defer Done() }` per the `waitgroup` linter. Adopt in fresh code.
  - Gotchas:
    - When the dispatcher is gated on a sem channel and ctx is cancelled, the `select { case sem <- {}: case <-ctx.Done(): }` branch must close out the remaining futures (write `getResult{err: ctx.Err()}`) so reader's `Read` doesn't block forever waiting on a future that no fetcher will ever fill. Easy to miss when only thinking about the happy path.
    - The legacy `radosReader` reads the FIRST chunk from in-chunk offset (bandwidth-saving) but reads the LAST chunk's full body (range emit truncation in `Read`). To preserve, the `chunkGetFn` callback receives `(off, length)` per chunk; for chunks NOT touched by the range first/last, `off=0, length=ref.Size`. For range queries strictly within one chunk, `off=offset-base, length=segLen`. Test `TestPrefetchReaderRangeStopsBeforeUnneededChunks` validates that fetches don't exceed the touched-chunk window.
    - Goroutine-leak test via `runtime.NumGoroutine()`: take baseline before constructing reader, allow `delta ≤ 2` after `Close()` returns + a 500ms drain window. Strict `delta == 0` is racey because runtime housekeeping goroutines may briefly appear during the test.
    - The `intFromEnv` helper in `parallel.go` is shared package-wide — `prefetch.go` reuses it instead of duplicating. New env helpers should keep using `intFromEnv` to avoid drift.
