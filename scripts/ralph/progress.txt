Locally I had go1.25.1; the Go toolchain auto-downloaded go1.25.10 on first `GOTOOLCHAIN=go1.25.10+auto go ...` invocation.   - **govulncheck callgraph trace shape**: `finding.trace[]` arrays — element 0 is the vuln site (`stdlib` / module package), last element is the calling Strata function. `trace.length == 1` means imported-but-not-called (no callgraph hit). `trace.length > 1` AND last frame's `module == github.com/danchupin/strata{,/cephimpl}` = real callgraph hit.   - **govulncheck exit codes**: `0` = no callgraph hits, `3` = at least one callgraph hit, anything else = scan error (binary missing, build failure). Local macOS dev hosts trip `librados.h not found` — the wrapper degrades to WARN+exit 0 in that case (matches `make vet`'s degradation pattern for the same reason). The CI matrix is the source of truth for cephimpl coverage.
  - **`scripts/lib/severity.sh` is shared infrastructure** — US-002 trivy, US-003 gosec, US-009 go-licenses are expected to source it for severity-classification helpers + ANSI color + Strata module allowlist. Adding helpers there is preferred over duplicating jq filters into each scanner wrapper.

- **Learnings for future iterations:**
  - **`gh api repos/<owner>/<repo>/git/refs/tags/<tag>` returns the tag-ref object SHA, NOT the commit SHA when the tag is annotated.** Two-hop resolution required: `gh api .../git/refs/tags/<tag> → .object.sha` then `gh api .../git/tags/<that-sha> → .object.sha` yields the commit SHA. Annotated tags are the upstream convention for both `trivy-action` and `codeql-action`. Lightweight tags would resolve in one hop (`.object.type == "commit"` directly). Always check `.object.type` first — `tag` = annotated, `commit` = lightweight. SHA-pinning to the tag-ref SHA instead of the commit SHA technically works but is brittle: a tag-rewrite changes the tag-ref SHA, breaking the pin. Pin to the commit SHA underneath.
  - **`aquasecurity/trivy-action` v0.36.0** (latest stable as of 2026-05-28) supports `trivy-config: <path>` input which loads a `trivy.yaml` shape. Action's other knobs (`severity`, `exit-code`, `format`, `output`) layer on top — the config file is for cycle-wide policy (scanners, timeout), the action inputs are for per-pass gates. Two passes (CRITICAL + HIGH) is the standard split — single pass with `severity: 'CRITICAL,HIGH'` would conflate gate semantics.
  - **`github/codeql-action/upload-sarif` `category` input** distinguishes multiple SARIF uploads in the same job — GitHub's Code Scanning UI merges entries by category. Setting `category: 'trivy-critical'` keeps this gate distinct from future SARIF emissions (US-003 gosec will use `category: 'gosec'`).
  - **`if: always()`** on the SARIF upload + HIGH WARN-only pass + artifact upload steps ensures the workflow continues past the CRITICAL gate failure — operators see the full HIGH-class findings + Code Scanning entry even when the gate hard-fails. Without `if: always()`, a CRITICAL failure short-circuits the job and Code Scanning never receives the SARIF.
  - **`trivy-image` rebuilds the image even though `docker-build` already built it** — buildx cache layers are job-local on GitHub Actions runners. Cross-job image artifact passing via `docker save | docker load` would transfer multi-GB tarballs over the Actions artifact backend, dominating job wall-clock. Rebuilding under `docker/setup-buildx-action@v3` warm cache is faster + simpler. Same trick as `e2e` / `e2e-full`. The PR review-time wall-clock delta is ~3-5 min image build + ~30s trivy scan — fits the +30-60s budget per S4 from PRD review.
  - **`scripts/check-trivy.sh` does NOT mirror the `check-govulncheck.sh` matrix axis pattern** — trivy scans the published image (`strata:ceph`), not Go source modules. The cephimpl module shape is irrelevant at the image layer — the final image already linked cephimpl via the `-tags ceph` build inside the Dockerfile. One image, one scan, one gate. The matrix axis is only relevant for Go-source-level scanners (govulncheck, gosec, go-licenses) where the cephimpl module is a separate Go module per CLAUDE.md.
  - **trivy 0.36 image-scan `format: 'sarif'`** writes SARIF v2.1.0 with `runs[0].tool.driver.name = 'Trivy'` — codeql-action SARIF upload validator accepts this out-of-the-box. No conversion step needed.
  - **trivy CRITICAL+HIGH dual pass duplicates DB download** — each invocation pulls the vuln DB fresh unless TRIVY_CACHE_DIR is set + warm. Future optimization (not blocking US-002): pin `cache-dir: /tmp/trivy-cache` + actions/cache the directory across the two steps. Skipping for now — DB download is ~5-10s, dominated by image-scan wall-clock.

  - **Scan by DIGEST, never by tag, for SBOM/sign jobs.** `image@sha256:<digest>` (from the build job output) is the single-digest contract — a tag could be re-pushed between the build and the SBOM job, breaking the "same image attested + signed + SBOM'd" invariant. All three downstream jobs (provenance US-006, sbom US-007, cosign US-008) consume `needs.build.outputs.digest`.
  - **`softprops/action-gh-release` auto-creates the release if absent** for the pushed tag, else appends the asset to the existing one. Gating on `startsWith(github.ref, 'refs/tags/v')` prevents a `workflow_dispatch` branch run from erroring on a nonexistent release. `fail_on_unmatched_files: true` makes a missing/misnamed SBOM file hard-fail rather than silently shipping a release with no SBOM.
  - **anchore/sbom-action `upload-artifact-retention` input** sets the workflow-artifact retention in days (90 here per AC). The artifact name is set by `artifact-name`; keeping it identical to `output-file` means the downloadable asset and the in-repo filename match the `strata-<tag>-sbom.spdx.json` shape US-010's verify recipe references.

- **Learnings for future iterations:**
  - **cosign v3.x changed the offline-signing flag contract.** `--tlog-upload=false` is DEPRECATED in v3 and HARD-CONFLICTS with the new `--use-signing-config=true` default ("`--tlog-upload=false is not supported with --signing-config`"). For offline key-based signing you MUST pass BOTH `--use-signing-config=false --tlog-upload=false`. On verify, pair with `--insecure-ignore-tlog=true`. The CI keyless path (cosign-installer-pinned version) does NOT use these flags — they are ONLY for the offline dev smoke. Brew installs bleeding-edge cosign (v3.0.6 here) whose flag surface drifts from the cosign-installer-pinned CI version, so the smoke must be version-defensive.
  - **A COLD cosign cache initialises its Sigstore TUF trust root on first use over the network** — the very first `cosign sign` on a fresh machine can hang indefinitely if the network is slow/absent (observed: first invocation hung >40s, second completed in <25s once the TUF root was cached at `~/.sigstore`). The smoke wraps every cosign call in `timeout ${COSIGN_TIMEOUT}` (portable: `timeout` → `gtimeout` → bare) and degrades to WARN on exit 124 rather than hanging the whole `make smoke-supply-chain`.
  - **Signing a LOCAL-only image requires it to live in a registry** — cosign operates on registry refs (`registry/repo@sha256:...`), NOT docker-daemon-local images. The smoke spins an ephemeral `registry:2` on a host port + `docker push`es to it. A local HTTP registry needs `--allow-http-registry --allow-insecure-registry` on BOTH sign and verify; without them cosign errors on the plaintext-HTTP transport.
  - **`docker inspect --format '{{index .RepoDigests 0}}'` resolves the pushed manifest digest** after `docker push` — this is the digest cosign must sign (the single-digest contract). Resolving the digest from the local image ref before push yields nothing (`RepoDigests` is populated by the push). Always push THEN inspect.
  - **The negative-verify pass (wrong key → expect FAIL) is the load-bearing assertion.** A verify that only checks the happy path can silently pass even when verification is a no-op (wrong flags, stubbed verify). Generating a SECOND unrelated key pair and asserting its verify FAILS is what makes the smoke meaningful — without it the smoke proves nothing about signature integrity.
  - **`set -euo pipefail` + capturing a command's exit code**: a bare `cmd; rc=$?` triggers `set -e` exit on non-zero BEFORE `rc=$?` runs. Use `rc=0; cmd || rc=$?` — the `|| ...` puts the command in a checked context so `set -e` is suppressed and the exit code is captured. Needed for the timeout-124-detection branch.
