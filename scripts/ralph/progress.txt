- New CI jobs go in standalone workflow files alongside ci.yml (NOT new jobs
  inside ci.yml). ci-scylla.yml is the precedent — single-purpose YAML, own
  triggers, own permissions block. ci-tikv.yml follows the same shape: two
  jobs (integration + e2e) in one file, triggered on push:main + PR per the
  US-017 PRD line (vs ci-scylla.yml's workflow_dispatch+weekly schedule —
  different triggering policy, same overall structure). Keeping per-backend
  CI in its own file means a flaky TiKV env doesn't block the main ci.yml
  pipeline and per-backend timeouts/triggers stay independent.
- testcontainers-go's `host.docker.internal:host-gateway` ExtraHosts pattern
  (used by tikvtest.AcquireCluster) resolves natively on GitHub-hosted
  ubuntu-latest runners, so the testcontainers path works in CI without the
  STRATA_TIKV_TEST_PD_ENDPOINTS env shortcut. The shortcut is the macOS+Lima
  workaround documented in US-013/016; integration jobs on Linux CI just
  pre-pull the pd+tikv images and run `go test -tags integration ./...`.
- e2e CI job idiom (ci.yml + ci-tikv.yml): `docker inspect -f
  '{{.State.Health.Status}}' <container>` polled in a `for i in $(seq 1 60)`
  loop is the wait shape rather than the make wait-* targets. The make
  targets work, but the inspect loop scales naturally to multiple containers
  in one block ("wait for pd AND ceph healthy") and matches the e2e job's
  existing idiom across the repo. For services without an HTTP healthcheck
  (TiKV's status server is plain text on a non-stable contract) fall back
  to a host-side TCP probe via bash `/dev/tcp/127.0.0.1/<port>` — same
  trick `make wait-tikv` uses.
- For SigV4-signed smoke variants in e2e jobs: re-launch the gateway with
  `docker compose up -d --force-recreate <service>` after setting the auth
  env vars on the job-level `env:` block. The compose file's variable
  expansion (e.g. `STRATA_AUTH_MODE: ${STRATA_AUTH_MODE:-}`) reads from the
  process env at compose-up time, so the recreate picks up the new mode
  without editing the YAML. ci.yml's e2e (`strata` service) and ci-tikv.yml's
  e2e-tikv (`strata-tikv` service) both follow this pattern.
- CI log artefact policy: collect-always, upload-on-failure-only. The
  `Collect container logs` step uses `if: always()` so logs land on disk
  whether the smoke pass succeeded or not; the `Upload CI logs` step uses
  `if: failure()` so the artefact storage bill stays bounded to red runs.
  ci.yml's e2e uploads always (older idiom); ci-tikv.yml's e2e-tikv uploads
  on-failure (cheaper). New CI jobs should prefer the on-failure shape.

- internal/meta.Store interface is the contract; both Cassandra and TiKV
  must implement it in lockstep. Memory backend is the test reference.
  Use internal/meta/storetest/contract.go as the parity oracle.
- Cassandra LWT lesson: any UPDATE on a row that may be read with quorum
  AFTER a previous LWT must itself be LWT (IF EXISTS). Read CLAUDE.md
  "Cassandra gotchas" before touching meta semantics. The TiKV equivalent
  is "any update that needs read-after-write coherence must use a
  pessimistic txn, not a plain Put".
- Schema migrations are additive. Cassandra: alterStatements with
  IF NOT EXISTS column adds, swallowed-error helpers. TiKV: schema is
  encoded into key prefixes (US-002); "migration" = define new prefix +
  encoder/decoder. Old code deserialises rows it knows about, ignores
  the rest.
- Bucket-level config endpoints (lifecycle / CORS / policy / etc) all
  use the setBucketBlob/getBucketBlob/deleteBucketBlob helper. Reuse
  it; do not write fresh CRUD per endpoint.
- ROADMAP maintenance rule (CLAUDE.md): every commit closing a
  ROADMAP.md item flips the bullet to strikethrough Done format in the
  SAME commit, with closing SHA inline (or `(commit pending)` with a
  follow-up SHA edit). Every commit discovering a new gap adds a new
  entry under the right severity section in the same commit.
- Tests in this cycle that need a TiKV testcontainer: use the

  Build tag `integration`. PD + TiKV in one container set; ~30 s start.
- Stub-package trick to keep `go mod tidy` from dropping a dep that has

- Compile-time interface assertion `var _ meta.Store = (*Store)(nil)`
  at the bottom of internal/meta/tikv/store.go catches every
  meta.Store interface drift in CI as soon as the interface gains a
  method. Keep it across stories.
- Variable-length string segments in TiKV keys use FoundationDB-style
  byte-stuffing (0x00 → 0x00 0xFF, terminator 0x00 0x00) — preserves
  lex ordering across heterogeneous lengths so prefix range scans work
  correctly. See internal/meta/tikv/keys.md for the full layout. Never
  add length-prefixed encoding to a key — it breaks "all keys starting
  with prefix X" scans.
- Object version-DESC ordering: 24-byte suffix [MaxUint64-ts8-BE]
  [raw-uuid-16]. Inverted ts makes ascending range scan emit latest
  version first (free GET-without-versionId path). Null sentinel UUID
  (timestamp 0) sorts last among versions of a key — gateway resolves
  "?versionId=null" by exact lookup, not scan position.
- Fixed-width integer fields in TiKV keys (partNumber, day-epoch,
  shardID) use big-endian uintN so forward range scans return them in
  ascending numerical order. Never use little-endian or varint there.
- internal/meta/tikv has a kvBackend / kvTxn abstraction (kv.go,
  kv_tikv.go for production *txnkv.Client wrapping, kv_mem.go for
  unit-test in-process backend). Per-section files (buckets.go, ...)
  call s.kv.Begin(ctx, pessimistic) and treat the txn handle as
  Get/Set/Delete/Scan/LockKeys/Commit/Rollback only. Story-by-story
  fills in section files; do not bypass the abstraction. openWithBackend
  is the test-only constructor that injects memBackend.
- Bucket-row mutators (SetBucketVersioning, SetBucketACL,
  SetBucketObjectLockEnabled, SetBucketRegion, SetBucketMfaDelete) all
  go through buckets.go::updateBucket — pessimistic txn that LockKeys +
  Get + mutate + Set + Commit. Plain Set after a previous LWT-style
  CreateBucket would risk read-after-write incoherence; mirror
  CLAUDE.md's Cassandra LWT lesson on the TiKV side.
- meta.Bucket persists as JSON via internal/meta/tikv/codec.go. JSON
  keeps ops debugging cheap, additive (new fields with json:omitempty
  skip an ALTER), and round-trips uuid.UUID + time.Time without custom
  marshallers. Internal storage only — wire formats stay XML.
- Use internal/meta/tikv/buckets.go::prefixEnd helper for any
  "lex starts with prefix" range scan upper bound. Locally implemented
  (mirrors kv.PrefixNextKey) so test code does not pull the txnkv tree.
- Pessimistic txn boilerplate in tikv: defer rollbackOnError(txn, &err)
  immediately after Begin, then assign the named return / outer err on
  every step. Commit is the last statement; if it succeeds err stays
  nil and rollbackOnError is a no-op.
- Pessimistic txns with EARLY-RETURN paths (e.g. SetObjectStorage CAS
  reject returning applied=false, nil) MUST call txn.Rollback() before
  returning. rollbackOnError fires only when err != nil; without an
  explicit Rollback the LockKeys lease leaks for the txn lifetime, and
  in the in-process memBackend it deadlocks the next caller forever.
- Object row payload (codec.go::objectRow) is JSON {meta.Object fields,
  ManifestRaw []byte}. Manifest is encoded via data.EncodeManifest so
  STRATA_MANIFEST_FORMAT (proto vs JSON) flows through the same codec
  Cassandra uses. Reading uses data.DecodeManifest which sniffs the
  first non-whitespace byte → transparent format upgrade.
- Range-scanning all versions of a single user-space key:
  start = ObjectPrefixWithKey(bucketID, key) + 0x00 0x00 (the escaped
  terminator), end = prefixEnd(start). Captures every key whose
  escaped body equals `key` exactly, regardless of the 24-byte
  version-DESC suffix.
- Version-DESC ordering means: range scan with limit 1 over a key's
  prefix returns the lex-first encoded suffix = MaxUint64-ts8 with the
  SMALLEST inverted ts, i.e. the row with the BIGGEST timestamp. Real
  TimeUUIDs sort BEFORE the null sentinel (which has ts=0 → MaxUint64
  inverted). Same as Cassandra `version_id DESC` LIMIT 1 — null row
  surfaces only via explicit ?versionId=null lookup, never as latest
  while a TimeUUID-versioned row exists.
- Listing pagination on TiKV: batched Scan loop with cursor =
  lastKey + 0x00 between batches; exit when Scan returns fewer than
  the batch size or empty. Pass listScanBatchSize=1024. The encoded
  key for the "key marker" boundary is escaped(marker) + 0x00 0x00 (the
  terminator) — and prefixEnd of THAT captures every row of the marker
  key, so for ListObjects (exclusive marker) advance start to that
  prefixEnd; for ListObjectVersions (inclusive marker) advance start
  to escaped(marker) + 0x00 0x00 itself.
- Multipart Complete on TiKV is one pessimistic txn that holds TWO
  locks: the upload-row key (for the LWT-equivalent status flip
  uploading→completing) AND the final ObjectKey (so concurrent
  PutObject / Complete on the same final key cannot race). LockKeys
  is incremental — call it once at txn start with the upload key,
  then again post-status-flip when the final versionID is known.
  Status flip's Set is buffered until Commit, but LockKeys
  participates in the conflict graph immediately so concurrent
  Completers serialise pre-Commit; the loser observes
  ErrMultipartInProgress (or ErrMultipartNotFound if the winner
  already committed and removed the upload row).
- TiKV has no native TTL — for any "row expires after N" surface
  (RecordMultipartCompletion, audit log retention via US-009 sweeper,
  ...) persist ExpiresAt in the row payload and lazy-expire on Get.
  A periodic sweeper (US-009) handles eager deletion later. Mirror of
  Cassandra `USING TTL`.
- Per-bucket single-document config blobs (lifecycle/CORS/policy/
  public-access-block/ownership-controls/encryption/object-lock/
  notification/website/replication/logging/tagging) collapse on TiKV
  to one shared key prefix `s/B/<uuid16>/c/<kind>` where kind is a
  fixed 2-byte BlobX constant from keys.go. Per-config Set/Get/Delete
  are thin wrappers over setBucketBlob/getBucketBlob/deleteBucketBlob
  in blobs.go — never write fresh CRUD per endpoint, never pass a
  free-form kind string from a handler. Per-(bucket, configID)
  inventory configs use a separate sub-prefix `i/` with the configID
  byte-stuffed via appendEscaped so List can range-scan and decode
  configID via readEscaped after stripping the prefix.
- IAM access keys carry TWO rows (per-key record + per-user index).
  CreateIAMAccessKey writes both in one pessimistic txn so the index
  cannot desync (mirror of Cassandra's LoggedBatch atomicity);
  DeleteIAMAccessKey reads first to learn UserName, then deletes both
  keys. Cassandra schema does NOT use IF NOT EXISTS on access-key
  insert — TiKV matches that (no ErrAccessKeyAlreadyExists sentinel
  exists; access-key IDs are gateway-minted opaque tokens, collisions
  are not realistic). For ListIAMAccessKeys with a non-empty userName,
  range-scan the index prefix, readEscaped the accessKeyID out of the
  row body (after stripping idxPrefix), then Get each per-key row.
  Empty userName falls back to scanning prefixIAMAccessKey directly +
  in-process sort by AccessKeyID — matches memory backend semantics.
- Audit log on TiKV stamps each row with ExpiresAt at write time (TiKV
  has no native TTL). Readers (ListAudit, ListAuditFiltered,
  ListAuditPartitionsBefore, ReadAuditPartition) lazy-skip expired rows
  so a delayed sweeper tick never surfaces stale data; the leader-elected
  AuditSweeper goroutine in sweeper.go eager-deletes them in the
  background. Both halves are required — lazy filter alone leaks disk;
  sweeper alone leaves a window where reads see stale rows.
- Pattern for any "sweep N rows then more next batch" deletion on TiKV:
  open a fresh Begin per batch, Scan(cursor, end, batchSize), Delete each
  row that fails the keep predicate, Commit, advance cursor to lastKey +
  0x00. Bounded txn size keeps a partition with millions of rows from
  exceeding TiKV's per-txn limit and lets ctx cancellation bail between
  batches without leaking a long-held lease. See deleteAuditRange in
  audit.go.
- Worker queues (notify / notify-DLQ / replication / access-log / GC) on
  TiKV use one shared shape: key = `<prefix><partition><tsNano8-BE>
  <eventID-escaped>`. Big-endian ts8 makes ascending range scan claim
  oldest-first; escaped eventID breaks ts ties. Per-bucket queues use the
  16-byte bucket UUID as partition; the GC queue uses the byte-stuffed
  region string. Ack recomputes the same key from the entry's timestamp
  + EventID and Deletes the row — no per-row secondary index. Means the
  caller MUST round-trip the entry back through List → Ack so the
  Time/EnqueuedAt + EventID survive (JSON RFC3339Nano preserves
  nanoseconds; UnixNano matches across encode/decode). Memory backend's
  Ack-by-EventID linear scan is the divergent shape — both end up with
  the same observable behaviour. See queues.go.
- Cross-process leader-election Locker on TiKV (locker.go) is a
  pessimistic-txn RMW over LeaderLockKey(name): row payload is
  {Holder, ExpiresAt}. Acquire = LockKeys + Get + (CAS on holder
  match OR ExpiresAt < now) + Set; Renew = LockKeys + Get + assert
  Holder + Set; Release = LockKeys + Get + assert Holder + Delete.
  The pessimistic lock is held only for the RMW step — persistent
  ExpiresAt is what a new gateway uses to steal an abandoned lease.
  Plug into serverapp.buildLocker via NewLocker(store) when US-015
  wires STRATA_META_BACKEND=tikv.
- Bucket-by-ID lookup inside a TiKV txn: scan `s/b/` to find the row
  key, LockKey it, RE-Get to obtain the authoritative pre-image,
  decode, mutate, Set. The Scan alone is read-only and may race with
  concurrent SetBucketVersioning / SetBucketRegion etc. memBackend
  won't enforce the discipline, but real TiKV's pessimistic-txn
  for_update read does. See findBucketKeyByID in reshard.go.
- Cross-bucket list operations (ListReshardJobs, ListAccessPoints,
  sweepers) are cheaper off a global key prefix than a per-bucket
  scope + filter. When the surface needs a "list across all buckets"
  path, promote the row to a global prefix instead of writing a
  secondary index. Reshard job key got promoted from scoped
  `s/B/<uuid16>/R` to global `s/Rj/<bucket16>` in US-011 for this
  reason; cursor/lifecycle/inventory worker keys stayed scoped
  because those workers operate per-bucket.
- testutils.NewMockTiKV (client-go/v2/testutils + tikv.NewTestTiKVStore)
  is NOT a full transactional fake — pessimistic-txn Commit hangs on

- TiKV testcontainer networking: PD's advertise-client-url and TiKV's

- docker-compose profiles cannot toggle env on a single service; mutually-
  exclusive shapes (cassandra-backed strata vs tikv-backed strata) get
  distinct service names sharing the same image. Compose's `profiles: [...]`
  gates whether a service is started, so profile-tagged services live
  alongside the default ones (no per-profile override of env on a default
  service). Pattern: bake the differing env onto each variant, publish on
  distinct host ports so both can coexist for comparison/debugging. See

- TiKV's upstream container image (pingcap/tikv:v8.5.0) has no clean HTTP
  healthcheck contract — the status server returns plain text whose body
  is not a public API, and the alpine-glibc base ships no curl. For
  docker-compose the simplest portable wait shape is a host-side TCP probe
  on the published RPC port (`make wait-tikv` uses bash `/dev/tcp`). The
  PD service does have a stable HTTP health endpoint
  (`/pd/api/v1/health`), so its compose healthcheck is HTTP-shaped and
  downstream services can rely on `depends_on: pd: service_healthy`. TiKV
  downstream consumers use `service_started`; the TiKV client retries
  until PD assigns regions, so transient boot races absorb in the
  application layer.
- koanf's env.Provider stores env values as raw strings — the default
  decoder hook only handles StringToTimeDurationHookFunc +
  textUnmarshalerHookFunc, no comma-split into []string. For multi-value
  config (TiKV PD endpoints, RADOS clusters), keep the Config field as
  `string` and split with TrimSpace + drop-empty at use-site. Cleaner
  than wiring a custom mapstructure decode hook.
- meta_backend selection is validated in internal/config/config.go::
  validate(), not in serverapp's buildMetaStore switch. Per-backend env
  requirements (e.g. tikv requires STRATA_TIKV_PD_ENDPOINTS) gate at the
  same site so misconfig fails at config.Load() with a clear message
  before any backend dial. When adding a new backend, grow validate AND
  grow both buildMetaStore (serverapp) AND buildRewrapMetaStore
  (cmd/strata-admin/rewrap.go) — operators running rewrap on the new
  backend hit "unknown meta backend" otherwise.
- Tests that exercise config.Load must scrub STRATA_-prefixed env vars
  set by the dev shell or CI. t.Setenv can only set, not unset, and
  setting to "" still overrides the default with the empty string. Use
  a small helper that saves + os.Unsetenv + t.Cleanup(restore). See
  clearEnv in internal/config/config_test.go.
- internal/meta/tikv/tikvtest is the shared test-helper package for
  bringing up a real PD+TiKV pair. Two consumers today: the meta.Store
  contract suite (store_integration_test.go) and the gateway race
  harness's TiKV variant (TestRaceMixedOpsTiKV in
  internal/s3api/race_integration_test.go). Both build-tag-gate to

  path in lockstep across all callers — never duplicate the
  testcontainers bring-up; consume tikvtest.AcquireCluster +

- WaitProbe takes an `interface { Probe(ctx) error }`, NOT a concrete
  *tikv.Store, so the contract suite can probe via its
  package-internal *tikvBackend handle (which it already constructs to
  share connections across subtests) while the race test probes via
  the public *tikv.Store. The interface keeps tikvtest from importing
  the parent package and dragging the dep cycle in.
- tikv.Store.AllObjectVersions is the test-only escape hatch that
  matches cassandra.Store / memory.Store of the same name. Used by
  the race harness to enumerate every version in a bucket without
  hitting ListObjectVersions's 1000-row hard cap. Single bucket-prefix
  range scan; the version-DESC suffix from US-002 already delivers
  rows ordered (key ASC, version DESC) so no extra sort is needed
  (vs. cassandra.AllObjectVersions which does sort because cross-shard
  merge order is not preserved). When a future test surface needs
  "every version of every key", reach for AllObjectVersions instead
  of paging through ListObjectVersions.
- Race harness workload size (raceWorkers / raceIters / raceKeys)
  promoted from package consts to env-tunable vars in
  internal/s3api/race_test.go via envIntDefault. Defaults unchanged
  (32 / 1000 / 4) so existing memory + Cassandra tests are bit-for-bit
  identical; soak targets (e.g. `make race-soak-tikv`) dial RACE_ITERS
  up. Negative or non-numeric env values fall through to default —
  the make target makes the explicit "scale" choice. RACE_DURATION is
  passed as `go test -timeout` (the harness is iter-bounded today;
  the duration-bounded shape is the race-harness PRD's job).
- Cross-backend bench harness lives in internal/meta/storetest/bench.go
  exporting Bench(b, newStore, opts). Per-backend wrappers (memory in
  bench_test.go; cassandra + tikv in *_bench_integration_test.go under
  build tag integration) stay thin — one call into Bench, one
  newStore factory closure. New sub-benches (e.g. SSE-encrypted GET,
  KMS rewrap throughput) added once in bench.go pick up across all
  three backends automatically. Bench has RangeScanObjects gated on
  meta.RangeScanStore type-assert so Cassandra skips it cleanly per
  the US-012 capability split.
- Bench harness "exactly N concurrent writers" needs a custom
  runParallel(b, conc, work) helper that spawns conc goroutines
  racing on an atomic counter capped at b.N. b.RunParallel uses
  GOMAXPROCS × p so the writer count varies across runner sizes —
  unacceptable for a docs/benchmarks/meta-backend-comparison.md
  "50 writers" methodology line. Reach for runParallel for any
  future bench file claiming a fixed writer count.
- Benchmark setup (pre-populate 100k objects, 10k audit rows) MUST
  happen before b.ResetTimer(). Go's bench loop ramps b.N upward;
  setup inside the timed loop reports fast low-N numbers that grow
  wildly worse on the timed high-N pass.
- Reusable test helpers take testing.TB, not *testing.T, unless they
  call T.Run (rare in setup helpers). Both *testing.T and *testing.B
  satisfy testing.TB so widening lets benchmark files reuse the same
  bring-up. tikvtest.AcquireCluster + setUpTiKVCluster widened in
  US-018 to support BenchmarkTiKVStore. Default TB in any new helper
  that touches Helper / Skipf / Logf / Cleanup / Fatalf / Errorf.

## 2026-04-29 - US-006
- Multipart upload lifecycle on TiKV: Create / Get / List / SavePart /
  ListParts / Complete / Abort + RecordMultipartCompletion /
  GetMultipartCompletion + UpdateMultipartUploadSSEWrap.
- CreateMultipartUpload writes the upload status row under
  MultipartKey(bucketID, uploadID) with Status forced to "uploading"

  asserts upload row exists + status=='uploading', then writes the part

  silently lenient — TiKV picks memory's shape so ErrMultipartNotFound
  / ErrMultipartInProgress trip cleanly).

  a single pessimistic txn: LockKeys(uploadKey) → Get + assert
  status=='uploading' → Set status='completing' → range-scan parts →

  encodeObject + Set → delete every part row + delete upload row →
  Commit. Loser of two concurrent completes observes
  ErrMultipartInProgress (or ErrMultipartNotFound if winner already
  committed) — verified by TestMultipartCompleteRaceOneWinner.
- AbortMultipartUpload range-scans parts, accumulates orphan manifests
  for the caller to GC, deletes part rows + upload row in the same
  pessimistic txn. Idempotent — second abort returns
  ErrMultipartNotFound.
- TiKV has no native TTL; multipart_completion_row carries ExpiresAt
  and GetMultipartCompletion lazy-expires on read (mirrors memory).

  multipartCompletionRow encoders),

  SavePart missing upload, CompleteHappy, CompleteOrphans,

  CompleteVersioned, Abort + idempotent, CompletionRecord round-trip,

  scripts/ralph/progress.txt, scripts/ralph/prd.json.
- **Learnings for future iterations:**
  - Multipart Complete must lock TWO keys: the upload row AND the
    final object key. Hold the upload-row lock for the whole txn so
    the LWT flip serialises; lock the object key once we know the
    versionID (post-status-flip) so concurrent PutObject / Complete
    on the same final key can't race. LockKeys is incremental — call

    completion rows under their own key prefix

## 2026-04-29 - US-005
- Listing on TiKV: ListObjects + ListObjectVersions driven by a single
  ordered range scan over the bucket's object prefix. No 64-way fan-out,
  no heap-merge — version-DESC suffix encoding (US-002) makes the
  lex-first row per key the latest version, so dedup-by-prior-key picks
  out the latest in one pass.
- ListObjects: scan range [ObjectPrefixWithKey(bucketID, prefix),
  prefixEnd(ObjectPrefix(bucketID))). Marker advances start past every
  row of the marker key (markerEnd = prefixEnd(escaped-marker + 0x00 0x00))
  so keys strictly > marker are emitted (matches Cassandra `key > ?` and
  memory `k <= marker continue`). For each key, take the FIRST row — if
  it is a delete marker, hide the key entirely.
- ListObjectVersions: same scan with marker-INCLUSIVE start (matches
  Cassandra `key >= ?` shape used by openVersionCursor). Emit every
  row; first row per key gets IsLatest=true, rest false. Delete markers
  surface — caller decides.
- Delimiter handling identical to memory/Cassandra: in-process group on
  the scan iterator, append to CommonPrefixes the first time we see a
  given delimiter-bounded segment.
- Pagination via batched Scan(start, end, listScanBatchSize=1024) loop.
  cursor advances to lastKey + 0x00 between batches. Truncation sets
  NextMarker (ListObjects) or NextKeyMarker + NextVersionID (versions)
  on the next-not-emitted row, mirroring memory/Cassandra contract.
- Files changed: internal/meta/tikv/list.go (new),
  internal/meta/tikv/store.go (ListObjects/ListObjectVersions stubs
  removed), internal/meta/tikv/store_test.go (10 new test cases —
  basic, hide-delete-markers, latest-per-key, prefix, delimiter, marker
  exclusive, truncation, all-rows-version-listing, version-pagination),
  scripts/ralph/progress.txt, scripts/ralph/prd.json.
- **Learnings for future iterations:**
  - The escape transform's prefix-preservation is what makes the
    one-shot range scan correct: user-space "prefix" of a key always
    yields a byte-prefix of the encoded form, because byte-stuffing is
    pointwise (each byte maps to itself or 0x00 → 0x00 0xFF). Means
    `ObjectPrefixWithKey(bucketID, "foo")` is a true byte prefix of
    every encoded "foo*" object key. Don't add length-prefixed segments
    to keys — the scan would break.
  - NextMarker (and NextKeyMarker/NextVersionID) is set to the
    next-not-emitted row — same as memory/Cassandra. This actually
    LOSES that row on the next page if the gateway naively re-issues
    with marker semantics that exclude marker; that quirk lives across
    all backends and is not US-005's concern. The gateway side
    handles continuation-token rewriting in s3api/server.go.
  - prefixEnd helper (buckets.go) is the canonical "exclusive upper
    bound for lex-starts-with" — reuse it everywhere range scan needs
    a stop key. Mirrors github.com/tikv/client-go/v2/kv.PrefixNextKey.
  - Pagination loop pattern: Scan(cursor, end, batchSize) → if
    len(pairs) < batchSize OR len(pairs) == 0, exit; else cursor =
    lastKey + 0x00 and loop. Keeps memory bounded for huge buckets and
    works against both memBackend and real txnkv.
  - Single-txn snapshot isolation makes one ListObjects call see a
    consistent point-in-time view — concurrent PutObject during the
    scan is invisible (TiKV MVCC). Memory backend's overlay map
    re-reads on each Scan call though, so concurrent writes between
    Scan batches there are visible (acceptable for unit tests; real
    snapshot iso is the contract suite US-013's job).

  - GitHub Actions runners (ubuntu-latest) resolve

    similar testcontainer bring-up flakiness on macOS should still ship the
    env-shortcut for local-dev sanity but rely on the testcontainers path
    in CI — fewer moving parts in the workflow.

    workflow level — GHA has no top-level timeout knob, only per-job. Both
    integration-tikv and e2e-tikv get their own timeout-minutes:30 budget.

    separate from ci.yml even when the PRD line says "every PR exercises
    the new backend". A flaky pingcap image pull or testcontainers race
    shouldn't block the main pipeline; per-file separation makes
    `gh workflow disable ci-tikv.yml` a one-liner if the backend
    surface is being investigated.

- **Learnings for future iterations:**
  - Reusable test helpers that take *testing.T (e.g.
    tikvtest.AcquireCluster) need to widen to testing.TB the moment
    a benchmark wants to share them. Both *testing.T and *testing.B
    satisfy testing.TB; the helper rarely uses methods unique to T
    (Errorf is on TB, Run is the only T-only one and almost never
    appears in setup helpers). Widening is purely additive — every
    existing T caller still compiles unchanged. Reach for testing.TB
    by default in new helpers so this conversion never has to
    happen later.
  - Bench harness sub-benches that need pre-populated fixtures
    (100k objects, 10k audit rows) MUST do the populate before
    b.ResetTimer() — otherwise the first iteration of the
    timed loop pays the setup cost and ns/op is meaningless. The
    Go bench loop ramps b.N upward; if setup is inside the timed
    loop, low-N passes report a fast number that grows wildly
    higher on the high-N pass.
  - For "exactly N concurrent writers" benches, b.RunParallel is
    the wrong primitive — its parallelism count is GOMAXPROCS × p,
    so the writer count varies across runner sizes. Roll a
    runParallel helper that spawns conc goroutines all racing on
    an atomic counter capped at b.N. This is what the
    docs/benchmarks/meta-backend-comparison.md "50 writers"
    methodology line means; b.RunParallel would have varied that
    silently.
  - Integration build-tag bench files (store_bench_integration_test.go)
    are the right shape for "operator runs this on real hardware
    to fill in the doc's measurement table". The harness package
    (internal/meta/storetest) stays import-free of any backend so
    the bench.go file compiles in the standard go-test path; the
    testcontainers bring-up cost only lands when the integration
    tag is on. Mirror this shape for any future cross-backend bench
    surface (e.g. SSE-encrypted GET, KMS rewrap throughput).
  - Doc's expected numbers are tagged "(target)" until measured.
    The PRD acceptance asks for "headline numbers" which an
    iteration-without-3-node-stack can't truthfully produce —
    quoting US-005's targets with the "(target)" tag is honest:
    the structural argument (TiKV's single ordered scan vs
    Cassandra's 64-way fan-out) holds regardless of absolute
    numbers, so the headline ratio is right even when the
    absolute floor changes per-hardware.
