### Phase 1 vs Phase 2 (this cycle is Phase 1)
- Phase 1 = bounded errgroup INSIDE the existing single-leader worker. Single replica still holds the lease. Throughput cap rises ~32× per worker via internal goroutine fan-out. Quick win + bench harness lands.
- Phase 2 (separate FUTURE cycle, NOT in this PRD): sharded leader-election (`gc-leader-0..N-1`, per-bucket `lifecycle-leader-<bucket>`) so multiple replicas process disjoint slices in parallel. Needs new `Meta.ListGCEntriesShard(ctx, region, shardID, shardCount, …)` API + per-bucket lease keying. Will compound on top of Phase 1's per-worker concurrency (M shards × C goroutines each = M×C parallel).
- Phase 1 is NOT wasted: (a) prep for Phase 2 — bench harness is a prerequisite for sizing Phase 2's targets; (b) single-replica deployments (dev/lab) win immediately; (c) errgroup pattern + log-and-continue reusable inside Phase 2's shard workers.

### gc.Worker structure (where to wedge errgroup)
- `internal/gc/worker.go::Worker` struct: Meta, Region, Interval, Grace, Batch, Logger, Metrics. Add `Concurrency int` as 7th field; zero/negative falls back to 1.
- `internal/gc/worker.go::drainCount` — outer loop paginates `ListGCEntries` (keep sequential, batch=500 default); INNER `for _, e := range entries { rados.Delete + meta.AckGCEntry }` is the wedge point. Wrap in `errgroup.Group` with `eg.SetLimit(w.Concurrency)`. `eg.Go(func() error { … log + return nil })` — log-and-continue, never bubble error to errgroup (the parent ctx must NOT be cancelled on per-entry failure).
- `processed` counter under `atomic.Int64` (not sync.Mutex — single counter, no compound state).
- Per-goroutine `defer func() { if r := recover(); r != nil { logger.Warn("gc panic", "entry", e, "recover", r) } }()` so a single bad entry never crashes the worker. Supervisor's panic backoff is for top-level panics, not per-entry.

### lifecycle.Worker structure
- `internal/lifecycle/worker.go::Worker` struct: Meta, Data, Region, Interval, AgeUnit, Logger. Add `Concurrency int` as 7th field.
- Nested loops: `runOnce` → `processBucket` → `for _, o := range res.Objects` → `for _, v := range res.Versions` → `for _, u := range uploads` (multipart sweep). Apply errgroup at the LEAF level (per-object / per-version / per-upload), NOT at the bucket level. Per-bucket parallelism is Phase 2 territory.
- CAS conflict path stays the same: `Store.SetObjectStorage` returning `applied=false` discards the freshly-written tier-2 chunks via the GC queue. Concurrent siblings see different `expectedClass` snapshots — the LWT enforces correctness regardless of order.
