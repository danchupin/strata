## Codebase Patterns

(none yet — populate as US-001..US-020 land. Seed entries below mirror what
prior cycles found useful; remove/replace if stale for this cycle.)

- internal/meta.Store interface is the contract; both Cassandra and TiKV
  must implement it in lockstep. Memory backend is the test reference.
  Use internal/meta/storetest/contract.go as the parity oracle.
- Cassandra LWT lesson: any UPDATE on a row that may be read with quorum
  AFTER a previous LWT must itself be LWT (IF EXISTS). Read CLAUDE.md
  "Cassandra gotchas" before touching meta semantics. The TiKV equivalent
  is "any update that needs read-after-write coherence must use a
  pessimistic txn, not a plain Put".
- Schema migrations are additive. Cassandra: alterStatements with
  IF NOT EXISTS column adds, swallowed-error helpers. TiKV: schema is
  encoded into key prefixes (US-002); "migration" = define new prefix +
  encoder/decoder. Old code deserialises rows it knows about, ignores
  the rest.
- Bucket-level config endpoints (lifecycle / CORS / policy / etc) all
  use the setBucketBlob/getBucketBlob/deleteBucketBlob helper. Reuse
  it; do not write fresh CRUD per endpoint.
- ROADMAP maintenance rule (CLAUDE.md): every commit closing a
  ROADMAP.md item flips the bullet to strikethrough Done format in the
  SAME commit, with closing SHA inline (or `(commit pending)` with a
  follow-up SHA edit). Every commit discovering a new gap adds a new
  entry under the right severity section in the same commit.
- Tests in this cycle that need a TiKV testcontainer: use the
  same testcontainers-go pattern as internal/meta/cassandra/store_integration_test.go.
  Build tag `integration`. PD + TiKV in one container set; ~30 s start.
- Stub-package trick to keep `go mod tidy` from dropping a dep that has
  not yet been used: blank-import the smallest leaf subpackage from the
  vendor (we used `_ "github.com/tikv/client-go/v2/config"` in the
  US-001 stub). Replace with the real import once a story uses the
  txnkv API for round-trip work.
- Compile-time interface assertion `var _ meta.Store = (*Store)(nil)`
  at the bottom of internal/meta/tikv/store.go catches every
  meta.Store interface drift in CI as soon as the interface gains a
  method. Keep it across stories.
- Variable-length string segments in TiKV keys use FoundationDB-style
  byte-stuffing (0x00 → 0x00 0xFF, terminator 0x00 0x00) — preserves
  lex ordering across heterogeneous lengths so prefix range scans work
  correctly. See internal/meta/tikv/keys.md for the full layout. Never
  add length-prefixed encoding to a key — it breaks "all keys starting
  with prefix X" scans.
- Object version-DESC ordering: 24-byte suffix [MaxUint64-ts8-BE]
  [raw-uuid-16]. Inverted ts makes ascending range scan emit latest
  version first (free GET-without-versionId path). Null sentinel UUID
  (timestamp 0) sorts last among versions of a key — gateway resolves
  "?versionId=null" by exact lookup, not scan position.
- Fixed-width integer fields in TiKV keys (partNumber, day-epoch,
  shardID) use big-endian uintN so forward range scans return them in
  ascending numerical order. Never use little-endian or varint there.
- internal/meta/tikv has a kvBackend / kvTxn abstraction (kv.go,
  kv_tikv.go for production *txnkv.Client wrapping, kv_mem.go for
  unit-test in-process backend). Per-section files (buckets.go, ...)
  call s.kv.Begin(ctx, pessimistic) and treat the txn handle as
  Get/Set/Delete/Scan/LockKeys/Commit/Rollback only. Story-by-story
  fills in section files; do not bypass the abstraction. openWithBackend
  is the test-only constructor that injects memBackend.
- Bucket-row mutators (SetBucketVersioning, SetBucketACL,
  SetBucketObjectLockEnabled, SetBucketRegion, SetBucketMfaDelete) all
  go through buckets.go::updateBucket — pessimistic txn that LockKeys +
  Get + mutate + Set + Commit. Plain Set after a previous LWT-style
  CreateBucket would risk read-after-write incoherence; mirror
  CLAUDE.md's Cassandra LWT lesson on the TiKV side.
- meta.Bucket persists as JSON via internal/meta/tikv/codec.go. JSON
  keeps ops debugging cheap, additive (new fields with json:omitempty
  skip an ALTER), and round-trips uuid.UUID + time.Time without custom
  marshallers. Internal storage only — wire formats stay XML.
- Use internal/meta/tikv/buckets.go::prefixEnd helper for any
  "lex starts with prefix" range scan upper bound. Locally implemented
  (mirrors kv.PrefixNextKey) so test code does not pull the txnkv tree.
- Pessimistic txn boilerplate in tikv: defer rollbackOnError(txn, &err)
  immediately after Begin, then assign the named return / outer err on
  every step. Commit is the last statement; if it succeeds err stays
  nil and rollbackOnError is a no-op.
- Pessimistic txns with EARLY-RETURN paths (e.g. SetObjectStorage CAS
  reject returning applied=false, nil) MUST call txn.Rollback() before
  returning. rollbackOnError fires only when err != nil; without an
  explicit Rollback the LockKeys lease leaks for the txn lifetime, and
  in the in-process memBackend it deadlocks the next caller forever.
- Object row payload (codec.go::objectRow) is JSON {meta.Object fields,
  ManifestRaw []byte}. Manifest is encoded via data.EncodeManifest so
  STRATA_MANIFEST_FORMAT (proto vs JSON) flows through the same codec
  Cassandra uses. Reading uses data.DecodeManifest which sniffs the
  first non-whitespace byte → transparent format upgrade.
- Range-scanning all versions of a single user-space key:
  start = ObjectPrefixWithKey(bucketID, key) + 0x00 0x00 (the escaped
  terminator), end = prefixEnd(start). Captures every key whose
  escaped body equals `key` exactly, regardless of the 24-byte
  version-DESC suffix.
- Version-DESC ordering means: range scan with limit 1 over a key's
  prefix returns the lex-first encoded suffix = MaxUint64-ts8 with the
  SMALLEST inverted ts, i.e. the row with the BIGGEST timestamp. Real
  TimeUUIDs sort BEFORE the null sentinel (which has ts=0 → MaxUint64
  inverted). Same as Cassandra `version_id DESC` LIMIT 1 — null row
  surfaces only via explicit ?versionId=null lookup, never as latest
  while a TimeUUID-versioned row exists.
- Listing pagination on TiKV: batched Scan loop with cursor =
  lastKey + 0x00 between batches; exit when Scan returns fewer than
  the batch size or empty. Pass listScanBatchSize=1024. The encoded
  key for the "key marker" boundary is escaped(marker) + 0x00 0x00 (the
  terminator) — and prefixEnd of THAT captures every row of the marker
  key, so for ListObjects (exclusive marker) advance start to that
  prefixEnd; for ListObjectVersions (inclusive marker) advance start
  to escaped(marker) + 0x00 0x00 itself.

## 2026-04-29 - US-005
- Listing on TiKV: ListObjects + ListObjectVersions driven by a single
  ordered range scan over the bucket's object prefix. No 64-way fan-out,
  no heap-merge — version-DESC suffix encoding (US-002) makes the
  lex-first row per key the latest version, so dedup-by-prior-key picks
  out the latest in one pass.
- ListObjects: scan range [ObjectPrefixWithKey(bucketID, prefix),
  prefixEnd(ObjectPrefix(bucketID))). Marker advances start past every
  row of the marker key (markerEnd = prefixEnd(escaped-marker + 0x00 0x00))
  so keys strictly > marker are emitted (matches Cassandra `key > ?` and
  memory `k <= marker continue`). For each key, take the FIRST row — if
  it is a delete marker, hide the key entirely.
- ListObjectVersions: same scan with marker-INCLUSIVE start (matches
  Cassandra `key >= ?` shape used by openVersionCursor). Emit every
  row; first row per key gets IsLatest=true, rest false. Delete markers
  surface — caller decides.
- Delimiter handling identical to memory/Cassandra: in-process group on
  the scan iterator, append to CommonPrefixes the first time we see a
  given delimiter-bounded segment.
- Pagination via batched Scan(start, end, listScanBatchSize=1024) loop.
  cursor advances to lastKey + 0x00 between batches. Truncation sets
  NextMarker (ListObjects) or NextKeyMarker + NextVersionID (versions)
  on the next-not-emitted row, mirroring memory/Cassandra contract.
- Files changed: internal/meta/tikv/list.go (new),
  internal/meta/tikv/store.go (ListObjects/ListObjectVersions stubs
  removed), internal/meta/tikv/store_test.go (10 new test cases —
  basic, hide-delete-markers, latest-per-key, prefix, delimiter, marker
  exclusive, truncation, all-rows-version-listing, version-pagination),
  scripts/ralph/progress.txt, scripts/ralph/prd.json.
- **Learnings for future iterations:**
  - The escape transform's prefix-preservation is what makes the
    one-shot range scan correct: user-space "prefix" of a key always
    yields a byte-prefix of the encoded form, because byte-stuffing is
    pointwise (each byte maps to itself or 0x00 → 0x00 0xFF). Means
    `ObjectPrefixWithKey(bucketID, "foo")` is a true byte prefix of
    every encoded "foo*" object key. Don't add length-prefixed segments
    to keys — the scan would break.
  - NextMarker (and NextKeyMarker/NextVersionID) is set to the
    next-not-emitted row — same as memory/Cassandra. This actually
    LOSES that row on the next page if the gateway naively re-issues
    with marker semantics that exclude marker; that quirk lives across
    all backends and is not US-005's concern. The gateway side
    handles continuation-token rewriting in s3api/server.go.
  - prefixEnd helper (buckets.go) is the canonical "exclusive upper
    bound for lex-starts-with" — reuse it everywhere range scan needs
    a stop key. Mirrors github.com/tikv/client-go/v2/kv.PrefixNextKey.
  - Pagination loop pattern: Scan(cursor, end, batchSize) → if
    len(pairs) < batchSize OR len(pairs) == 0, exit; else cursor =
    lastKey + 0x00 and loop. Keeps memory bounded for huge buckets and
    works against both memBackend and real txnkv.
  - Single-txn snapshot isolation makes one ListObjects call see a
    consistent point-in-time view — concurrent PutObject during the
    scan is invisible (TiKV MVCC). Memory backend's overlay map
    re-reads on each Scan call though, so concurrent writes between
    Scan batches there are visible (acceptable for unit tests; real
    snapshot iso is the contract suite US-013's job).

## 2026-04-29 - US-004
- Object CRUD on TiKV: PutObject / GetObject / DeleteObject /
  DeleteObjectNullReplacement / SetObjectStorage. Listing is deferred
  to US-005.
- PutObject is a single Put under ObjectKey(bucketID, key, versionID).
  Non-versioned writes wipe every prior version row in the same
  pessimistic txn; Suspended-mode null PUT replaces only the prior
  null row, leaving TimeUUID siblings intact.
- GetObject empty version-id is a Scan(prefix, prefixEnd, limit=1) —
  the version-DESC suffix encoding makes the lex-first row = latest
  TimeUUID. Explicit version-id (including the "null" literal) is a
  single Get on ObjectKey.
- DeleteObject mirrors memory/Cassandra: by-version exact delete,
  versioned-no-version writes a delete marker, non-versioned-no-
  version drops the entire key prefix.
- SetObjectStorage uses a pessimistic txn: LockKeys + Get + assert
  storage_class==expected + Set (or unconditional update on empty
  expected). The CAS-reject path explicitly Rolls back so the lease
  is released — see new entry in Codebase Patterns above.
- Files changed: internal/meta/tikv/objects.go (new),
  internal/meta/tikv/codec.go (objectRow + encode/decodeObject +
  uuidFromString), internal/meta/tikv/store.go (object stubs replaced
  by per-section file; ListObjects/ListObjectVersions stubs preserved
  for US-005), internal/meta/tikv/store_test.go (object test cases),
  scripts/ralph/progress.txt, scripts/ralph/prd.json.
- **Learnings for future iterations:**
  - rollbackOnError pattern is fine for "every error path returns the
    sentinel err" but breaks the moment a function has a happy-path
    early return (CAS reject). Audit every pessimistic-txn function
    for "can we exit without setting err?" — if yes, add an explicit
    txn.Rollback() before that return.
  - The contract test caseVersioningSuspendedReplaceNull asserts
    "GetObject empty after marker == ErrObjectNotFound", which holds
    on the memory backend (where Suspended PUT prepends to the version
    slice) but is naturally not satisfied on Cassandra/TiKV (where
    DESC by version_id puts TimeUUID rows ahead of the null sentinel).
    Documented behaviour for now; a parity fix lives in a future cycle
    if the deployment story requires it. US-004 matches Cassandra
    semantics — the contract suite (US-013) will surface the divergence
    centrally if/when we want to converge.
  - objectRow JSON keys are short ("b","k","v","sc",...) to keep TiKV
    row payloads tight; the encoding is internal only so the
    abbreviations are safe. New fields land additively with
    json:"...,omitempty".

    json:omitempty and every reader treats zero values as legal
    defaults. Keep this as the per-row pattern for the per-bucket
