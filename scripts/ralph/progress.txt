## Codebase Patterns
- `internal/data/s3/backend.go` is single-bucket-per-instance today: `Backend{bucket, client, uploader, opTimeout, sseMode, sseKMSKeyID}`. Lift to `Backend{clusters map[string]*s3Cluster, classes map[string]ClassSpec, mu sync.Mutex}` with `s3Cluster{spec, client, uploader}` private struct (matches `rados.Backend.connFor` semantics from US-044).
- `S3ClusterSpec` is BUCKET-LESS: endpoint, region, force_path_style, part_size, upload_concurrency, max_retries, op_timeout_secs, sse_mode, sse_kms_key_id, credentials. Bucket lives on `ClassSpec.Bucket`. Two classes can share one S3 cluster but route to different buckets.
- `ClassSpec{Cluster, Bucket}` — both REQUIRED. NO DefaultCluster fallback. `resolveClass` errors `ErrUnknownStorageClass` if class missing; `ErrClassMissingCluster` if Cluster empty.
- `CredentialsRef{Type string; Ref string}` discriminator: `chain` (SDK default chain — env/IRSA/EC2 metadata); `env:VAR1:VAR2` (read named env vars at client-build time); `file:/path[:profile]` (`awsconfig.LoadSharedConfigProfile`). NEVER plaintext in spec.
- `connFor(ctx, id) (*s3Cluster, error)` lazy-builds the SDK client + uploader on first use under `b.mu`. Mirrors `rados.Backend.connFor`. SDK config resolved at build time per spec.Credentials.
- `s3.New(cfg)` validates BOOT-TIME (fail-fast): every `cfg.Classes[*].Cluster` references a known cluster id; every `cfg.Clusters[*].Credentials` resolves (`chain` → `LoadDefaultConfig` test, `env:` → env var set, `file:` → file exists). Failures bubble up — gateway refuses to start with descriptive error in systemd/k8s status.
- AWS SDK client has no `Close` — for tests/teardown drop reference from `s3Cluster.client` under `b.mu`; GC handles. `*awss3.Client` is goroutine-safe per AWS SDK docs.
- ALL 13 `STRATA_S3_BACKEND_*` envs RETIRED in US-004: `_ENDPOINT`, `_REGION`, `_BUCKET`, `_ACCESS_KEY`, `_SECRET_KEY`, `_FORCE_PATH_STYLE`, `_PART_SIZE`, `_UPLOAD_CONCURRENCY`, `_MAX_RETRIES`, `_OP_TIMEOUT_SECS`, `_MULTIPART_TIMEOUT_SECS`, `_SSE_MODE`, `_SSE_KMS_KEY_ID`. Replaced by `STRATA_S3_CLUSTERS` (JSON array) + `STRATA_S3_CLASSES` (JSON object). NO migration shim, NO legacy compat — deployments break loudly on upgrade.
- `STRATA_S3_CLUSTERS` JSON shape: `[{\"id\":\"primary\",\"endpoint\":\"https://...\",\"region\":\"us-east-1\",\"force_path_style\":false,\"part_size\":...,\"credentials\":{\"type\":\"chain\"}},...]`. `STRATA_S3_CLASSES` JSON shape: `{\"STANDARD\":{\"cluster\":\"primary\",\"bucket\":\"hot-tier\"},\"COLD\":{\"cluster\":\"cold-eu\",\"bucket\":\"glacier\"}}`. Parsers in US-001.
- `ParseClusters` validates: duplicate id → error; empty id/endpoint/region → error; unknown CredentialsRef.Type → error. `ParseClasses` validates: empty Cluster/Bucket per class → error. Cross-validation (class.Cluster references known cluster) happens in `s3.New`, not at parse time — keeps parsers backend-independent.
- Bootstrap workflow: set `STRATA_S3_CLUSTERS` + `STRATA_S3_CLASSES` envs in systemd unit / k8s configmap, start gateway. Adding cluster = update env, rolling restart of N replicas. Load balancer drains replica before stopping → no client-visible downtime.
- `s3test.NewFixture(t)` (US-005) collapses inline test setup. `Fixture{Backend, ClusterID, ClassName, Bucket}` constructor wires ONE cluster spec pointing at httptest server + ONE class mapped to that cluster + random bucket name. Variants `WithClusters(...)`, `WithClass(...)` for multi-cluster tests.
- Race detector: `go test -race ./internal/data/s3/...` MUST stay clean. Watcher-less design — only `b.mu` protects lazy `connFor` mutations of the clusters map's client field.
- Multipart methods (`CreateBackendMultipart` / `UploadBackendPart` / `CompleteBackendMultipart` / `AbortBackendMultipart`) thread the resolved bucket via `resolveClass`. `meta.MultipartUpload.BackendUploadID` carries the upload's cluster + bucket binding for subsequent UploadPart calls — verify shape, persist cluster id alongside if needed.
- Per-cluster SSE knobs (`SSEMode`, `SSEKMSKeyID`) live in `s3Cluster.spec`, not on top-level Backend. Per-class SSE settings remain a future P3 (one class, one cluster, one SSE config).
- NO dynamic registry, NO admin API, NO meta.Store cluster table. Per design decision 2026-05-11 (the prior `ralph/dynamic-clusters` cycle was reverted in `055ef26`). Cluster set is restart-only.
- Existing single-cluster test suite (`backend_test.go`, `multipart_test.go`, `presign_test.go`, `cors_test.go`, `lifecycle_test.go`, `metrics_test.go`) refactored in US-005 to use `s3test.NewFixture` — net LOC reduction; semantics unchanged.

## 2026-05-11 - US-002
- Refactored `internal/data/s3/backend.go` from single-bucket `Backend{bucket, client, uploader, opTimeout, sseMode, sseKMSKeyID}` to multi-cluster `Backend{clusters map[string]*s3Cluster, classes map[string]ClassSpec, mu sync.Mutex, httpClient *http.Client}`. Per-cluster fields moved into private `s3Cluster{spec, client, uploader, opTimeout, multipartTimeout, sseMode, sseKMSKeyID, legacyAccessKey, legacySecretKey}`.
- Added `New(cfg Config) (*Backend, error)` multi-cluster constructor. Validates `cfg.Validate()` (every class's Cluster references a known cluster id; non-empty Clusters + Classes); resolves credentials per-cluster at boot via `validateClusterCredentials` (env vars must be set; file path must Stat; chain defers to SDK first-connect because IMDS calls at boot are too costly).
- Added `connFor(ctx, id)` lazy client+uploader builder under `b.mu`, mirroring rados.Backend.connFor semantics. Cached `*awss3.Client` + `*manager.Uploader` per cluster.
- Added `resolveClass(class)` returning `(cluster, bucket, err)`. Sentinels `ErrUnknownStorageClass` + `ErrClassMissingCluster` mirror the rados shape.
- Added `singleCluster(ctx)` transitional helper picking the lex-lowest configured class for methods that don't yet take a class arg (Put / Get / GetRange / DeleteObject / DeleteBatch / Probe / CORS / Lifecycle / Multipart / Presign / Health). US-003 will swap singleCluster→resolveClass per method.
- Retained `Open(ctx, cfg)` as the legacy single-cluster shim: synthesises one cluster under "default" + one class under "STANDARD" from `cfg.{Endpoint,Region,Bucket,AccessKey,SecretKey,...}` and delegates to New. Existing `STRATA_S3_BACKEND_*`-driven tests + integration paths keep working until US-004 retires them.
- Refactored every existing data-plane file (multipart.go, presign.go, cors.go, lifecycle.go, health.go) to route through `singleCluster` / `connFor` instead of top-level fields. Per-cluster SSE helpers (`s3Cluster.applyPutSSE`, `applyMultipartSSE`, `backendSSEActive`, `manifestSSE`) replaced the old `Backend.*` versions.
- Removed `func New() *Backend` stub constructor — its name now belongs to the multi-cluster builder. Test stubs use `&Backend{}` literal instead; methods on a zero-clusters Backend continue to return `errors.ErrUnsupported` (semantic equivalent).
- New tests in `backend_multicluster_test.go` cover New validation (unknown cluster ref, empty maps), credential resolution (env: rejects unset vars, file: rejects missing path, chain: defers), `resolveClass` happy/unknown paths, `connFor` caching idempotence.
- `go vet ./...` clean. `go test ./...` clean. `go test -race ./internal/data/s3/...` clean.
- **Learnings for future iterations:**
  - `singleCluster` is a transitional helper — US-003 should grep for it and replace every call site with a per-class `resolveClass + connFor` pair. The helper picks the lex-lowest class name so single-config behaviour is deterministic, but it is WRONG semantics for multi-class deployments (always returns the same bucket).
  - `Open(ctx, Config{Bucket,Region,...})` legacy shape coexists with `New(cfg{Clusters,Classes})`. The dispatch keys off `len(cfg.Clusters) > 0 || len(cfg.Classes) > 0`. Tests + integration tests + `internal/serverapp/buildDataBackend` still use Open; US-004 will switch serverapp to call New directly and retire Open + the legacy Config fields in the same change.
  - The legacy Open path synthesises `clusterID="default"` + `className="STANDARD"`. Multi-cluster tests must use New (or pass Clusters+Classes via Config to Open).
  - Plaintext `AccessKey` / `SecretKey` from the legacy path ride on private `s3Cluster.legacyAccessKey` / `legacySecretKey` fields (not exposed via JSON in `S3ClusterSpec.Credentials`). New env-driven path never populates these — only `Open` does. US-004 removes them.
  - `validateClusterCredentials` for `chain` is intentionally a no-op (does NOT call `LoadDefaultConfig.Retrieve()`): IMDS/IRSA web-identity round-trips at boot break in environments where the chain is resolved lazily by the SDK. Trust the SDK to error at first connect. If you want stricter boot-time validation per cluster, add a `Probe` per cluster via `Open(..., SkipProbe: false)` — that's what `probeEachCluster` does.
  - `DataHealth` reports one pool row per (class, bucket) pair instead of one per cluster — multi-class deployments need this granularity so the storage page shows whether every routing target is reachable.
  - `manager.Uploader` is deprecated by `feature/s3/transfermanager` (SDK >=v2 modules), but the deprecation is a soft warning; out of scope for this cycle.
  - macOS `ld LC_DYSYMTAB` warning persists on the host; harmless.

## 2026-05-11 - US-003
- Replaced `singleCluster` routing in the production data-plane methods (`PutChunks`, `Delete(m)`, `PresignGetObject`, plus the four multipart entry-points) with a new `clusterForClass(ctx, class)` helper that runs `resolveClass` + `connFor` in one shot. `PutChunks` routes via the requested class arg; `Delete(m)` and `PresignGetObject` route via `m.Class`; multipart routes via the class on Create and recovers the (cluster, bucket) pair from the opaque handle on UploadPart / Complete / Abort.
- Bumped the multipart handle format from `key\x00uploadID` (2 components) to `cluster\x00bucket\x00key\x00uploadID` (4 components) so subsequent UploadPart / Complete / Abort calls re-resolve routing without re-walking the class table. `encodeHandle` / `decodeHandle` signatures + every call site updated.
- Refactored `s3.Backend.DeleteObject` to delegate to a shared `deleteFromCluster(ctx, c, bucket, oid, versionID)` helper so the class-routed `Delete(m)` and the legacy single-cluster `DeleteObject` (test / admin / bench surface) share one body. `DeleteBatch`, `Put`, `Get`, `GetRange`, `Probe`, CORS, lifecycle, and health stay on `singleCluster` — they are not class-aware surfaces today.
- New `internal/data/s3/multi_cluster_test.go` covers AC: PutChunks routes across two clusters + three classes (class A → eu/bucket-a, class B → eu/bucket-b, class C → us/bucket-c); unknown class returns `ErrUnknownStorageClass`; Delete(m) routes via `m.Class`; the multipart handle carries `(cluster, bucket)` so two classes on different clusters mint distinguishable handles.
- Existing `multipart_test.go` updated: stub-handle literals bumped to the 4-component shape, encode/decode round-trip extends to all four components, and the lifecycle integration test asserts the handle's cluster=`default` + bucket=`strata-test` for the legacy Open() shape.
- ROADMAP `Known latent bugs` gained an entry for `TestThreeReplicaDistribution` — the test failed `-count=20` on `main` with the US-003 changes stashed (flake rate ~30 %), confirming pre-existing flake driven by random UUID hash distribution into 3 FNV-32a buckets. Filed with a deterministic-seed / widen-guard fix sketch.
- `go vet ./...`, `go build ./...`, `go test ./internal/data/s3/...`, `go test -race ./internal/data/s3/...` all clean.
- **Learnings for future iterations:**
  - The multipart handle is the source of truth for routing after Create. The class arg to `CompleteBackendMultipart` exists only because it predates US-003 — the handle already carries everything Complete needs (cluster + bucket + key + uploadID), and the class arg is now used only to stamp `Manifest.Class` on the returned manifest. US-004 / future cleanup can drop it if a callsite audit is worth the churn.
  - Stub-test guards (`len(b.clusters) == 0 → errors.ErrUnsupported`) MUST run before `decodeHandle` in UploadBackendPart / Complete / Abort — otherwise zero-cluster stub tests with hand-crafted handles trip the malformed-handle error path instead of the expected `ErrUnsupported`. The order is: stub-check → decode → connFor.
  - The class-routed Delete(m) path keys off `m.Class` — class transitions via lifecycle (`SetObjectStorage`) update Class atomically, so the routing stays correct across transitions. No need to persist cluster id in `BackendRef` (the design alternative considered + rejected).
  - `clusterForClass(ctx, class)` is the canonical class-routed entry-point. Future data-plane methods that take a storage class should use it (not raw resolveClass + connFor) to inherit stub-tests-friendly `ErrUnsupported` semantics on a zero-cluster Backend.
  - `bucketFromPath(p)` (test helper) extracts the bucket from a path-style URL like `/bucket-a/<key>` — useful pattern for asserting routing on in-process httptest servers. Each capturing server gets its own goroutine-safe counter + lastBucket field.
  - The `manager.Uploader` deprecation warnings are aesthetic; out of scope this cycle (and would be reverted by US-005's fixture refactor anyway).
