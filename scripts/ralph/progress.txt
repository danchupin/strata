## Codebase Patterns
- meta.Store is the single interface every storage backend (memory + cassandra) must implement; adding a method requires updating both backends or builds break (compiler enforces).
- Bucket-scoped blobs use the setBucketBlob/getBucketBlob/deleteBucketBlob helpers in both backends — model new bucket-level resources (grants, encryption, website, …) on this pattern.
- Auth credentials use auth.CredentialsStore (NOT auth.Store) and auth.ErrNoSuchCredential (NOT auth.ErrCredentialNotFound). PRDs sometimes use the wrong names; trust internal/auth/types.go.
- auth.MultiStore (internal/auth/multi.go) composes any number of CredentialsStore backends with a 60s positive+negative TTL cache and Invalidate(accessKey). Use it as the gateway's Middleware.Store; append new backends instead of replacing the static one.
- cmd/strata-gateway/main.go obtains the cassandra session via metaStore.(*metacassandra.Store).Session() — reuse this idiom whenever an auxiliary subsystem needs CQL access without re-opening a session.
- Cassandra cannot UPDATE the buckets table by `id` (PK is `name`); for bucket_id-keyed blobs add a separate table keyed by bucket_id, not a column on `buckets`.
- Cassandra schema evolution: append CREATE TABLE IF NOT EXISTS to tableDDL; append additive `ALTER TABLE ... ADD` to alterStatements. isColumnAlreadyExists handles re-runs.
- s3api handlers accept XML bodies with io.ReadAll(io.LimitReader(r.Body, 1<<20)).
- Tests live in internal/s3api as a black-box `package s3api_test` and use `newHarness(t)` from testutil_test.go.
- IAM-style JSON policy types live in internal/auth/policy. Use policy.Evaluate(doc, principal, action, resource, ctx) where principal is "*" for anonymous, action is e.g. "s3:GetObject", resource is the AWS ARN, ctx is map[string]string of condition keys.
- internal/s3api/access.go centralises policy gating: principalForRequest(r) returns "*" for anonymous, otherwise the AuthInfo.Owner; bucketARN/objectARN build the canonical resource strings; requireAccess(w, r, b, action, arn) returns false (and writes 403) on policy denial. No-policy buckets bypass the gate so behaviour stays unchanged when no JSON policy is set.
- ACL gating lives in the same access.go: requireObjectAccess(w, r, b, key, action) wraps requireAccess (policy) then requireACL (canned + persisted grants). Use it for every object-level dispatch. Owner FC = info.Owner == b.Owner short-circuits both gates; per-object grants override bucket ACL on s3:GetObject only.
- Test-only auth injection: testutil_test.go wraps the api with an X-Test-Principal header sniffer that builds &auth.AuthInfo{Owner: p}. Tests use h.doString(method, path, body, "X-Test-Principal", "owner"|"alt") to simulate non-anonymous callers. Omit the header for anonymous requests.
- IAM admin endpoints (?Action=…) live at the root path (bucket==""). extractIAMAction(r) reads Action from query OR application/x-www-form-urlencoded body; handleIAM gates on auth.FromContext(...).Owner == s3api.IAMRootPrincipal ("iam-root"). When extending IAM, add the action to the dispatch switch in handleIAM and reuse iamParam(r, "Key") to read string params from query OR PostForm. Tests pass principal via X-Test-Principal: iam-root.
- Access keys land in two cassandra tables: access_keys (PK access_key) for SigV4 lookup, iam_access_keys_by_user (PK (user_name, access_key)) for ListAccessKeys. Writes use a logged BATCH so both stay in sync; reads come from access_keys directly. Memory backend keeps a single accessKeys map keyed by AccessKeyID. memory.CredentialStore wraps the memory backend so memory-meta gateways have a usable SigV4 credential lookup (mirrors cassandra.CredentialStore). Server.InvalidateCredential is the hook that DeleteAccessKey calls to drop the MultiStore cache entry; main.go wires it to multi.Invalidate. Tests that need a fully signed request use the in-file SigV4 helper in iam_access_keys_signed_test.go (canonical request → derive signing key → HMAC).
- Temporary credentials (STS AssumeRole) carry Credential.SessionToken; middleware compares the request's x-amz-security-token to it after signature passes — mismatch → ErrInvalidToken. auth.STSStore implements CredentialsStore so STS keys flow through MultiStore like static/cassandra/memory creds; MultiStore skips caching creds with non-empty SessionToken so revoke/expiry take effect immediately. STSStore.SetClock(now func()) drives deterministic expiry in tests.
- Object-level checksums (US-009): meta.Object.Checksums map[string]string round-trips through both backends (cassandra column `checksums map<text,text>` added via additive ALTER). Server-side verification lives in internal/s3api/checksum.go: parseRequestChecksums(r) reads every x-amz-checksum-<algo> header for algos in supportedChecksumAlgorithms (CRC32, CRC32C, SHA1, SHA256, CRC64NVME), checksumWriter fans body bytes to all hashers via io.TeeReader on the put path, verifyChecksums compares after PutChunks and writeChecksumHeaders echoes them on GetObject/HeadObject. Mismatch returns 400 BadDigest and deletes the freshly-written chunks. CRC-64/NVME is vendored locally (newCRC64NVME) since the stdlib lacks it.
- Multipart checksums (US-010): meta.MultipartPart.Checksums map[string]string is persisted through SavePart / surfaced by ListParts in both backends (cassandra ALTER multipart_parts ADD checksums map<text,text>). uploadPart re-uses the parseRequestChecksums + io.TeeReader(checksumWriter) + verifyChecksums chain from US-009; mismatch deletes the data manifest and returns 400 BadDigest. completeMultipart calls ListParts BEFORE CompleteMultipartUpload to grab per-part Checksums (the meta-level Complete deletes parts on success), then composeMultipartChecksums computes the AWS composite (base64(hash(concat raw digests))-N) for every algo present on every requested part. The composite is set on obj.Checksums (so GetObject/HeadObject echo it) and emitted on CompleteMultipartUploadResult as both response headers and ChecksumCRC32/SHA256/etc. XML elements (omitempty).
- SSE-S3 metadata (US-011): meta.Object.SSE + meta.MultipartUpload.SSE strings round-trip through both backends (cassandra ALTER objects ADD sse text + ALTER multipart_uploads ADD sse text). Bucket-level default uses the standard setBucketBlob/getBucketBlob pattern (table bucket_encryption, ErrNoSuchEncryption sentinel). internal/s3api/encryption.go centralises validateSSEAlgorithm (AES256 ok; aws:kms/aws:kms:dsse → 501 NotImplemented; else 400) and resolveSSE(r,b) — request header wins, bucket default fills in, no config => empty. resolveSSE runs BEFORE PutChunks so kms/bogus headers reject the request without writing chunks. SSE flows: putObject sets obj.SSE, initiateMultipart sets mu.SSE, uploadPart + completeMultipart echo header, completeMultipart copies mu.SSE to obj.SSE. GetObject/HeadObject emit x-amz-server-side-encryption when obj.SSE is non-empty. Errors live in errors.go: ErrNoSuchEncryption (404 ServerSideEncryptionConfigurationNotFoundError), ErrInvalidEncryptionAlgorithm (400), ErrKMSNotImplemented (501).

## 2026-04-25 - US-002 (IAM-style policy evaluator package)
- New package internal/auth/policy with:
  - Decision (Allow/Deny) and constants EffectAllow/EffectDeny.
  - Document, Statement, Principal types; StringOrSlice / StringSlice unmarshallers cope with the AWS "string OR list" JSON shape.
  - Principal accepts the bare string "*" or the object form {"AWS": "arn..." | ["arn...", ...]}.

- **Learnings for future iterations:**
  - AWS's "string OR list" JSON convention is widespread (Action, Resource, Principal.AWS, Condition values). Centralised the unmarshaller (unmarshalStrings) so future condition-key arrays reuse it.
