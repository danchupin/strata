## 2026-05-06 - US-003 Multipart composite checksum CRC32 / CRC32C / CRC64NVME
- Implemented:
  - `internal/s3api/checksum.go`: extended `composeMultipartChecksums` to take a `checksumType` arg. SHA1/SHA256 stay on the COMPOSITE shape (`BASE64(HASH(concat(raw_i)))-N`). CRC32 / CRC32C / CRC64NVME branch on `checksumType="FULL_OBJECT"` and combine per-part CRCs into the whole-stream CRC via standard zlib-style `crc32_combine` / `crc64_combine` math (no `-N` suffix). When `checksumType` is empty or `COMPOSITE`, CRC family keeps the legacy concat-then-hash shape (still passes today's tests + boto3 default).
  - New helpers in the same file: `combineCRCParts(algo, raws, parts)`, `crc32Combine(poly, crc1, crc2, len2)`, `crc64Combine(poly, crc1, crc2, len2)`, plus their `gf2MatrixTimes/Square` workhorses. Polynomials use the reversed/right-shift convention (`0xedb88320` IEEE, `0x82f63b78` Castagnoli, `crc64NVMEPoly` for NVME) so they line up with `hash/crc32` and our `crc64NVMEHash` Sum values without an extra inversion.
  - `checksumPart` grew a `Size int64` field — required to feed `len2` into the combine math. Multipart Complete site already has this on `meta.MultipartPart`, so the wire is `requested = append(requested, &checksumPart{Checksums: sp.Checksums, Size: sp.Size})`.
  - `multipart.go::completeMultipartUpload` now passes `mu.ChecksumType` into `composeMultipartChecksums`. The XML response and HEAD/GET (with `ChecksumMode=ENABLED`) echo the FULL_OBJECT shape unchanged via the existing `o.Checksums` + `o.ChecksumType` plumbing.
  - Added `internal/s3api.CombineCRCPartsForTest` (test-only export) to drive the FULL_OBJECT combine math against the AWS-published s3-tests vectors without spinning up the HTTP harness.
- Files changed:
  - `internal/s3api/checksum.go`
  - `internal/s3api/multipart.go`
  - `internal/s3api/multipart_checksum_test.go` — added `TestFullObjectCRCS3TestVectors` (locks the published `test_multipart_use_cksum_helper_crc{32,32c,64nvme}` per-part + composite values for the 5 MiB / 5 MiB / 5 MiB shape) and `TestMultipartFullObjectCRCCombine` (HTTP round-trip for all 3 CRC algos with non-uniform part sizes, asserts no `-N` suffix and ChecksumType=FULL_OBJECT on Complete + HEAD).
- Learnings for future iterations:
  - **AWS FULL_OBJECT for CRC family = `crc(concat(bytes))`, NOT `crc(concat(raw_per_part_crcs))`.** PRD AC #2 said "may be concat-then-CRC; verify against AWS docs at story start" — that's the COMPOSITE shape, not what s3-tests assert. The s3-test calls explicitly set `ChecksumType="FULL_OBJECT"` for CRC* and the published composite values have no `-N` suffix; reproducing them required the standard `crc32_combine` formula (zlib's gf2 matrix square+times pattern) and per-part sizes. This is the third PRD-vs-s3-tests divergence in this cycle (US-002 was the same pattern — trust the s3-test source).
  - **Per-part Size is required for combine math.** Don't drop `Size` from `checksumPart`. Even though the existing COMPOSITE path doesn't read it, the FULL_OBJECT branch needs `len2` for every i>0 step. Sourced from `meta.MultipartPart.Size`, already plaintext (SSE-aware in `uploadPart`).
  - **Reversed polynomial convention.** Go's `hash/crc32` and our CRC-64/NVME implementation operate on the right-shift / reflected polynomial. zlib's `crc32_combine` does too — feed `0xedb88320` (IEEE), `0x82f63b78` (Castagnoli), `crc64NVMEPoly` (NVME) into the `odd[0]` slot. Don't flip with the forward-poly form (`0x04c11db7` etc.) — combine results will be off by the reflection.
  - **SHA + FULL_OBJECT is unreachable in normal flow.** AWS rejects `ChecksumType=FULL_OBJECT` + SHA on Initiate. We fall back to COMPOSITE shape for SHA in the helper anyway, so a misbehaving client still gets a well-formed (if technically wrong-typed) response.
  - **s3-tests vector locking pattern**: when a story closes a wire-shape gap, lock the published vectors as a unit test. The 3-line `cases := []struct{...}` table with per-part b64 + sizes + expected composite is cheap; it catches polynomial / reversal / size-handling regressions without needing the live s3-tests run.

When refactoring multipart-Complete-style ops in the future: if there's a status-flip-via-LWT and a payload-validation step, validation MUST come first so the flip never happens when the request itself is invalid.

On unix, `os.Setenv` with an empty value writes an empty string into the env block and `os.Getenv` returns `""` with the second return `true`. That is exactly what the koanf env provider sees and is what triggers the empty-value callback. Don't use `t.Setenv` to "unset" — use the `clearEnv` helper that calls `os.Unsetenv`.
  - **TOML default for empty-env regression must come via `STRATA_CONFIG_FILE`, not just `defaults()`.** The defaults() in-memory path pre-loads via the `structs.Provider` BEFORE the env merge — so it always passes the `30s` assertion regardless. Routing through the file-load proves both the file provider AND the env provider treat empty values consistently.
  - **Cassandra raw SELECT in the test file:** `store.Session() *gocql.Session` is the public hook; `cassandra_test` package has it. To convert `uuid.UUID` (google) → `gocql.UUID`, copy bytes via `var bg gocql.UUID; copy(bg[:], u[:])`. Don't reach for the private `gocqlUUID()` helper — exporting it just to be tested would widen the API surface for no production benefit.
  - **`ALLOW FILTERING` is fine for single-row test fixtures** but NEVER in production code. The cassandra-gotchas section of CLAUDE.md flags this; the test comment also calls it out so it doesn't get copied into a hot path by a future refactor.
  - **Separate test functions vs shared container:** added a second `tccassandra.Run` rather than refactoring `TestCassandraStoreContract` to share. ~30s extra cold boot is well under the 15-min CI integration budget; the alternative (TestMain or sync.Once package var) would entangle test lifecycles. Reuse this pattern when adding focused integration tests.
  - **`tccassandra.Run` second-test discovered keyspace name uniqueness matters** — the contract test uses `strata_<seq>` with an atomic counter; `TestCassandraNullSentinelOnDisk` uses a fixed `strata_nullsentinel` since it's a single-keyspace test. Don't reuse a numbered keyspace name across separate containers — they don't share state but the visibility/labeling stays clearer with distinct names.
