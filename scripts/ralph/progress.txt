## Codebase Patterns
- 3-story cycle closing 1 P3 ROADMAP entry ("Drain progress UI shows manifest counts instead of physical chunks"). Operator-visible fix: `<DrainProgressBar>` confused operators showing "0 Migrating" during the GC-grace window between manifest CAS rewrite and physical chunk deletion. Surface physical chunk count as the primary metric; manifest count moves to collapsible detail.
- **Manifest vs physical semantics**: rebalance worker rewrites `BackendRef.Cluster` via `meta.Store.SetObjectStorage` CAS, then enqueues old chunks for GC. Manifest count drops to 0 immediately on CAS success; physical pool still has chunks until `STRATA_GC_GRACE` (default 5 min) elapses + next gc worker tick. The operator-perceived "drain" is physical cleanup, not manifest scan.
- **Existing US-006 safety gate unchanged**: `deregister_ready` already ANDs `total_chunks==0 && gc_queue_pending==0 && no_open_multipart==0`. Physical chunk count is OBSERVABILITY signal, NOT safety predicate. Adding `physical_chunks_on_cluster==0` to the gate would block deregister on backends without physical probe (S3, memory).
- **New probe interface** (US-001): `ClusterObjectCountProbe` in `internal/data/backend.go`, separate from existing `ClusterStatsProbe` (signature unchanged for rebalance-worker callsite back-compat). RADOS implementation in `internal/data/rados/health.go` iterates `b.classes` filtered by clusterID, opens ioctx per (cluster, pool, ns), calls `ioctx.GetPoolStats()`, sums `stat.Num_objects` across pools. Reuses existing path used by `DataHealth`. Returns `(0, err)` on per-pool error — no silent zero-fill.
- **Why GetPoolStats() not `ceph df` JSON**: existing `DataHealth` codepath already uses GetPoolStats per pool; reusing same path avoids JSON field-name verification risk (`objects` vs `num_objects` etc).
- **New cache** (US-001): `internal/data/placement/clusterstats_cache.go` (location parity with `draincache.go`). `ClusterStatsCache` with `TTL: 10 * time.Second` constant. `Get(clusterID) (bytes, objects int64, ok bool)` + `Set(clusterID, bytes, objects int64)`. sync.RWMutex. Lazy refresh: drain-progress builder calls Get → on miss/expired calls both probes once + Set + returns. Cache invalidation: TTL only. No drain/undrain event invalidation (underlying ceph df doesn't change at those events).
- **drain-progress response** (US-001): three new additive fields in `internal/s3api/admin_clusters.go::buildDrainProgressResponse` (or wherever the builder lives — verify): `physical_chunks_on_cluster *int64` (null if no ClusterObjectCountProbe), `physical_bytes_on_cluster *int64` (null if no ClusterStatsProbe), `gc_queue_pending int` (explicit counter; len of existing `ListChunkDeletionsByCluster(cluster)` call already in US-006 safety gate).
- **Wire shape additivity**: existing fields (`chunks_on_cluster`, `bytes_on_cluster`, `not_ready_reasons`, `deregister_ready`) preserved verbatim. New fields are JSON-additive — old consumers ignore unknown fields. `chunks_on_cluster` semantic shifts (primary on null-physical backends, secondary on RADOS); documented in migration note.
- **New counter** (US-001): `strata_drain_progress_probe_errors_total{cluster, probe}` with label `probe ∈ {stats, object_count}`. Increments on probe error; response still succeeds (cache fallback or null).
- **UI 3-state machine** (US-002): primary = physical when non-null, else manifest. States: Migrating (primary > 0 && manifest > 0) / Awaiting GC cleanup (primary > 0 && manifest == 0) / Ready to deregister (existing flow). Collapsible detail expands to show `Manifest chunks: X, GC queue: Y, Physical bytes: Z B`.
- **No precise ETA** (per design review): static tooltip on Awaiting GC state: `Physical delete completes after STRATA_GC_GRACE elapses (~5m default) plus the next gc worker tick.` Precise per-deploy ETA requires new `/admin/v1/gc-config` endpoint + formula `eta_min = grace_min + ceil(gc_queue / (batch_size × shards) × interval_min)` — parked as P3 follow-up in US-003.
- **Null-physical UX** (US-002): no transient state distinction (probe-err vs unsupported vs cache cold all render same). One rendering path. Manifest count as primary + `(physical count unavailable on this backend)` tooltip. S3 + memory backends hit this path; RADOS during probe error also hits if cache cold.
- **Smoke harness** (US-003): `scripts/smoke-drain-progress-ui.sh` (NEW script, NOT extending existing `smoke-drain-transparency.sh` — cleaner separation). Uses `STRATA_REBALANCE_RATE_MB_S=1` to slow migration + `STRATA_GC_GRACE=60s` to shorten grace; 300 chunks via PUT with `Placement={default:1, cephb:1}` mode=weighted; poll every 3 s; 5 min timeout. Asserts ALL THREE states observed during one drain cycle.
- **Smoke env**: throttle + shortened grace are smoke-only — document in script comment + the migration note that these don't apply to prod defaults. Prod default rebalance rate (100 MB/s) + GC grace (5 min) would either pass migration too fast or stretch the smoke past CI timeout.
- **End-to-end discipline** (per `feedback_cycle_end_to_end.md`): walkthrough covers happy path (drain cephb evacuate → 3 states observed → green chip) + edge cases (null physical / S3-memory backends, count divergence from direct-PUT on cephb, MonCommand transient fail, gateway restart mid-drain, multi-replica HA, hundreds-of-millions-of-objects pool). State Truth Tables document UI state derivation table + probe capability matrix + ClusterStatsCache lifecycle. Cache Invalidation Ledger: ONE new cache (ClusterStatsCache 10s TTL per-replica, TTL-only invalidation). Safety Claims Preconditions: existing US-006 deregister-ready gate unchanged; back-compat tested via Playwright null-physical mock; per-poll cost bounded by 10s cache hit ratio.
