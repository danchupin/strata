## Codebase Patterns

(none yet — populate as US-001..US-020 land. Seed entries below mirror what
prior cycles found useful; remove/replace if stale for this cycle.)

- internal/meta.Store interface is the contract; both Cassandra and TiKV
  must implement it in lockstep. Memory backend is the test reference.
  Use internal/meta/storetest/contract.go as the parity oracle.
- Cassandra LWT lesson: any UPDATE on a row that may be read with quorum
  AFTER a previous LWT must itself be LWT (IF EXISTS). Read CLAUDE.md
  "Cassandra gotchas" before touching meta semantics. The TiKV equivalent
  is "any update that needs read-after-write coherence must use a
  pessimistic txn, not a plain Put".
- Schema migrations are additive. Cassandra: alterStatements with
  IF NOT EXISTS column adds, swallowed-error helpers. TiKV: schema is
  encoded into key prefixes (US-002); "migration" = define new prefix +
  encoder/decoder. Old code deserialises rows it knows about, ignores
  the rest.
- Bucket-level config endpoints (lifecycle / CORS / policy / etc) all
  use the setBucketBlob/getBucketBlob/deleteBucketBlob helper. Reuse
  it; do not write fresh CRUD per endpoint.
- ROADMAP maintenance rule (CLAUDE.md): every commit closing a
  ROADMAP.md item flips the bullet to strikethrough Done format in the
  SAME commit, with closing SHA inline (or `(commit pending)` with a
  follow-up SHA edit). Every commit discovering a new gap adds a new
  entry under the right severity section in the same commit.
- Tests in this cycle that need a TiKV testcontainer: use the
  same testcontainers-go pattern as internal/meta/cassandra/store_integration_test.go.
  Build tag `integration`. PD + TiKV in one container set; ~30 s start.
