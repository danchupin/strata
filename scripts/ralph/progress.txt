## 2026-05-31 - US-003 Reshard worker actually rewrites rows + cleanup
- Cassandra now implements the optional meta.ReshardMigrator interface (new in internal/meta/store.go). MigrateReshardKey(bucketID,key) relocates EVERY version row of a key from its source shard (fnv%active) to its target shard (fnv%target) via INSERT … IF NOT EXISTS, then DELETEs the source orphan. Idempotent + crash-safe: IF-NOT-EXISTS never clobbers a newer concurrent target write; unconditional source DELETE is a no-op once gone; re-run after a partial crash converges. Returns rows actually moved (applied==true only).
- New cassandra helpers (internal/meta/cassandra/reshard.go): scanAllVersionsAtShard (iter over all versions of a key in one shard, full column projection, closes iter on every path) + insertObjectRowAtShard (full-column INSERT mirroring PutObject EXACTLY, optional IF NOT EXISTS via MapScanCAS, NO bucket_stats bump — a move conserves totals) + objectSelectCols const. Refactored scanObjectLimit1/scanObjectByVersion onto the const (no semantic change; reviewer verified byte-identical column order).
- internal/reshard/reshard.go runJob rewritten: walks the in-flight union ListObjectVersions key-by-key, drives MigrateReshardKey per DISTINCT key (dedup via lastMigrated since a key's versions are adjacent in key-ASC order) under a LastKey watermark, with a forward-progress guard for the inclusive (key>=marker) over-large-key edge. RunOnce still calls CompleteReshard AFTER runJob drains every key → cleanup-before-flip guaranteed (flip only on nil-error walk completion). Stats.ObjectsCopied now = real rows moved (0 on shard-agnostic backends).
- Files: internal/meta/store.go, internal/meta/cassandra/reshard.go, internal/meta/cassandra/store.go, internal/reshard/reshard.go, internal/reshard/reshard_test.go, internal/meta/cassandra/store_integration_test.go, CLAUDE.md, ROADMAP.md, scripts/ralph/prd.json.
- Proof: integration TestCassandraReshardWorkerMovesRows (64→128, 300 unversioned + one 3-version diverging key; asserts ObjectsCopied==diverging+3, post-flip every key point-GETs, LIST emits each once, all 3 versions reachable by id + latest ordering) — RED against the row-walking skeleton (post-flip the diverging half 404s from empty target partitions), GREEN after. Worker unit tests rewritten onto a recordingStore meta.ReshardMigrator fake so the dedup-per-key + watermark-resume assertions stay genuine on the shard-agnostic memory backend.
- strata-correctness-reviewer: no blocking issues; all 5 requested verifications pass (column-list parity, INSERT/arg parity, MapScanCAS not ScanCAS, no iter leak, cleanup-before-flip guaranteed).
- **Learnings for future iterations:**
  - The reshard worker is GENERIC over meta.Store; backend-specific row movement is an OPTIONAL interface (meta.ReshardMigrator), type-asserted in runJob — same pattern as meta.RangeScanStore. Memory/TiKV deliberately DON'T implement it → the worker is an immediate-complete no-op there (this already gives US-004 the memory/TiKV no-op semantics for the worker path; US-004 still reworks TiKV's StartReshard job-queueing).
  - To copy a Cassandra objects row faithfully across partitions, scan→Object→re-INSERT is the GET→PUT round-trip the read/write paths already preserve. insertObjectRowAtShard MUST keep its INSERT column list in lockstep with PutObject's; objectSelectCols MUST match scanObjectQuery's Scan order. A column-order drift silently corrupts reads — there is now a shared const for the SELECT half but the INSERT half is still duplicated (PutObject not refactored, to avoid hot-path risk).
  - ListObjectVersions marker is INCLUSIVE (openVersionCursor: key >= ?). A worker loop that resumes with Marker=lastKey re-lists that key; dedup per-key + a forward-progress guard (cursor==startMarker → cursor+="\x00") are required so a key with > pageLimit versions can't pin the marker forever. MigrateReshardKey moves ALL of a key's versions on first sight, so stepping past the key is safe.
  - cleanup-before-flip is the load-bearing ordering: MigrateReshardKey copies-THEN-deletes the source inline, so once the walk drains, the source layout holds no diverging row; CompleteReshard (LWT IF EXISTS) runs only after. Copy-first (not delete-first) is deliberate — delete-first opens a window where the key is in NEITHER partition (breaks US-002 stable-key-set). The trade is a named narrow race: a concurrent specific-version DELETE in the copy window can resurrect that version (ROADMAP P2, Known latent bugs).
  - A single in-process store still can't discriminate the post-reshard-404 bug for point-GET in a same-process test (writer+reader agree). The integration test seeds at DefaultShardCount=64, StartReshard(128), runs the REAL worker.RunOnce, then asserts post-flip point-GET — the skeleton (no row move) leaves the diverging half in shards [0,64) while fnv%128 addresses [64,128) → 404. That's the discriminator.

The only per-key <Error> rows are MalformedXML (empty key), AccessDenied (object lock), InternalError (other delete err). So "mixed success/NoSuchKey" in R1 == existing+missing both succeed, not an error row. Don't assert a NoSuchKey error row — it won't exist.
  - The 1000-key cap is `len(req.Objects) == 0 || len(req.Objects) > 1000` -> ErrMalformedXML (delete_objects.go:61). So 1000 is accepted, 1001 rejected, 0 rejected. Same MalformedXML code as a parse failure.

  - Versioned response shaping (delete_objects.go:124-132): request-VersionId echoed into entry.VersionID first; then if versioned && o.IsDeleteMarker -> DeleteMarker=true + DeleteMarkerVersionId=wireVersionID(o) (and VersionId stays the request value, "" for an unversioned-style delete); else if explicit versionId -> entry.VersionID=wireVersionID(o) (actual removed version). Deleting a delete-marker BY its versionId also yields a DeleteMarker=true row.
