- **Learnings for future iterations:**
  - The two-backend symmetry rule still holds: `Backend.PutChunks` guard must live in BOTH rados (per-chunk `pickCluster` closure) and s3 (`clusterForClassStrict` pre-flight). The two paths diverge slightly because s3 resolves the bucket on the picked cluster before the connection while rados maps directly to OIDs — fold the refusal into the picker callback that already knows the resolved class-default cluster id.
  - In-flight multipart graceful semantics are a property of the handle shape, not the strict-mode code: as long as UploadPart/Complete/Abort `decodeHandle()` the cluster id and route through it (no re-call into `PickClusterExcluding`), drains never break open uploads. New backends that participate in placement MUST persist cluster id in the multipart handle the same way.
  - `data.DrainingClustersFromContext` returns a `map[string]bool` populated by the gateway hot path via `s3api.Server.dataCtxForPut`. The refusal hook only needs the resolved class-default cluster id; the picker result is "" or a peer (not a drained id). Don't widen the refusal to "any draining target" — that would break the `TestPutChunksDoesNotRefuseAlternateCluster` rail (picker correctly skips drained, lands on peer).
  - WARN+ignore for legacy env: the boot-time `slog.Warn` happens inside `Config.validate()` so it fires regardless of which command (`strata` or `strata-admin`) parses the config. The line is single-shot — koanf reads env once at Load(); subsequent in-process reads of the env are not surveilled.
  - Counter rename is a hard wire-shape change: any operator with a dashboard panel querying `strata_putchunks_refused_total{reason="drain_strict"}` must update to `drain_refused` post-deploy. Documented in placement-rebalance.md's "Drain refusal semantics" subsection so the migration note rides into the rendered docs.
  - The `docs/site/public/` HTML is built artefact and intentionally NOT edited per-PR — `make docs-build` regenerates it. Stale `drain_strict` strings in `docs/site/public/` will disappear on the next build.
