## Codebase Patterns

(seed entries for this cycle. Replace/remove as US-001..US-005 land.)

- internal/auth/streaming.go (the existing streaming decoder) decodes
  chunk framing via bufio.Reader: ReadString('\n') for size+optional
  ;chunk-signature header, ParseInt(size, 16, 64) for hex length, then
  forwards `size` bytes verbatim to the consumer. Today it does NOT
  recompute the chain — US-002 changes the read loop to a
  buffer-then-validate shape (read full chunk → SHA-256 → expected sig
  via chunkSigner.Next → hmac.Equal → forward on match, error on
  mismatch).
- internal/auth/sigv4.go::deriveSigningKey(secret, date, region,
  service) returns the 32-byte SigV4 derived signing key — the same
  key the outer canonical-request hash used. US-001's chunkSigner
  consumes this key directly. computeSignature is the reference for
  HMAC-SHA256(key, string-to-sign) shape.
- internal/auth/middleware.go:76-77 wraps r.Body with
  newStreamingReader when r.Header.Get('x-amz-content-sha256') ==
  streamingBody constant. US-002 extends this wire point to also pass
  the seed-sig + signing-key + scope into the decoder so the chain
  validator has everything it needs.
- AuthInfo struct in internal/auth/types.go currently exposes
  identity-only. US-002 adds either an unexported field (cycle of
  staleness shorter) or a context.WithValue carrier for the seed
  signature + signing key + scope. Pick one; document in a one-line
  comment so future readers see the choice.
- hmac.Equal([]byte, []byte) is the Go stdlib idiom for constant-time
  authentication-tag comparison. Use it for chunk-signature compare
  (US-001 + US-002). Plain == on hex strings is a timing footgun.
- Streaming-shape rule: peak buffer is ONE chunk (≤16 MiB cap), never
  the whole body. For typical SDK output the per-chunk buffer is
  ≤8 KiB. The "no full-body buffer" invariant is preserved.
- aws-chunked-trailer (US-003) is detected via x-amz-trailer request
  header. ROADMAP "Known latent bugs" already documents the older
  decoder breaking on trailer-format with a confusing error; US-003
  promotes that to a clean 501 NotImplemented response. This is
  detection-only; real trailer-format support is a separate future
  PRD.
- ROADMAP maintenance rule (CLAUDE.md): every commit closing a
  ROADMAP item flips the bullet to ~~strikethrough~~ Done format in
  the same commit, with closing SHA inline (or `(commit pending)`
  with a follow-up SHA edit). Every commit discovering a new gap adds
  a fresh entry under the right severity section in the same commit.
- s3-tests baseline lives in scripts/s3-tests/README.md as an
  append-only history. US-001 captures the cycle-start headline in
  its commit message footer; US-005 re-runs scripts/s3-tests/run.sh
  and asserts the new headline >= the captured baseline. Chunk
  validation MUST NOT regress upload paths — a regression blocks
  cycle close.
- AWS test vectors for the chain are transcribed verbatim into Go
  literals in internal/auth/streaming_test.go. Inline copy so the
  test stays offline-runnable across AWS doc URL rotations. Source:
  https://docs.aws.amazon.com/AmazonS3/latest/API/sigv4-streaming.html
  "Example Calculations".
- No metadata schema changes; no Cassandra/TiKV/memory backend
  touches; no RADOS/S3-backend touches. Pure auth-layer fix scoped
  to internal/auth/* + internal/s3api/sigv4_streaming_test.go +
  ROADMAP.md + docs/security/.

## 2026-05-01 - US-002
- Rewrote streamingReader (internal/auth/streaming.go) to buffer-then-
  validate: each chunk is fully read into a per-chunk buffer (≤16 MiB
  cap via maxChunkPayload), SHA-256-hashed, chain-validated against
  ;chunk-signature=<hex> via chunkSigner.Next + hmac.Equal, and only
  then forwarded to the consumer. Mismatch → ErrChunkSignatureMismatch
  + zero bytes leaked.
- Added exported sentinel ErrChunkSignatureMismatch and helper
  IsChunkSignatureMismatch(err) for type-safe handler-side translation.
- Extended newStreamingReader signature to (body, seedSig, signingKey,
  isoDate, scope). Plumbing: pass-through args, no AuthInfo field, no
  ctx stash — documented inline in middleware.go (US-002 comment).
  Middleware computes signingKey via deriveSigningKey + scope via
  credentialScope, hands them straight in.
- Helper s3api.MapBodyError(err) returns (ErrSignatureDoesNotMatch,
  true) when errors.Is(err, auth.ErrChunkSignatureMismatch). Wired
  into putObject (internal/s3api/server.go), uploadPart's PutChunks
  path AND BackendUploadID pass-through (internal/s3api/multipart.go).
- Tests added in internal/auth/streaming_test.go:
  - TestStreamingReader_ValidChain — pristine 3-chunk body decodes to
    concatenated payload with no error.
  - TestStreamingReader_MutatedChunk_Rejects — flips one byte of
    chunk-2's payload; verifies ErrChunkSignatureMismatch AND that
    chunk-2's bytes never reach the consumer (buffer-then-validate
    invariant).
  - TestStreamingReader_MutatedTrailer_Rejects — flips trailer's
    chunk-signature value; verifies the empty-chunk's chain check
    catches truncation attacks.
  - TestStreamingReader_OversizedChunk_Rejects — size header claiming
    32 MiB returns a framing error (NOT signature mismatch) without
    OOM-allocating the payload.
- Test helpers buildChunkedBody + intToHex craft aws-chunked framing
  with valid chain signatures so future stories (US-004 integration
  test) can reuse the pattern.
- Files changed: internal/auth/streaming.go, internal/auth/middleware.go,
  internal/auth/streaming_test.go, internal/s3api/errors.go,
  internal/s3api/server.go, internal/s3api/multipart.go.
- **Learnings for future iterations:**
  - Plumbing choice: pass-through constructor args (NOT AuthInfo
    field, NOT ctx stash). The chain state is needed only at
    streamingReader construction; nothing downstream re-derives it.
    Adding fields to AuthInfo would have leaked auth-internal state to
    every handler.
  - Empty trailer chunk MUST be chain-validated. Current AWS spec says
    sig(final-empty) is part of the chain; skipping it lets an
    attacker truncate the body and forge the final marker. The reader
    treats size==0 as "validate signature, then EOF".
  - aws-chunked framing has a trailing CRLF AFTER the payload of each
    non-empty chunk. Empty trailer chunk format is "0;chunk-signature=
    <hex>\r\n\r\n" — but the reader stops after consuming the size
    line and the trailer's CRLF before the payload, since size==0 has
    no payload and no trailing payload-CRLF.
  - Helper MapBodyError lives in s3api/errors.go (not auth package)
    because the APIError type is s3api-internal. Body-error translation
    is the s3api layer's responsibility — auth surfaces a typed Go
    error, s3api maps it to the S3 wire format.
  - intToHex must produce LOWERCASE hex (matches AWS aws-cli output).
    fmt.Sprintf("%x", n) also works but tiny custom impl avoids
    dragging fmt into the test signing path.
  - Existing tests in internal/auth/* don't construct streaming
    bodies, so no callsite updates were needed beyond middleware.go.
    Future tests that build streaming PUT requests can reuse the
    buildChunkedBody helper from streaming_test.go (move to a shared
    test helper file if needed by US-004).

## 2026-05-01 - US-003
- Added auth.ErrTrailerFormatUnsupported sentinel (internal/auth/types.go)
  + streamingUnsignedTrailer constant + isStreamingSentinel helper
  (internal/auth/sigv4.go).
- internal/auth/middleware.go validateHeader now short-circuits with
  ErrTrailerFormatUnsupported when X-Amz-Trailer is non-empty AND
  x-amz-content-sha256 is one of the streaming sentinels (signed-chunked
  STREAMING-AWS4-HMAC-SHA256-PAYLOAD or unsigned-chunked-with-trailer
  STREAMING-UNSIGNED-PAYLOAD-TRAILER). Detection fires AFTER signature
  validation so unauth probes can't enumerate features. When
  X-Amz-Trailer is empty/missing, the pre-existing streaming reader path
  runs unchanged.
- s3api.WriteAuthDenied (internal/s3api/errors.go) translates
  errors.Is(err, auth.ErrTrailerFormatUnsupported) → s3api.ErrTrailerNotImplemented
  (HTTP 501, <Code>NotImplemented</Code> + the PRD-mandated trailer-format
  message body).
- Tests:
  - internal/auth/middleware_test.go (new) — TestMiddlewareTrailerFormat_Rejected
    (signed-chunked + trailer → 501-path error), …_UnsignedTrailerSentinel
    (the unsigned-trailer variant), and …_NoTrailerHeader_PassesThrough
    (streaming PUT without trailer header → no error, info populated).
  - internal/s3api/errors_test.go (new) — TestWriteAuthDenied_TrailerFormat
    asserts the 501 status + XML body shape.
  - signRequest helper in middleware_test.go reusable for future
    middleware-level SigV4 tests (sets X-Amz-Date, computes signature
    via existing computeSignature/canonicalRequest helpers).
- ROADMAP.md "Known latent bugs" entry updated to reflect that
  trailer-format requests are now explicitly rejected with 501 instead
  of falling into the chunk decoder.
- Files changed: internal/auth/middleware.go, internal/auth/sigv4.go,
  internal/auth/types.go, internal/auth/middleware_test.go (new),
  internal/s3api/errors.go, internal/s3api/errors_test.go (new),
  ROADMAP.md.
- **Learnings for future iterations:**
  - DenyHandler is the only response hook auth.Middleware exposes for
    short-circuit errors. Rendering a 501 (vs the deny handler's usual
    403) is done by extending the s3api WriteAuthDenied switch with an
    errors.Is(err, auth.ErrTrailerFormatUnsupported) branch. Don't add
    a separate "notImpl handler" hook to Middleware — one switch arm in
    the deny handler is the lighter shape.
  - Trailer detection fires AFTER signature validation. This keeps the
    feature surface invisible to unauthenticated probers and matches the
    code path's existing layering (auth → bodyHash decisions). If a
    later story needs a pre-auth feature gate, that's a new shape.
  - The test helper signRequest in internal/auth/middleware_test.go
    needs the host header to be set explicitly via req.Host (not
    req.Header.Set("Host", …)) — httptest.NewRequest pulls from URL but
    canonicalHeaders reads from req.Host. Future SigV4 middleware tests
    must follow this pattern.
  - Two streaming sentinels recognised: STREAMING-AWS4-HMAC-SHA256-PAYLOAD
    and STREAMING-UNSIGNED-PAYLOAD-TRAILER. The third
    STREAMING-AWS4-HMAC-SHA256-PAYLOAD-TRAILER (signed chunks + signed
    trailer) is intentionally NOT in isStreamingSentinel — PRD lists
    only two and current SDKs that emit it are uncommon. Add later if
    field traffic shows up.

## 2026-05-01 - US-004
- New file internal/s3api/sigv4_streaming_test.go: end-to-end test that
  exercises the full HTTP path through auth.Middleware{ModeRequired} so
  the chain validator built in US-002 runs against real network bytes.
- signedHarness helper wraps s3api.Server with auth.Middleware (test-
  local) and exposes signRequest / putBucketSigned / getObjectSigned /
  putStreamingObject. SigV4 signing helpers (deriveSigningKeyTest,
  sigHmacTest, sigSha256HexTest, chainSigTest) replicate the auth
  package's unexported constants — values are AWS-spec frozen so
  duplication is safe and keeps the auth surface unchanged.
- TestStreamingSigV4_PristineUpload_Succeeds (Variant A): PUT bucket →
  streaming PUT with valid chain (chunkA, chunkB, final-empty) → 200,
  then GET returns the concatenated body. Confirms the validator does
  not over-reject pristine uploads.
- TestStreamingSigV4_MutatedChunk_Rejected (Variant B): same path but
  one byte inside chunk-2's payload is XOR'd 0x01 mid-stream (chunk-
  signature header untouched). PUT returns 403 SignatureDoesNotMatch
  (XML body checked); subsequent GET returns 404 NoSuchKey (XML body
  checked) — proves buffer-then-validate dropped the mutated bytes
  before the meta store recorded the object.
- Files changed: internal/s3api/sigv4_streaming_test.go (new).
- **Learnings for future iterations:**
  - Existing testHarness in internal/s3api/testutil_test.go bypasses
    auth (it serves s3api.Server directly via httptest). To exercise
    middleware-driven code paths (chain validation, trailer rejection,
    presign auth) the test must build its own wrapper —
    httptest.NewServer(mw.Wrap(api, s3api.WriteAuthDenied)).
  - The auth package keeps its SigV4 helpers unexported. For tests in
    s3api_test that need to sign requests, replicate the algorithm
    locally (deriveSigningKey is just three nested HMAC-SHA256 calls
    seeded with "AWS4"+secret). Do NOT export auth internals just to
    satisfy a test; the AWS-spec constants are frozen.
  - canonicalQuery is empty when the request URL has no query string;
    keep the explicit "" + "\n" line in canonical-request construction
    so the four-line frame between method and signed-headers stays
    correct.
  - canonicalHeaders treats "host" specially — it reads from req.Host,
    not req.Header. When using httptest the URL host is auto-populated
    by http.NewRequest, so signing code must read req.Host (NOT
    req.Header.Get("Host")).
  - For streaming SigV4 the canonical-request hashedPayload is the
    sentinel STREAMING-AWS4-HMAC-SHA256-PAYLOAD literally, not a hash
    of the body. The chain signatures hash the actual chunk payloads.
  - mutateAt is the absolute byte offset in the assembled chunked body
    (NOT the offset inside the chunk payload). The pristine-build pass
    in TestStreamingSigV4_MutatedChunk_Rejected locates chunkB inside
    the framed body before flipping a byte; reusing this pattern lets
    future tests target any byte the SDK could plausibly mutate.
  - bytes.Index works on the framed body even though chain signatures
    differ run-to-run because chunkB's literal bytes appear once
    between two CRLF runs and the size header has fixed width.
