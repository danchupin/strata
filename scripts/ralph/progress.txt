**Learnings for future iterations:**
- `BucketSigningResolver.ClassifyUnwrap` keeps the `auth` package
  import-free of `kms` — kms-side sentinels translate to auth-side
  sentinels at wiring time. Future provider additions (e.g. GCP KMS)
  can return `kms.ErrKMSUnavailable` for transient and the existing
  serverapp classifier picks them up without code changes.
- `kms.WrapTransient` is the canonical "decide between 503 and 401"
  shim. New providers MUST call it at every external-call boundary so
  the auth path can fail-closed correctly. The string-marker list in
  `classify.go::transientMarkers` is intentionally short — extend with
  caution to avoid hiding permanent misconfigurations as 503.
- `adminapi.Server.SigningKey` is the shared admin-side surface for the
  3 rotate endpoints. Plumbing more rotation tooling (key-history,
  CMK migration) lands here without touching the gateway middleware.
- The DEK plaintext is returned ONCE in `POST /rotate` response —
  consumers (the operator console UI, scripted automation) must capture
  it client-side immediately. Subsequent `GET /status` does NOT expose
  the secret. Same one-shot pattern as the IAM access-key create flow.
- `meta.Bucket` schema is unchanged for US-002 — US-001 already shipped
  the trio (`signing_wrapped_dek`, `signing_key_id`,
  `signing_key_created_at`). When extending the surface (e.g. key
  history), prefer extending the existing trio over adding new columns;
  blob-style append works well for JSON history lists.
- `cmd/strata/workers` has a flaky unit test pre-existing; ignore on
  one-run failure and re-run.
