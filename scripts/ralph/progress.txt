## Codebase Patterns
- `data.Manifest` field rename + JSON migration: rename old Go field, drop the old JSON tag, and write a `func (m *Manifest) UnmarshalJSON(b []byte) error` shim that sniffs the legacy shape (`json.RawMessage` â†’ try new struct, fall back to old type). Proto wire compat is "free" if the field number is preserved â€” only the label changes.
- Per-backend manifest construction lives inside `Store.CompleteMultipartUpload` in each meta backend (memory/cassandra/tikv). Shared shape goes through an exported helper in `internal/meta/store.go` (e.g. `meta.BuildPartRange`) so all three backends stay lockstep.
- After editing `internal/data/manifest.proto`, regenerate via `protoc -I internal/data --go_out=. --go_opt=module=github.com/danchupin/strata internal/data/manifest.proto` (NOT `--go_opt=paths=source_relative` â€” that drops the file alongside the .proto, ignoring the `manifestpb` package directory).
- Streaming-checksum on copy/transform paths: single `hash.Hash` allocated by `newChecksumHasher(algo)`, body wrapped via `io.TeeReader(reader, hasher)`, finalized to base64 after the downstream consumer (PutChunks) returns. Compare against the client-supplied `x-amz-checksum-<algo>` header only when non-empty; otherwise the server-computed digest is authoritative. Persist into `MultipartPart.Checksums[algo]` so US-001 PartRange + composite Complete carry it forward.
- ListObjects delimiter+prefix folding lives at the s3api layer (`Server.listObjectsFolded`), NOT in the meta backends. The meta backends still implement the prefix-folding path for callers that pass Delimiter, but the gateway strips it for ListObjects to avoid (a) duplicate-prefix on continuation (no `pfx <= marker` filter), (b) wrong NextMarker shape (next-unadded vs last-emitted), and (c) over-truncation. The fan-out / heap-merge in `cassandra/store.go::ListObjects` is intentionally untouched (US-005 AC).
- Meta backends (`memory.ListObjects`, cassandra, tikv) return `NextMarker` in the **next-unadded** shape (the first key NOT included in the page). With `Marker` semantics being **exclusive** (`k <= opts.Marker` filter), passing that NextMarker back as Marker would SKIP that key. Fix lives at the s3api layer:
  - V1 (`listObjectsFolded`): set `res.NextMarker = lastEmitted` (last item actually returned).
  - V2 (`listObjects` no-delim path): override `nextMarker = res.Objects[len-1].Key` before encoding into the opaque continuation token. The delim path goes through `listObjectsFolded` and already has last-emitted shape.
- ListObjectsV2 continuation token is opaque base64-RawURL JSON (`listV2Token{Marker, Shard}`). Encoder/decoder live in `internal/s3api/listv2_token.go`. Decode falls back through (a) base64-Std for clients that re-encoded our raw form via permissive decoders, then (b) literal-marker for pre-US-006 in-flight tokens. Empty inner Marker is rejected so a hand-crafted `{"m":""}` does not silently advance to "". V1 `marker` parameter is untouched â€” opaque only on V2.
