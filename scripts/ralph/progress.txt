- **Learnings for future iterations:**
  - meta.Bucket extension auto-flows through TiKV: bucket rows are persisted via `json.Marshal(b)` in `internal/meta/tikv/codec.go::encodeBucket`, so adding a json-tagged field gives schema-additive TiKV persistence for free. Memory backend stores the in-process bucket struct, so it propagates as well — only cassandra needs an explicit SELECT column update.
  - Cassandra `GetBucket` AND `ListBuckets` both have their own column lists — when adding a buckets column, edit BOTH or ListBuckets returns zero-values for the new field. Look in `internal/meta/cassandra/store.go` for `SELECT ... FROM buckets`.
  - `alterStatements` runs at boot via `ensureTables` and is idempotent (`isColumnAlreadyExists` swallows the AlreadyExists error). Just append.
  - Pattern for "field absent vs zero-value" on PUT bodies: use a `*string` (or `*T`) on a separate internal request struct rather than a shared response/request struct. Lets the handler distinguish "operator did not include mode in body" from "operator explicitly set mode to empty string".
  - The bucket-row LWT lesson from `SetBucketVersioning` (CLAUDE.md cassandra gotchas) applies to every new buckets-column writer: `UPDATE ... IF EXISTS` keeps the write in the Paxos lineage that `CreateBucket` (`INSERT ... IF NOT EXISTS`) started, so LOCAL_QUORUM reads see the new value immediately. Skipping LWT would leak the read-after-write incoherence noted in the SetBucketVersioning comment.
  - The racetest oracle (`internal/racetest/TestRunCleanRunNoInconsistencies`) is timing-sensitive when the full test suite runs in parallel — failed once during `go test ./...`, passed on isolated re-run. Treat as a flake unless reproducible with `-count=N`.
