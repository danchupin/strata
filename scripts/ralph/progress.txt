## Codebase Patterns
- `data.Manifest` field rename + JSON migration: rename old Go field, drop the old JSON tag, and write a `func (m *Manifest) UnmarshalJSON(b []byte) error` shim that sniffs the legacy shape (`json.RawMessage` → try new struct, fall back to old type). Proto wire compat is "free" if the field number is preserved — only the label changes.
- Per-backend manifest construction lives inside `Store.CompleteMultipartUpload` in each meta backend (memory/cassandra/tikv). Shared shape goes through an exported helper in `internal/meta/store.go` (e.g. `meta.BuildPartRange`) so all three backends stay lockstep.
- After editing `internal/data/manifest.proto`, regenerate via `protoc -I internal/data --go_out=. --go_opt=module=github.com/danchupin/strata internal/data/manifest.proto` (NOT `--go_opt=paths=source_relative` — that drops the file alongside the .proto, ignoring the `manifestpb` package directory).

- **Learnings for future iterations:**
  - PRD-spec field name `PartChunks []PartRange` collides with the existing `PartChunks []int`. Solution: rename the old Go field (PartChunkCounts), add a custom UnmarshalJSON shim that sniffs the legacy JSON shape. Proto wire-compat preserved by keeping field number 6 (only the label changed).
  - `protoc --go_opt=paths=source_relative` drops generated code adjacent to the `.proto` file regardless of the `option go_package` directive. Use `--go_out=. --go_opt=module=<go-module>` instead so the generator respects the package path.
  - `MultipartPart.Checksums` may carry multiple algos (rare); `meta.BuildPartRange` consults a deterministic precedence (`CRC64NVME → SHA256 → SHA1 → CRC32C → CRC32`) before falling back to map iteration.
  - The s3api-side AC ("populate PartChunks in completeMultipartUpload before writing the object row") was satisfied inside the meta backend instead — that's where MultipartPart info is in scope and where the object row is actually written. Documented in PRD notes.
