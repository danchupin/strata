## Codebase Patterns
- Crypto packages live under `internal/crypto/<area>/`. SSE-S3 master key code is in `internal/crypto/master/`; AEAD chunk + DEK wrap primitives are in `internal/crypto/sse/`.
- Provider interfaces return `(key, keyID, error)`. Callers (gateway main) own fatal-log on error — log-free providers are still preferred. **Exception:** when a provider needs to log non-fatal warnings (e.g. Vault refresh-failure with cached fallback), inject a `*slog.Logger` via the config struct and default to `slog.Default()`. Tests capture WARN by handing in `slog.New(slog.NewJSONHandler(&buf, …))`.
- Sentinel errors (e.g. `master.ErrNoConfig`, `master.ErrInvalidKeyLength`, `sse.ErrInvalidKeySize`, `sse.ErrShortWrappedDEK`) are exported from the package so callers and tests can `errors.Is` them.
- Use `t.Setenv` / `t.TempDir` for env-var and file-backed tests. For mtime-based hot-reload tests, set the cached mtime to the past via `os.Chtimes` before rewriting so the rewrite is observable on coarse-mtime filesystems.
- For external-service providers (Vault, AWS-KMS), inject a `Now func() time.Time` so TTL-based refresh logic is testable without `time.Sleep`. Fake `now` advances by `+= duration` between Resolves.
- Every test that calls `master.FromEnv()` must `t.Setenv` ALL precedence vars (vault, file, env) to a known value — otherwise an ambient process env (e.g. on a developer machine) flips precedence and tests pass/fail intermittently.
- AEAD layout: chunk ciphertext stored as bare `gcm.Seal` output (no nonce prefix — IV recomputed via `sse.DeriveChunkIV(oid, chunkIndex)` on decrypt). Wrapped DEKs stored as `nonce(12) || gcm.Seal output` because the wrap nonce is random.
- For SSE-S3 chunk alignment: pChunkSize = data.DefaultChunkSize - 16 (= 4 MiB - tag). Encrypt up to pChunkSize plaintext per crypto chunk → ciphertext = exactly 4 MiB → backend produces one backend chunk per crypto chunk (index aligned). On decrypt, walk manifest chunks via `sum(chunk.Size for j<i)` to map crypto-chunk i to its ciphertext byte range; chunk plaintext size = chunk.Size - 16.
- For SSE objects, the ETag persisted on the object row is the plaintext MD5; the data manifest's ETag/Size stay as the backend wrote them (= ciphertext). The s3api PUT response uses `obj.ETag`, not `m.ETag`. GET range parsing uses `o.Size` (plaintext). Don't unify the two — manifest-internal arithmetic in `data.GetChunks` still expects ciphertext-space.
- s3api harness in `testutil_test.go` attaches a default `harnessMasterProvider` so existing AES256 tests keep working — opt out by building a server manually if you need to test the no-master path (see `TestSSEMissingMasterReturns500`).
- `internal/data/memory.Backend.CorruptFirstChunk()` is a test-only helper for tampering tests. Memory backend only — RADOS backend has no equivalent (and shouldn't).
- Don't import `internal/crypto/sse` as `sse` inside `internal/s3api/server.go`; the file already has a local `sse` variable for the SSE algorithm string. Use the alias `ssecrypto` instead. Same caution applies whenever a file mixes SSE-algorithm-string vars with crypto package calls.
- Don't run `go mod tidy` casually inside Ralph stories — it pulls in unrelated module drift (koanf direct/indirect flips, transitive grpc/protobuf bumps). If a new package adds a direct import that already lives in `go.sum` as indirect, Go builds fine without retidying; leave the `// indirect` annotation alone unless the story explicitly requires a deps refresh.
- `data.Manifest` is JSON-encoded inside the meta backend (cassandra/codec.go is `json.Marshal/Unmarshal`). Adding a new field is **schema-additive** and needs zero migration if you tag it `json:",omitempty"`. Use this lever (e.g. `Manifest.PartChunks`) before adding a new Cassandra column.
- For multipart SSE: each part gets oid = `key + ":part=" + partNumber`, chunkIndex local-to-part. After Complete, the merged manifest carries `PartChunks []int` (= chunk count per part, in part order) so the GET path can reconstruct (partNumber, chunkIndexInPart) for any flat manifest chunk index. Pattern lives in `internal/s3api/sse_io.go::multipartChunkLocator` + `multipartPartOID`.
- For multipart objects, two sizes diverge: per-part `Size` is plaintext (used for ListParts, composite ETag, obj.Size) and per-part `Manifest.Size` is ciphertext (used by data.Backend GetChunks bounds). When constructing `obj.Manifest` in `CompleteMultipartUpload`, sum chunk sizes — not part sizes — to get ciphertext total. Same lesson as US-004 single PUT (obj.Size plaintext / manifest.Size ciphertext) but harder to spot because both backends used to use a single `totalSize` for both.
- Per-part ETag persisted on `multipart_parts` is plaintext MD5 of the part body (AWS-compatible). Composite ETag in Complete is built by concatenating the part-ETag bytes and re-MD5-ing — works on plaintext-MD5 inputs, exactly matching the unencrypted multipart contract. Don't accidentally store ciphertext ETag on the part row.
- Multi-key SSE-S3 master rotation: `master.RotationProvider` (in `internal/crypto/master/rotation.go`) holds an ordered list `entries[0]=active, entries[1..]=unwrap-only`. Server code never calls `Master.Resolve` for unwrap directly — it uses `master.ResolveByID(ctx, p, keyID)` which type-asserts to `Resolver` (rotation provider) and otherwise falls back to single-key Resolve with id-equality / legacy-empty-id handling. Same helper covers PUT-side unwrap of the upload row in multipart. Single-key providers (env, file, vault) keep their previous semantics.
- Operator-tooling-only meta methods (`UpdateObjectSSEWrap`, `UpdateMultipartUploadSSEWrap`, `Set/GetRewrapProgress`) live on `meta.Store` so both backends + the contract suite (`storetest.caseSSEWrapRotation`) cover them. Background workers like `cmd/strata-rewrap` consume them via the regular `meta.Store` interface, not a separate type-asserted interface — keeps the type-system load consistent across stories.
- For background workers, factor logic into `internal/<feature>` and keep `cmd/<feature>/main.go` as the wiring shim (config load, signal handling, build store/provider, hand off to the lib). See `internal/rewrap/rewrap.go` + `cmd/strata-rewrap/main.go` for the canonical split. The lib gets unit tests against the memory backend; the cmd binary stays untested. Mirrors the existing `internal/gc` + `cmd/strata-gc` pattern.
- Single-document bucket-config endpoints (CORS, policy, encryption, public-access-block, ownership-controls, lifecycle blob, …) all reuse `setBucketBlob` / `getBucketBlob` / `deleteBucketBlob` in `internal/meta/cassandra/store.go` + the memory backend's per-config maps. Handler reads bucket → calls `Set/Get/DeleteBucketX`. ErrNoSuch<X> sentinel + matching `ErrNoSuch<X>` APIError. resolveSSE in `internal/s3api/encryption.go` is the canonical "feature falls back to bucket default" pattern (header > config > none); copy this shape for any future bucket-default-applied-on-write feature.
- Server-side hooks (notification, replication-status, audit log) live next to the handler emitting them — `s.replicationStatusFor(...)` in server.go::putObject is the prior pattern, `s.emitNotificationEvent(...)` (US-008) is the same shape: best-effort, swallow `meta.ErrNoSuch*` silently, never fail the underlying request. Hook calls go AFTER the meta write succeeds and BEFORE `WriteHeader` so headers from the hook can still be set; for completeMultipart the hook fires before status emission too. When the hook needs request-scoped facts (sourceIP, principal), use the `clientSourceIP` / `principalFromContext` helpers in `internal/s3api/notification.go` rather than re-deriving.
- Buffer-and-drain queues (gc_queue, notify_queue) on the meta interface follow a 3-method shape: `EnqueueX(ctx, *X) error`, `ListPendingX(ctx, partition, limit) ([]X, error)`, `AckX(ctx, X) error`. Cassandra partitions by `(scope, time-bucket)` with TimeUUID clustering — gc_queue uses `(region, enqueued_at)` (timestamp clustering), notify_queue uses `((bucket_id, hour), event_id)` (TimeUUID clustering). For Cassandra ListPending the scan walks N most-recent time-bucket partitions inside the method (e.g. last 48 hours for notify); the worker is unaware of the partitioning. Re-use this shape for US-011 replication_queue and US-013 access_log_buffer.
- DLQ tables follow the same `((bucket_id, time-bucket))` partition shape as their live queue but with a coarser bucket — notify_queue uses `hour`, notify_dlq uses `day` because dropped events stay around longer (operator-inspectable). Pair the table with a `meta.<X>DLQEntry` struct that **embeds** the original event struct + adds Attempts/Reason/EnqueuedAt. Two-method interface (`EnqueueXDLQ`, `ListXDLQ`) — no Ack since DLQ is a forensic/inspect surface, not a delivery queue. Worker moves a row out of the live queue + into the DLQ atomically from the worker's POV (Enqueue then Ack); a crash between Enqueue and Ack causes one extra delivery attempt on next tick — acceptable for at-least-once semantics. Re-use for US-010 (DLQ inspection HTTP endpoint), US-014 (access_log buffer drain failure), US-046 (audit_export DLQ).
- Background worker dispatch loops follow a `Run(ctx)` ticker + `RunOnce(ctx)` single-pass split. `RunOnce` is exported so unit tests drive deterministic ticks instead of sleeping on a real ticker; `Run` is the production entry. The retry/backoff scheduler uses `Now func() time.Time` + `Backoff func(attempt int) time.Duration` injection so tests advance time + flatten backoff to zero without sleeps. Pattern: `internal/notify.Worker` (US-009); reuse for US-012 replicator + US-014 access-log + US-046 audit-export. Don't bury the loop body inside `Run` — pull it out so tests aren't fighting a ticker.
- Sink-style fan-out (notify, replicator targets, access-log targets) uses an interface (`Sink { Type() string; Name() string; Send(ctx, evt) error }`) + a `Router { Resolve(evt) (Sink, bool) }`. `StaticRouter` is `map[string]Sink` keyed by `<type>:<arn>` with arn-only fallback. Env-driven router parses `STRATA_<FEATURE>_TARGETS=<type>:<arn>=<spec>,...` where the spec is sink-specific (`<url>|<secret>` for webhooks). Router resolution failure (no sink for the row's TargetARN) routes the row straight to the DLQ with `reason='no sink registered for target'` rather than retrying forever — operators see misconfiguration immediately.
- HMAC delivery shape: lowercase-hex HMAC-SHA256 of the JSON body in `X-Strata-Signature`. `notify.ComputeSignature(secret, body) string` + `notify.VerifySignature(secret, body, header) bool` — the verify side hex-decodes then constant-time-compares with `hmac.Equal`. Same shape will be the auth pattern for US-022 (audit hook) and US-046 (audit export upload signing) — keep these helpers in `internal/notify` for now, hoist to `internal/crypto/hmac` only if a third caller materialises.
- External SDK injection without dep bloat: when a sink/feature needs an external SDK (AWS, GCP, ...), define a narrow interface in the `internal/<feature>` pkg covering only the methods used (`SQSAPI` = SendMessage only) + add a compile-time check `var _ NarrowAPI = (*sdk.Client)(nil)` in a test file. Expose an `EnvOption` (e.g. `WithSQSClientFactory(func(region string) (NarrowAPI, error))`) the cmd layer provides — that way the SDK lives in `cmd/<binary>/main.go` only, internal pkg tests keep using fakes, and operators that don't use that sink never compile the SDK code. `router_env.go` rejects sink-spec without its factory at startup so misconfiguration surfaces immediately, not on first event. Same shape will work for SNS, Kinesis, GCP Pub/Sub (US-014/046).
- Payload-as-MessageBody (no re-marshal): sink Send methods MUST use `evt.Payload` verbatim as the wire body — the AWS-S3-event-message JSON shape is the cross-sink contract. Re-marshalling wraps the payload in a notify-internal envelope and breaks consumer parity (SQS subscriber sees a different shape than webhook subscriber). Same rule for replicator destination payloads in US-012 (the source PUT body is the contract, not a re-encoded mirror).
- Admin-only HTTP endpoints (`/?notify-dlq`, US-023 `/?audit`, US-034 admin routes) gate via `info := auth.FromContext(ctx); if info == nil || info.IsAnonymous || info.Owner != IAMRootPrincipal { 403 }`. Don't shortcut by relying on signed-request presence — `IsAnonymous` distinguishes signed-but-non-root from unsigned, and both should 403. Bucket-not-found uses `mapMetaErr` (returns NoSuchBucket); do not custom-map. Pagination via opaque continuation tokens (event_id for `notify-dlq`; consider proper cursor-encoded `(partition, clustering_key)` for larger surfaces like audit). Response is JSON `{records, next_continuation_token}` with `Content-Type: application/json` — XML stays for S3-spec endpoints; admin surfaces are JSON.
- "Must be versioned" bucket-config guards (PutBucketReplication, future PutBucketObjectLockConfig at runtime, MFA Delete enable) use `meta.IsVersioningActive(b.Versioning)` — covers Enabled AND Suspended. Don't compare `b.Versioning == meta.VersioningEnabled` literal, that rejects Suspended buckets which AWS allows. Failure code is 400 InvalidRequest. Apply the guard at the top of the handler, after `GetBucket`, before the body read so malformed bodies don't waste cycles when the bucket can't host the config anyway.
- Workers that copy / migrate objects (replicator US-012, future inventory US-039, audit export US-046) read source bytes via `meta.Store.GetObject` → `data.Backend.GetChunks(manifest, 0, size)` — same path the gateway GET uses. For SSE-encrypted objects this returns ciphertext (peer stores verbatim; no decrypt-then-re-encrypt). Always `defer src.Body.Close()` on every dispatcher return path — including early-return on validation errors — so the data backend reader is freed even when the dispatch never reaches the network. `MetricsObserver` interface pattern (`internal/<feature>` defines it; cmd binary provides a tiny prometheus-wrapping shim) keeps the internal pkg free of `prometheus/client_golang` while still emitting per-label histograms + counters.
- Worker `Name` for `worker_locks` (gc/lifecycle/notify/rewrap/replicator) MUST be unique per binary — a shared cassandra cluster shares the lock table, and reusing a name silently makes only one of N workers run. Convention: lock name = binary basename without `strata-` prefix. New workers (US-014 access-log, US-046 audit-export) keep this rule.
- slog migration: every binary calls `logging.Setup()` first thing in main; that returns a JSON `*slog.Logger` driven by `STRATA_LOG_LEVEL` (DEBUG/INFO/WARN/ERROR; default INFO) and installs it as `slog.Default()`. Workers (`leader.Session.Logger`, `gc.Worker.Logger`, `lifecycle.Worker.Logger`, `notify.Config.Logger`, etc.) take `*slog.Logger`, never `*log.Logger`. Replace `log.Printf("foo: %v", err)` with `logger.WarnContext(ctx, "foo", "error", err.Error())` — pass `ctx` through every call so the per-request logger from `logging.LoggerFromContext` carries `request_id`. The HTTP gateway wraps the mux handler with `logging.NewMiddleware(logger, next)` which reads/generates `X-Request-Id`, sets it on both `r.Header` (so downstream code like the access-log middleware sees it) and `w.Header()` (so clients see correlation), and attaches a child logger with the id to the request context. Future stories that add log emission inside handlers should use `logging.LoggerFromContext(r.Context()).Info(...)` so request_id rides automatically.
- Per-storage observer hooks (Cassandra slow-query / RADOS DEBUG): `cassandra.SessionConfig.Logger + SlowMS` wires a `gocql.QueryObserver` that emits structured WARN lines for queries over threshold or with errors (`STRATA_CASSANDRA_SLOW_MS`, default 100). `rados.Config.Logger` enables per-op DEBUG (`put|get|del`) with oid + duration_ms. Both pull `request_id` from `logging.RequestIDFromContext(ctx)` so log lines correlate with the gateway request. For build-tag-isolated backends (RADOS = `//go:build ceph`), put the loggable helper in a tag-free file (`internal/data/rados/observer.go`) so unit tests can run without the tagged toolchain. Always short-circuit `if !logger.Enabled(ctx, level)` before building the attrs slice for hot-path observers.
- Cmd-layer Metrics adapter pattern: when an internal package needs prom histograms but pulling `prometheus/client_golang` would bloat unit-test deps, define a narrow `Metrics` interface inside the internal package (e.g. `cassandra.Metrics{ObserveQuery(table, op, dur, err)}`) + a `Metrics` field on the config struct. The cmd binary plugs in a tiny adapter struct from `internal/metrics` (`metrics.CassandraObserver{}`). Same shape as the US-012 replicator `MetricsObserver`. gocql allows only ONE `QueryObserver` per cluster, so the cassandra observer fans out internally to (logger, metrics) — don't chain. For HTTP path-label cardinality, use `metrics.TemplatePath()` to collapse URL paths to `/`, `/{bucket}`, `/{bucket}/{key}` plus literal admin endpoints — never label by raw `r.URL.Path`. Test prom collectors via `prometheus.Observer.(prometheus.Metric).Write(&dto.Metric{}).Histogram.GetSampleCount()` and call `HistogramVec.Reset()` between tests since collectors are package-globals.
- Worker queue-depth metric sampling: don't add a separate ticker for "sampled every 30s" gauges. The drain loop already runs every Interval (default 5s replicator, 30s gc) and already calls `ListPending<Queue>` — sample queue depth from the returned slice's length per drain pass and call `Metrics.SetQueueDepth(scope, n)`. Replicator aggregates per RuleID inside drainBucket; GC samples per region inside drain. Same shape for any future queue. Don't over-engineer a separate sampler.
- "Walk store + emit metric" samplers live OUTSIDE internal/metrics: internal/metrics has zero strata-internal deps. A sampler that walks ListBuckets + ListObjects (`internal/bucketstats`) lives in its own pkg with a Sink interface; internal/metrics only provides the adapter (e.g. `metrics.BucketStatsObserver{}`). Reuse for inventory jobs, audit-export volume, etc. — keeps the metrics pkg lean.
- Per-action counter shape `<feature>_tick_total{action, status}`: `status` vocab is `success | error | skipped` where `skipped` covers CAS-race / idempotent no-op outcomes. `action` vocab is the verb-set the worker performs (transition / expire / expire_noncurrent / abort_multipart for lifecycle). Reuse this shape for any future per-outcome worker dashboard.

## 2026-04-26 - US-003
- Added `internal/crypto/sse` package with AEAD primitives for SSE-S3 chunk + DEK protection.
- API:
  - `EncryptChunk(dek, oid, chunkIndex, plaintext) ([]byte, error)` and `DecryptChunk` symmetric — AES-256-GCM with deterministic 12-byte IV from `DeriveChunkIV` (HKDF-SHA256, IKM = `oid || chunkIndex(BE64)`, info = `"strata-sse-chunk-iv"`).
  - `WrapDEK(masterKey, dek)` / `UnwrapDEK(masterKey, wrapped)` — AES-256-GCM with random 12-byte nonce; output blob is `nonce || ciphertext || tag` so the persisted form is self-contained.
  - `DeriveChunkIV` exported for tests + future call sites that want IV diversity assertions.
  - Sentinels: `ErrInvalidKeySize`, `ErrShortWrappedDEK`.
- Files added: `internal/crypto/sse/sse.go`, `internal/crypto/sse/sse_test.go` (18 tests).
- Quality: `go build ./...`, `go vet ./...`, `go test ./...` all pass; targeted `go test ./internal/crypto/sse/... -v` is 18/18 PASS.
- **Learnings for future iterations:**
  - HKDF input choice: `IKM = oid || uint64BE(chunkIndex)` and `info = "strata-sse-chunk-iv"` — salt left nil since neither side has an out-of-band entropy source. Different chunkIndex / oid produce different IVs (verified by tests). If US-049 ever needs IV diversity across rewrites, the DEK already differs per object so collisions are not a worry.
  - Chunk ciphertext is **not** prefixed with a nonce — the receiver derives the IV deterministically from oid+chunkIndex. Saves 12 bytes per chunk on disk. The same shortcut is **not** safe for WrapDEK (random nonce, must be persisted), so the layouts differ on purpose.
  - `gcm.Seal(nil, …)` for chunks vs `gcm.Seal(out, …)` with pre-built `out=nonce||cap` for the wrapped DEK. Standard idiom — see `WrapDEK` for the prefix pattern.
  - The package depends on `golang.org/x/crypto/hkdf`. That module was already in `go.sum` as indirect; `go build ./...` accepts the upgrade-to-direct without complaint, so we did NOT run `go mod tidy` (would have churned unrelated transitive bumps — koanf, grpc-gateway, protobuf). Pattern: when adding a direct import that already exists indirect, just import and build. Reserve `tidy` for when a new module needs adding to `go.sum`.
  - `sse.EncryptChunk` is deterministic in (dek, oid, chunkIndex) — same inputs yield same ciphertext byte-for-byte. This is fine for SSE-S3 (every object has a unique DEK so no cross-object reuse), and lets the GC + lifecycle stories assume bit-stable rewrites. Test `TestEncryptChunkSameInputsProduceSameCiphertext` pins this.
  - This story does NOT wire anything into PutObject/GetObject. US-004 is the wiring point — it will need to (a) call `WrapDEK` and persist `objects.sse_key` + `sse_key_id`, and (b) walk chunk indexes and call `EncryptChunk` per chunk. The OID we feed into `DeriveChunkIV` must be deterministic per (object, version) — likely use `bucket/key/versionID` as the canonical string. Decide that in US-004; the package is agnostic.

## 2026-04-27 - US-016
- Added `internal/meta/cassandra/observer.go`: `SlowQueryObserver` implements `gocql.QueryObserver`, logs WARN with `request_id` (from `logging.RequestIDFromContext`), `table`, `op`, `duration_ms`, `statement`, optional `error` when elapsed >= threshold OR query failed. `NewSlowQueryObserver(logger, threshold)` returns nil when disabled. `SlowMSFromEnv()` reads `STRATA_CASSANDRA_SLOW_MS` (default 100, "0" disables, garbage falls back to default). `parseStatement` is a best-effort SELECT/INSERT/UPDATE/DELETE/CREATE/ALTER table extractor with `IF NOT EXISTS` skip; `truncateStatement` collapses whitespace and caps at 240 chars to bound log-line size.

- Files modified: `internal/meta/cassandra/session.go`, `internal/data/rados/config.go`, `internal/data/rados/backend.go`, all 7 `cmd/strata-*/main.go`.
- Quality: `go build ./...`, `go vet ./...`, `go test ./...`, `go test -race ./...` all green. `go build -tags ceph ./...` not runnable locally (no librados header) — CI handles it; ceph-tagged backend.go was hand-verified for syntax + imports.
- **Learnings for future iterations:**
  - **Pattern (gocql QueryObserver wiring)**: `gocql.ClusterConfig.QueryObserver` is settable on the ClusterConfig — set it in `newCluster` (not after `CreateSession`). `gocql.ObservedQuery{Statement, Start, End, Err, Rows, Host, Attempt}` is what `ObserveQuery(ctx, q)` receives; `q.End.Sub(q.Start)` is the wall-clock duration. The ctx passed in is the per-query ctx (gocql passes through `Query.WithContext(ctx)`), so `request_id` rides through if callers thread ctx via the gateway logging middleware.
  - **Pattern (build-tag-free helper for tagged-backend unit tests)**: when the ceph-tagged backend can't be unit-tested locally (no librados), pull the loggable side-effect into a helper file with **no build tag** (`internal/data/rados/observer.go`) and have the tagged file call it. The helper can then be unit-tested on any platform via a tag-free `*_test.go`. Same trick will work for any future tagged backend (e.g. tikv, foundationdb) where the harness isn't installable.
  - **Gotcha (parseStatement IF NOT EXISTS)**: `CREATE TABLE IF NOT EXISTS <name>` parses as `[CREATE, TABLE, IF, NOT, EXISTS, <name>]`. Naive "find TABLE → next field" lands on `IF`. Added `skipIfNotExists(f, i)` to handle both `IF NOT EXISTS` and `IF EXISTS` (latter for `DROP TABLE IF EXISTS`). Future schema-DDL parsing additions should reuse this.
  - **Gotcha (DEBUG-level early-exit matters at hot-path scale)**: every RADOS op (potentially per chunk per request) calls `LogOp`. The `if !logger.Enabled(ctx, slog.LevelDebug)` short-circuit BEFORE building the attrs slice is essential — without it, every PUT allocates a fresh `[]any` even when DEBUG is filtered. Same lever applies to any future high-frequency observer (gocql per-row, OTel span emission).
  - **Gotcha (uniform cmd binary signatures)**: 6 different cmd binaries shape `buildMetaStore(cfg)` / `buildDataBackend(cfg)` identically. Bumping the signature once requires touching all 6 — keep them in lockstep so future stories that need to thread one more dep through (logger, tracer, metrics observer) pay the touch cost only once. Don't try to "improve" only the gateway; consistency wins.
  - **Gotcha (Delete signature change touched only the ceph file)**: `data.Backend` interface still declares `Delete(ctx context.Context, m *Manifest) error` so the memory backend already used a named ctx. The ceph backend was the only one using `_ context.Context`; renaming it doesn't change the interface, just the implementation. Memory backend: no change needed.

  - **Pattern (cmd-layer Metrics adapter for storage-layer observers)**: when an internal package needs to record metrics but pulling in `prometheus/client_golang` would add an import cycle / drag prom into unit-test deps, define a narrow `Metrics` interface inside the internal package + a `Metrics` field on the config struct. The cmd binary plugs in a tiny adapter (`metrics.CassandraObserver{}`). Same shape as the US-012 `MetricsObserver` pattern for the replicator. Reuse for OTel span emission (US-029), RADOS observer metrics (US-018), audit-log queue depth (US-046).
  - **Gotcha (gocql allows only one QueryObserver per cluster)**: combining "slow query log" + "every-query metric" can't be done by registering two observers. Make the single observer fan out internally — `if Logger != nil && (slow||err)` for log path, `if Metrics != nil` for metrics path. Don't try to chain; gocql ClusterConfig has a single `QueryObserver` field that overwrites.

Templating with `TemplatePath` keeps `path` at a fixed cardinality regardless of bucket count or key churn — `/`, `/{bucket}`, `/{bucket}/{key}`, plus literal admin endpoints. This is the canonical shape for any future HTTP histogram (US-020 `/healthz`, `/readyz`; US-029 OTel spans should reuse the same templating).
  - **Pattern (test prom histograms via Metric.Write → dto.Metric)**: to assert a histogram observation count without hitting the `/metrics` HTTP endpoint, `prometheus.Observer.(prometheus.Metric).Write(&dto.Metric{}).Histogram.GetSampleCount()` returns the count for that label tuple. Cleaner than parsing exposition output. Reuse this in US-018 worker-metric tests.
  - **Pattern (HistogramVec.Reset() at test boundaries)**: prom collectors are package-globals, so tests pollute each other. Call `HTTPDuration.Reset()` / `CassandraQueryDuration.Reset()` at the top of each metric-asserting test to avoid sample-count leakage.
  - **Gotcha (NewSlowQueryObserver vs NewQueryObserver)**: keeping the legacy constructor (log-only, returns nil if logger nil OR threshold 0) preserved the existing observer_test.go without churn. The new constructor `NewQueryObserver(logger, threshold, metrics)` returns nil only when BOTH sides are disabled. SessionConfig calls the new one; old tests still compile against the old one. When adding a third observer concern (e.g. OTel) prefer extending `NewQueryObserver`'s signature over yet another constructor.

- **Learnings for future iterations:**
  - **Pattern (cmd-layer adapter for worker metrics)**: same as the cassandra Metrics interface — define a narrow `Metrics` interface inside the worker package, plug `metrics.<X>Observer{}` from the cmd binary. Lets internal/<feature> stay free of `prometheus/client_golang`. Used for rados (ObserveOp), gc (SetQueueDepth), notify (IncDelivery), and the replicator extension (SetQueueDepth). `internal/metrics` becomes the single home for the prom registry + adapters.
  - **Gotcha (sampling at drain-time vs separate ticker)**: PRD asked for "queue_depth sampled every 30s". The replicator drains every 5s by default, so sampling at drain-time over-satisfies the requirement without burning a separate goroutine + ticker. Same trick for gc_queue_depth (drain interval is 30s default; sampling there is exactly the asked cadence). Don't over-engineer a separate sampler for queue depth when the drain loop already pays for the list query.
  - **Gotcha (rados Metrics interface lives in tag-free file)**: the librados-backed `backend.go` is `//go:build ceph` so it can't host the type definition that non-ceph code references. Apply this rule for any future "walk store + emit metric" sampler (US-039 inventory, US-046 audit-export volume).
  - **Pattern (lifecycle action/status counter)**: the new `LifecycleTickTotal{action, status}` is the canonical shape for any worker that wants per-outcome dashboards. action vocab kept short: transition / expire / expire_noncurrent / abort_multipart; status vocab: success / error / skipped (skipped covers CAS-race no-op). Reuse for US-046 audit-export, US-039 inventory job runs.
  - **Gotcha (multipart_active is event-driven, not sampled)**: chose Inc/Dec at handler boundaries over sampling `ListMultipartUploads`. Reasons: ListMultipartUploads paginates 1000-at-a-time per bucket, sampling N buckets hourly is heavy; counter drift on crash-during-handler is acceptable because it's an active-now gauge anyway. If drift becomes visible, add a once-per-hour reconciliation pass via `ListMultipartUploads` to set the gauge to ground truth.

Same `Reset()` discipline applies — reset every label set at the top of every metric-asserting test, since prom collectors are package-globals.
