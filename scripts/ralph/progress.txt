- `initiateMultipartUpload` (`multipart.go:30..125`) builds `mu.ChecksumAlgorithm` from `x-amz-checksum-algorithm` request header (uppercased) and `mu.ChecksumType` via `normalizeChecksumType(r.Header.Get(\"x-amz-checksum-type\"))`. `normalizeChecksumType` returns `\"FULL_OBJECT\"` / `\"COMPOSITE\"` for valid input, EMPTY otherwise. Empty type passes through to Complete.
- `composeMultipartChecksums` (`checksum.go:201`) — `fullObject := strings.EqualFold(checksumType, \"FULL_OBJECT\")`. When type is empty, fullObject = false → COMPOSITE shape (`BASE64(HASH(concat(raws)))-N`) is applied to ALL algos including CRC family. This is the bug for CRC64NVME — AWS spec says CRC64NVME only supports FULL_OBJECT (no COMPOSITE+CRC64NVME on AWS). boto3 1.36+ default-on FlexibleChecksum sends `ChecksumAlgorithm=CRC64NVME` with FULL_OBJECT validation expectation; on GET it streams body + recomputes raw CRC, divergent from server's COMPOSITE-with-`-N` shape → `FlexibleChecksumError`.
- US-001 fix: in `initiateMultipartUpload`, after `mu` is built, if `mu.ChecksumAlgorithm == \"CRC64NVME\"` and `mu.ChecksumType == \"\"` → force `mu.ChecksumType = \"FULL_OBJECT\"`. Reject `CRC64NVME + COMPOSITE` explicitly as 400 InvalidArgument. For CRC32 / CRC32C with empty type → default FULL_OBJECT (modern boto3 default). SHA1 / SHA256 stay empty / COMPOSITE.
- The bug is NOT specific to UploadPartCopy. boto3's high-level transfer manager triggers auto-checksum default-on, while native multipart tests in US-003 of prior cycle set algorithm/type explicitly and avoided the gap. Fix at Initiate level applies to both copy and non-copy paths.
- TestUploadPartCopyChecksumPersistsToCompositeOnComplete (`multipart_copy_test.go:143`) uses SHA256 explicitly — passes today because SHA256 + COMPOSITE is the supported shape. Doesn't catch CRC64NVME bug.
- `completeMultipartUpload` strict-ascending check at `internal/s3api/multipart.go:430` is `if p.PartNumber <= prev { writeError ErrInvalidPartOrder }`. Confirmed by code read. US-002 fix: `< prev` + dedupe-to-LAST per PartNumber matches AWS / RGW take-latest semantics.

## 2026-05-10 — US-001 ChecksumType defaults for CRC family on multipart Initiate
- `internal/s3api/multipart.go::initiateMultipart` now folds Initiate-time
  ChecksumType defaulting through a new `applyChecksumTypeDefault(mu, w, r)`
  helper that runs immediately after `mu` is built. The helper is the single
  source of truth for the AWS-parity rules the boto3 1.36+ FlexibleChecksum
  default-on hot path leans on:
    - `CRC64NVME` with empty type → `FULL_OBJECT` (CRC64NVME has no COMPOSITE
      shape on AWS; GET-side body recompute would otherwise diverge from a
      `BASE64(HASH(concat(raws)))-N` server echo and fire `FlexibleChecksumError`).
    - `CRC64NVME` with explicit `COMPOSITE` → 400 `InvalidArgument` with the
      AWS-parity message "The CRC64NVME checksum algorithm only supports the
      FULL_OBJECT checksum type."
    - `CRC32` / `CRC32C` with empty type → `FULL_OBJECT` (modern boto3 default;
      AWS accepts both shapes, FULL_OBJECT lets the body-stream-recompute path
      succeed transparently).
    - `SHA1` / `SHA256` with empty type → unchanged (Complete-time fallback to
      COMPOSITE; AWS rejects SHA + FULL_OBJECT explicitly so we keep it
      unreachable here).
