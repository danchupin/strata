  - Cassandra LWT batches across partitions are rejected — "batch a user_stats bump in the same

    bookkeeping there is free.
  - The PRD's "userBucketCount via ListBuckets" trap: even small test harnesses pay an O(N) scan;

## 2026-05-19 - US-002 - byte_seconds trapezoid integration
- New env knob `STRATA_USAGE_ROLLUP_SAMPLES_PER_DAY` (default 24 = hourly, clamp
  [1, 1440] with WARN log when out-of-range; treats 0 as "use default" so
  callers passing the zero-value Config still get 24-sample shape).
- New file internal/usagerollup/trapezoid.go: pure `Trapezoid(samples, totalSeconds)`
  + `AverageObjects` / `MaxObjects` helpers + `sampleRing` (sync.Mutex-guarded
  map keyed on `RingKey{BucketID, Class}`, cap-bounded at SamplesPerDay so a
  long-running ring window does not leak). Trapezoid math:
  `Σ_{i=0..N-2} (s[i]+s[i+1])/2 × Δt + s[N-1] × Δt` where `Δt = totalSeconds / N`
  — last segment carries last sample forward as a rectangle so a flat day
  integrates exactly back to `used_bytes × 86400`. `N=1` short-circuits to
  `samples[0] × totalSeconds` (= legacy v0 single-sample math, intentional
  fallback).
- internal/usagerollup/worker.go: gained `SamplesPerDay` Config field, the
  `ring *sampleRing` Worker field, `SampleOnce(ctx, now)` method (snapshot every
  bucket's `GetBucketStats` into the ring), and the daily roll-up tick now
  drains the per-(bucket, class) ring + computes `ByteSeconds = Trapezoid(samples, 86400)`,
  `ObjectCountAvg = mean(samples)`, `ObjectCountMax = max(samples)`.
  Empty-ring path falls back to a fresh `GetBucketStats` + single-sample
  math so a bucket created mid-day still gets a roll-up row.
- Run loop rewritten as single-select: ctx.Done / sample-ticker (fires every
  Interval/SamplesPerDay when SamplesPerDay>1) / daily-timer (fires at the
  configured At time, then resets to next-day). Replaces the previous
  one-timer + sleep-Interval shape so sample ticks and the daily flush can
  share the same goroutine.
- cmd/strata/workers/usage_rollup.go: reads `STRATA_USAGE_ROLLUP_SAMPLES_PER_DAY`
  via the existing `intFromEnv` helper (same shape as STRATA_GC_*); env
  unset → DefaultSamplesPerDay (24).
- Hard cutover per PRD: no `method` column on usage_aggregates; old `used × 86400`
  math is the N=1 fallback path, not a parallel branch. Schemas unchanged
  (Cassandra + TiKV + memory).
- Tests:
  - trapezoid_test.go: table-driven (flat, step-up at noon, step-down at noon,
    sawtooth, single-sample, empty, zero-duration, avg/max helpers, ring drain
    + cap-to-window). Step-up hand math: 11×(100·3600) + (100+200)/2·3600 +
    11×(200·3600) + 200·3600 = 13_140_000 — within 1.4 % of the
    perfect 12_960_000.
  - worker_test.go: existing tests continue passing (single-sample fallback
    preserves `used × 86400` math). New cases:
    - TestRunOnceTrapezoidsFullVirtualDay: drives 24 manual SampleOnce calls
      across hours 0..23 (step-up at noon, 100→200 bytes, 1→4 objects),
      asserts ByteSeconds == 13_140_000, ObjectCountAvg == 2,
      ObjectCountMax == 4.
    - TestRunOnceFallsBackToSingleSampleWhenRingEmpty: bucket created
      mid-day with no sample ticks → fallback to `used × 86400`.
    - TestNewClampsSamplesPerDayOutOfRange: 5000 clamps to 1440; -3 clamps
      to 1.
    - TestNewDefaultsSamplesPerDayWhenZero: zero-value Config gets 24.
- docs/site/content/best-practices/quotas-billing.md: replaced the v1 caveat
  block with a "byte-seconds trapezoid integration" section that explains
  the formula + N=1 fallback + worked example (100→200 step-up bills
  13_140_000 B·s vs the v0 over-count of 17_280_000). Added the new env
  knob to the env table.
- Files changed: internal/usagerollup/trapezoid.go (new),
  internal/usagerollup/trapezoid_test.go (new), internal/usagerollup/worker.go,
  internal/usagerollup/worker_test.go,
  cmd/strata/workers/usage_rollup.go,
  docs/site/content/best-practices/quotas-billing.md.
- Quality: `go vet ./...` green, `go build ./...` green, `make docs-build`
  green, `go test -race ./internal/usagerollup/...` green, full
  `go test -race ./...` green.
- **Learnings for future iterations:**
  - The ring is per-process; a leader flip mid-day starts a fresh ring on
    the new leader. First post-flip day will have fewer samples and
    degrade gracefully (worst case 1 sample → v0 math). No persisted state
    needed — the daily roll-up reads bucket_stats at the boundary anyway.
  - Run-loop refactor pattern: when you need N intermediate ticks PLUS a
    daily-clock fire, use a single select with three cases (ctx.Done +
    ticker + timer). Reset the daily timer on each daily fire to track
    the new nextFire; do NOT use a ticker for the daily fire because the
    "at HH:MM UTC" semantic is a wall-clock landmark, not a fixed cadence.
  - `SamplesPerDay` Config field uses 0 as a sentinel for "use default"
    so callers passing `Config{Meta: store}` (the worker_test shape) still
    get the production 24-sample behavior. Negative values clamp to 1
    (the trapezoid fallback path) with a WARN log; values above
    MaxSamplesPerDay=1440 clamp down with a WARN.
  - Memory backend has no `SetBucketStatsForTest` shortcut — tests must
    drive `BumpBucketStats` with deltas. Easy to forget when porting
    test fixtures from a Cassandra integration shape.
  - Worker tests that exercise the daily-flush path do NOT need to enter
    `Run()` — call `SampleOnce` + `RunOnce` directly. The full `Run` loop
    is exercised by `TestRunCtxCancelExits` already.

## 2026-05-20 - US-008 - Range zero-length edge tests
- Single behavior change in internal/s3api/server.go::parseRange: suffix
  range `Range: bytes=-N` against a zero-length object now returns
  (offset=0, length=0, status=200, ok=true) instead of (..., status=206,
  ok=true). The call site's `if status == http.StatusPartialContent &&
  partNumber == 0` guard suppresses Content-Range when status is 200, so
  the response is bare 200 + Content-Length: 0 + empty body — AWS parity
  for empty objects. Open-ended `bytes=0-` and explicit `bytes=0-9` cases
  on zero-length already returned 416 InvalidRange via the existing
  `start >= size` guard; verified with a small standalone repro and the
  new test cases.
- New file internal/s3api/range_test.go consolidates the four zero-length
  Range cases per PRD acceptance criteria (no Range / open-ended /
  suffix / explicit). Nonzero-size happy path remains in
  TestObjectRangeGet in object_test.go — both files now cover the Range
  surface with no overlap.
- Files changed: internal/s3api/server.go, internal/s3api/range_test.go
  (new), scripts/ralph/prd.json (US-008 passes:true).
- Quality: `go vet ./...` green; `go test -race ./internal/s3api/...`
  green (13.7s wall-clock including the four new sub-tests). No s3-tests
  run on this iteration — the fix only flips a single edge case (suffix
  range on zero-length) that the suite is unlikely to exercise; US-010
  will run the full suite + record pass-rate delta vs main baseline.
- **Learnings for future iterations:**
  - parseRange returns (offset, length, status, ok) — the call site uses
    `ok=false` as the 416 InvalidRange signal AND status to drive the
    response code on success. Returning status=200 for a "valid but
    empty" suffix range is the AWS-parity escape hatch: ok=true keeps
    the success path, status=200 keeps the response code, and the
    `status == StatusPartialContent` guard naturally elides
    Content-Range. Both fields are independent — don't conflate them.
  - For zero-length objects the GetChunks path through memory.Backend
    handles (offset=0, length=0) cleanly (length<=0 → length=Size-offset
    = 0 → empty memReader). No need to short-circuit body emission at
    the s3api layer.
  - Range test fixtures should always assert both Content-Range header
    (presence vs absence) AND status code — the two together pin the
    AWS-wire response. A status-only check would have let the previous
    206-with-empty-Content-Range shape slip through.

## 2026-05-20 - US-009 - Latent fix: aws-chunked-trailer streaming decoder (sha256 only)
- Recognised the `STREAMING-AWS4-HMAC-SHA256-PAYLOAD-TRAILER` content-sha256
  sentinel in `internal/auth/middleware.go` alongside the existing
  `STREAMING-AWS4-HMAC-SHA256-PAYLOAD` branch.
- `X-Amz-Trailer: <algo>` request header parsed at middleware time. Only
  `x-amz-checksum-sha256` (case-insensitive) accepted — `crc32` / `crc32c` /
  `sha1` reject with new `auth.ErrUnsupportedChecksumAlgorithm` sentinel,
  surfaced by the s3api WriteAuthDenied switch as HTTP 400
  `InvalidRequest` per PRD scope.
- `newStreamingTrailerReader(...)` shares the existing aws-chunked decoder
  state machine and just toggles trailerMode + a `sha256.New()`
  bodyHash accumulator. The final 0-chunk consumes the trailer header
  block, lex-sorts the canonical lines, recomputes the AWS4-HMAC-SHA256-
  TRAILER stringToSign (`chainHMAC(prevSig=finalChunkSig, hex(SHA256(canonical-trailers)))`),
  and verifies the trailer-signature + base64(SHA256(body)) match —
  mismatch surfaces as `ErrSignatureInvalid` per AWS parity (not a
  ChecksumMismatch code).

  non-sha256 trailer algos are batched in US-010 per cycle convention.
- **Learnings for future iterations:**
  - The trailer variant reuses the existing `streamingReader` struct
    rather than forking a parallel reader, because the chunk-signature
    chain is identical and only the EOF/trailer block differs. Setting
    `trailerMode=true` + a `hash.Hash` accumulator on the same shape
    means future trailer-checksum algos (crc32, crc32c, sha1) just
    swap the accumulator and add a new validate-helper — no extra
    decoder. The unsupported-algo path stays the only middleware-level
    branch.
  - AWS's trailer-signature stringToSign concatenates the
    `AWS4-HMAC-SHA256-TRAILER` algo line, the request timestamp (NOT
    the credential-scope date), the credential scope, the previous
    (= final-chunk) signature, and the hex(SHA256(canonical-trailer-
    headers)). Canonical trailer headers are `<lower-name>:<trim-value>\n`
    sorted lex by name, EXCLUDING `x-amz-trailer-signature` itself but
    INCLUDING the algo's checksum header. Missing this exclusion would
    flip the chain into a self-referential mismatch.
  - On EOF, mismatches surface as `ErrSignatureInvalid` (AWS parity),
    not as `ErrChecksumMismatch`. AWS S3 collapses both into the
    SignatureDoesNotMatch wire code; preserving that on the gateway
    avoids confusing clients that expect the SDK-shaped retry classes.
  - `httptest.Server` flips the request `Host` to `127.0.0.1:PORT` on
    the wire. When constructing a SigV4-signed request to drive against
    the test server, sign the canonical request with that host (not
    `example.com`) — otherwise the gateway recomputes against the wire
    host and trips ErrSignatureInvalid. The integration test embeds
    this gotcha in `buildTrailerPUT(..., host, ...)`.
  - `internal/racetest` `TestRunCleanRunNoInconsistencies` is a 13–45 s
    race-fuzz oracle that exercises the streaming-payload PUT path. It
    is sensitive to scheduling under full-suite race contention; pass
    rate is high in isolation. Future flakes here should reproduce
    against clean HEAD before blaming the WIP — the streaming-payload
    wire shape itself is unaffected by US-009.
