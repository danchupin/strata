- **Learnings for future iterations:**
  - **`gh api repos/<owner>/<repo>/git/refs/tags/<tag>` returns the tag-ref object SHA, NOT the commit SHA when the tag is annotated.** Two-hop resolution required: `gh api .../git/refs/tags/<tag> тЖТ .object.sha` then `gh api .../git/tags/<that-sha> тЖТ .object.sha` yields the commit SHA. Annotated tags are the upstream convention for both `trivy-action` and `codeql-action`. Lightweight tags would resolve in one hop (`.object.type == "commit"` directly). Always check `.object.type` first тАФ `tag` = annotated, `commit` = lightweight. SHA-pinning to the tag-ref SHA instead of the commit SHA technically works but is brittle: a tag-rewrite changes the tag-ref SHA, breaking the pin. Pin to the commit SHA underneath.
  - **`aquasecurity/trivy-action` v0.36.0** (latest stable as of 2026-05-28) supports `trivy-config: <path>` input which loads a `trivy.yaml` shape. Action's other knobs (`severity`, `exit-code`, `format`, `output`) layer on top тАФ the config file is for cycle-wide policy (scanners, timeout), the action inputs are for per-pass gates. Two passes (CRITICAL + HIGH) is the standard split тАФ single pass with `severity: 'CRITICAL,HIGH'` would conflate gate semantics.
  - **`github/codeql-action/upload-sarif` `category` input** distinguishes multiple SARIF uploads in the same job тАФ GitHub's Code Scanning UI merges entries by category. Setting `category: 'trivy-critical'` keeps this gate distinct from future SARIF emissions (US-003 gosec will use `category: 'gosec'`).
  - **`if: always()`** on the SARIF upload + HIGH WARN-only pass + artifact upload steps ensures the workflow continues past the CRITICAL gate failure тАФ operators see the full HIGH-class findings + Code Scanning entry even when the gate hard-fails. Without `if: always()`, a CRITICAL failure short-circuits the job and Code Scanning never receives the SARIF.
  - **`trivy-image` rebuilds the image even though `docker-build` already built it** тАФ buildx cache layers are job-local on GitHub Actions runners. Cross-job image artifact passing via `docker save | docker load` would transfer multi-GB tarballs over the Actions artifact backend, dominating job wall-clock. Rebuilding under `docker/setup-buildx-action@v3` warm cache is faster + simpler. Same trick as `e2e` / `e2e-full`. The PR review-time wall-clock delta is ~3-5 min image build + ~30s trivy scan тАФ fits the +30-60s budget per S4 from PRD review.
  - **`scripts/check-trivy.sh` does NOT mirror the `check-govulncheck.sh` matrix axis pattern** тАФ trivy scans the published image (`strata:ceph`), not Go source modules. The cephimpl module shape is irrelevant at the image layer тАФ the final image already linked cephimpl via the `-tags ceph` build inside the Dockerfile. One image, one scan, one gate. The matrix axis is only relevant for Go-source-level scanners (govulncheck, gosec, go-licenses) where the cephimpl module is a separate Go module per CLAUDE.md.
  - **trivy 0.36 image-scan `format: 'sarif'`** writes SARIF v2.1.0 with `runs[0].tool.driver.name = 'Trivy'` тАФ codeql-action SARIF upload validator accepts this out-of-the-box. No conversion step needed.
