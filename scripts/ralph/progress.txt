## 2026-05-01 - US-005 — Listing delimiter+prefix interaction
- internal/meta/memory/store.go::ListObjects + internal/meta/cassandra/
  store.go::ListObjects now (1) skip a CommonPrefix `pfx` when
  `pfx <= Marker` (so a continuation page does not re-emit the prev
  page's CP via roll-up) and (2) report `NextMarker` on truncation as
  the LAST EMITTED key/CP, not the next-candidate CP about to be
  rejected by the limit gate. Also marks the skipped CP as seen so
  follow-on keys folding into it are filtered cheaply.
- internal/s3api/server.go::listObjects now branches on
  `list-type=2`: V2 keeps the prior ListBucketResult / KeyCount /
  NextContinuationToken / ContinuationToken / StartAfter shape. V1
  (no list-type) emits a NEW `listBucketResultV1` XML (added in
  internal/s3api/xml.go) carrying `<Marker>`, `<NextMarker>` (only
  when truncated AND delimiter set per AWS spec), `<MaxKeys>`,
  `<Delimiter>`, `<IsTruncated>`, `<Contents>`, `<CommonPrefixes>`.
  V1 reads `?marker=` from the query, V2 reads `?continuation-token=`
  with `?start-after=` fallback.
- internal/meta/storetest contract suite: new case
  caseListDelimiterPrefixPaging mirrors s3-tests
  `test_bucket_list_delimiter_prefix` shape (asdf, boo/bar,
  boo/baz/xyzzy, cquux/thud, cquux/bla) — 3-page max-keys=1 paging
  with prefix='' AND prefix='boo/'. Runs against memory + cassandra
  (under -tags integration) lockstep.
- internal/s3api/list_delimiter_prefix_test.go: TestListV1Delimiter
  Prefix mirrors test_bucket_list_delimiter_prefix exactly (V1
  Marker / NextMarker semantics). TestListV1DelimiterPrefixEndsWith
  Delimiter mirrors test_bucket_list_delimiter_prefix_ends_with_
  delimiter. TestListV2DelimiterPrefix mirrors
  test_bucket_listv2_delimiter_prefix.
- **Learnings for future iterations:**
  - V1 vs V2 MUST be detected by `list-type=2` query param. V1
    response shape has `<Marker>` + `<NextMarker>`; V2 has
    `<ContinuationToken>` + `<NextContinuationToken>` + `<KeyCount>`
    + `<StartAfter>`. boto3 `client.list_objects` (V1) reads
    `<NextMarker>`; without it response['NextMarker'] is None.
  - V1 NextMarker is included in the body ONLY when truncated AND a
    delimiter was provided. Without delimiter, V1 clients use the
    last key as the next marker. Setting NextMarker unconditionally
    breaks AWS-conformant clients.
  - On truncation, NextMarker = the LAST EMITTED key/CP — NOT the
    next-candidate that the limit gate rejected. The gate must sit
    AFTER the seenPrefixes + Marker checks, otherwise it pre-stamps
    NextMarker to a CP that wasn't returned.
  - When Marker IS itself a CommonPrefix the prev page returned
    (e.g., "boo/"), the next page's roll-up must skip CPs where
    `pfx <= Marker`. Without this filter the new page re-emits
    "boo/" because its seenPrefixes map is per-call.
  - Object keys in URLs need URL-encoding when they contain a slash
    that isn't a path separator (e.g., key="asdf/" → PUT path
    "/bkt/asdf%2F"). Otherwise the router strips the trailing
    slash and the key becomes "asdf".
  - meta.ListResult.NextMarker semantics: "" means not-truncated;
    the V1 XML guard prevents emitting `<NextMarker></NextMarker>`
    via `omitempty`.

## 2026-05-01 - US-004 — Multipart copy FlexibleChecksum path

  `x-amz-checksum-algorithm` from the request (falls back to

  ReadCloser is wrapped in `io.TeeReader(rc, hash.Hash)` so the named

  `ChecksumAlgorithm` (the algo column is left blank when the digest
  is empty so opt-out rows stay clean) and is echoed on the response
  via the `x-amz-checksum-<algo>` header. From there US-001's
  PartChunks pipeline carries it into Complete's COMPOSITE formula.
- New `ErrBadDigest` (HTTP 400, "BadDigest") in errors.go.
- Test coverage in internal/s3api/multipart_copy_checksum_test.go:
  TestMultipartCopyFlexibleChecksum runs SHA256/SHA1/CRC32/CRC32C

  XML check). TestMultipartCopyChecksumMismatch pins BadDigest on a
  wrong client digest. TestMultipartCopyInheritsUploadAlgorithm pins
  algo inheritance from the multipart session.
- **Learnings for future iterations:**
  - `io.TeeReader` is the right pattern for "hash bytes while PutChunks

## 2026-05-01 - US-003 — Per-part composite-checksum response shape
- Added MultipartChecksum{Algorithm,Type,} fields onto data.Manifest
  with `omitempty` JSON tags + protobuf field numbers 19/20/21
  documented in manifest.proto.
- Added ChecksumAlgorithm + ChecksumType to meta.MultipartUpload and
  ChecksumValue + ChecksumAlgorithm to meta.MultipartPart.
- Cassandra schema.go ALTERs add checksum_algorithm + checksum_type to
  multipart_uploads, checksum_value + checksum_algorithm to
  multipart_parts. INSERT/SELECT on CreateMultipartUpload /
  GetMultipartUpload / ListMultipartUploads / SavePart / ListParts
  carry the new columns.
- internal/s3api/checksum.go adds normalizeChecksumAlgo,
  newChecksumHasher, checksumHeader, compositeChecksum helpers.
- initiateMultipart reads `x-amz-checksum-algorithm` +
  `x-amz-checksum-type` (default COMPOSITE when algo set), stamps
  onto MultipartUpload. Invalid type → ErrInvalidArgument.
- uploadPart captures `x-amz-checksum-<algo>` onto MultipartPart
  (both native and US-010 backend pass-through paths) and echoes on
  the response. Native path stores via PutChunks; pass-through
  routes to UploadBackendPart. Same fields reach PartChunks at
  Complete time.
- completeMultipart: when algorithm is set, computes COMPOSITE hash-
  of-hashes from per-part digests (algorithm-specific:
  sha256/sha1/crc32-IEEE/crc32-Castagnoli; raw digest concat → hash →
  base64 → `-<numparts>`), OR adopts client-supplied
  `x-amz-checksum-<algo>` from the Complete request for FULL_OBJECT.
  Stamps onto obj.Manifest.MultipartChecksum* as composite-checksum
  hints; meta layer copies them onto the persisted manifest.
- Meta CompleteMultipartUpload preAssembled sentinel tightened from
  `obj.Manifest != nil` to `obj.Manifest != nil && obj.Manifest.BackendRef != nil`
  so a hint-only manifest does not short-circuit chunks-shape
  assembly. Both memory + cassandra mirror.
- completeMultipartResult XML expanded with ChecksumCRC32 / CRC32C /
  SHA1 / SHA256 / ChecksumType (`omitempty` so opt-out users see the
  pre-cycle response shape unchanged).
- getObject (HEAD/GET, no partNumber) emits
  `x-amz-checksum-<algo>` + `x-amz-checksum-type` from the persisted
  manifest when `x-amz-checksum-mode: ENABLED`.
- getObjectPart (?partNumber=N) emits the per-part checksum from
  PartChunks[N-1].ChecksumValue under the same opt-in.
- Test coverage in internal/s3api/multipart_checksum_test.go:
  TestMultipartFlexibleChecksumComposite covers SHA256/SHA1/CRC32/
  CRC32C round-trip across UploadPart echo, Complete XML composite,
  HEAD composite, HEAD partNumber. TestMultipartFullObjectChecksum
  pins the FULL_OBJECT pass-through. TestMultipartChecksumOptOut
  pins the opt-out negative.
- **Learnings for future iterations:**
  - `obj.Manifest` was a binary "preAssembled" sentinel; this story
    needed it to also carry composite-checksum hints in the chunks-
    shape native path. Tightened sentinel to BackendRef!=nil. Future
    additions that hitchhike on obj.Manifest as a hint blob can
    layer cleanly without re-touching the sentinel.
  - The composite digest format MATCHES the named algorithm
    (SHA256-of-hashes for SHA256, CRC32-of-hashes for CRC32, etc.) —
    NOT always SHA-256 as the PRD note implies. Each algo hashes
    its own raw-digest concatenation. Wire format `<base64>-<n>`
    for COMPOSITE; FULL_OBJECT is `<base64>` (no suffix) and only
    legal for CRC variants in modern AWS SDKs.
  - x-amz-checksum-<algo> header values are base64 of RAW digests
    (binary), not hex. Decode to bytes before concatenation.
  - HEAD without `x-amz-checksum-mode: ENABLED` must NOT emit
    checksum headers — boto's auto-validator otherwise re-validates
    against the body and trips when the SDK didn't expect a
    composite.
  - Cassandra ALTER list grows; `nilIfEmpty` keeps zero-value
    columns NULL so legacy rows (no algorithm) decode as
    `ChecksumAlgorithm == ""`.

## 2026-05-01 - US-002 — ?partNumber=N GET serves the exact part body
- Added `Server.getObjectPart` helper in internal/s3api/server.go.
  Dispatched from `getObject` when `?partNumber=N` query param is
  present (and the existing `uploadId` branch did NOT already steal
  the request — that path is intercepted higher up in handleObject).
- Reads PartChunks[N-1] for Offset/Size/ETag. 416
  `InvalidPartNumber` for: non-multipart object (`PartChunks` nil/0),
  N <= 0, N > len(PartChunks), unparseable N.
- New `ErrInvalidPartNumber` (HTTP 416) in internal/s3api/errors.go.
- Headers set on success (GET + HEAD): `x-amz-mp-parts-count`,
  `Content-Length=<part size>`, `ETag` of the PART (not whole
  object — falls back to whole-object ETag only if PartChunks
  entry has empty ETag, which won't happen for memory/cassandra
  meta but defends pre-existing rows), `Content-Range: bytes
  <abs-off>-<abs-end>/<whole-object-size>`. SSE + version-id
  headers mirror the regular GET path. Status is always 206
  Partial Content.
- Range header stacks part-relative: parseRange is run against
  pr.Size (so `bytes=0-9` inside part 2 yields part2[0:10]); the
  absolute byte range passed to Data.GetChunks is shifted by
  pr.Offset; Content-Range absolute against the whole-object
  total (matches AWS observable: a 206 always reports absolute
  coords, never part-relative).
- Skips the presign-redirect short-circuit — the backend URL
  would not respect part semantics.
- Test coverage in internal/s3api/multipart_partnumber_test.go:
  3-part MPU (5MiB each). Subtests: part-2 body match, HEAD
  part-3 headers, part-relative Range, part ETag differs from
  whole-object ETag, 416 for N=4 / N=0 / N=-1, 416 for
  single-PUT object.
- **Learnings for future iterations:**
  - `?partNumber=N` + `uploadId` is intercepted in handleObject
    and routed to UploadPart — getObject only sees partNumber
    when uploadId is absent. Don't second-guess this in the
    GET handler.
  - 416 wire body is `<Error><Code>InvalidPartNumber</Code>...`
    not `InvalidRange` (S3 distinguishes these — Range
    out-of-range is `InvalidRange`, partNumber out-of-range is
    `InvalidPartNumber`). Both use HTTP 416.
  - The status MUST be 206 (not 200) for a partNumber GET so
    AWS SDKs treat the response as a Range response and
    consult Content-Range. Returning 200 will make boto's
    autochecker think the body is the whole object.
  - When stacking Range on top of partNumber, Content-Range's
    `<total>` field is the whole-object size (not the part
    size). The relevant invariant: every 206 from S3 reports
    absolute coords, never part-relative — matches what the
    existing TestObjectRangeGet asserts for non-partNumber
    206.
  - HEAD must set Content-Length to the part size (not 0) so
    SDKs can pre-allocate. Standard net/http does NOT strip
    Content-Length on HEAD when WriteHeader is called
    explicitly — set it directly.
  - Adding the partNumber branch BEFORE
    maybeBackendPresignRedirect is load-bearing: the presign
    URL can't carry partNumber semantics, and a redirect
    would short-circuit the PartChunks lookup.
