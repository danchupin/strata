- internal/meta.Store interface is the contract; both Cassandra and TiKV
  must implement it in lockstep. Memory backend is the test reference.
  Use internal/meta/storetest/contract.go as the parity oracle.
- Cassandra LWT lesson: any UPDATE on a row that may be read with quorum
  AFTER a previous LWT must itself be LWT (IF EXISTS). Read CLAUDE.md
  "Cassandra gotchas" before touching meta semantics. The TiKV equivalent
  is "any update that needs read-after-write coherence must use a
  pessimistic txn, not a plain Put".
- Schema migrations are additive. Cassandra: alterStatements with
  IF NOT EXISTS column adds, swallowed-error helpers. TiKV: schema is
  encoded into key prefixes (US-002); "migration" = define new prefix +
  encoder/decoder. Old code deserialises rows it knows about, ignores
  the rest.
- Bucket-level config endpoints (lifecycle / CORS / policy / etc) all
  use the setBucketBlob/getBucketBlob/deleteBucketBlob helper. Reuse
  it; do not write fresh CRUD per endpoint.
- ROADMAP maintenance rule (CLAUDE.md): every commit closing a
  ROADMAP.md item flips the bullet to strikethrough Done format in the
  SAME commit, with closing SHA inline (or `(commit pending)` with a
  follow-up SHA edit). Every commit discovering a new gap adds a new
  entry under the right severity section in the same commit.
- Tests in this cycle that need a TiKV testcontainer: use the
  same testcontainers-go pattern as internal/meta/cassandra/store_integration_test.go.
  Build tag `integration`. PD + TiKV in one container set; ~30 s start.
- Stub-package trick to keep `go mod tidy` from dropping a dep that has
  not yet been used: blank-import the smallest leaf subpackage from the
  vendor (we used `_ "github.com/tikv/client-go/v2/config"` in the
  US-001 stub). Replace with the real import once a story uses the
  txnkv API for round-trip work.
- Compile-time interface assertion `var _ meta.Store = (*Store)(nil)`
  at the bottom of internal/meta/tikv/store.go catches every
  meta.Store interface drift in CI as soon as the interface gains a
  method. Keep it across stories.
- Variable-length string segments in TiKV keys use FoundationDB-style
  byte-stuffing (0x00 → 0x00 0xFF, terminator 0x00 0x00) — preserves
  lex ordering across heterogeneous lengths so prefix range scans work
  correctly. See internal/meta/tikv/keys.md for the full layout. Never
  add length-prefixed encoding to a key — it breaks "all keys starting
  with prefix X" scans.
- Object version-DESC ordering: 24-byte suffix [MaxUint64-ts8-BE]
  [raw-uuid-16]. Inverted ts makes ascending range scan emit latest
  version first (free GET-without-versionId path). Null sentinel UUID
  (timestamp 0) sorts last among versions of a key — gateway resolves
  "?versionId=null" by exact lookup, not scan position.
- Fixed-width integer fields in TiKV keys (partNumber, day-epoch,
  shardID) use big-endian uintN so forward range scans return them in
  ascending numerical order. Never use little-endian or varint there.
- internal/meta/tikv has a kvBackend / kvTxn abstraction (kv.go,
  kv_tikv.go for production *txnkv.Client wrapping, kv_mem.go for
  unit-test in-process backend). Per-section files (buckets.go, ...)
  call s.kv.Begin(ctx, pessimistic) and treat the txn handle as
  Get/Set/Delete/Scan/LockKeys/Commit/Rollback only. Story-by-story
  fills in section files; do not bypass the abstraction. openWithBackend
  is the test-only constructor that injects memBackend.
- Bucket-row mutators (SetBucketVersioning, SetBucketACL,
  SetBucketObjectLockEnabled, SetBucketRegion, SetBucketMfaDelete) all
  go through buckets.go::updateBucket — pessimistic txn that LockKeys +
  Get + mutate + Set + Commit. Plain Set after a previous LWT-style
  CreateBucket would risk read-after-write incoherence; mirror
  CLAUDE.md's Cassandra LWT lesson on the TiKV side.
- meta.Bucket persists as JSON via internal/meta/tikv/codec.go. JSON
  keeps ops debugging cheap, additive (new fields with json:omitempty
  skip an ALTER), and round-trips uuid.UUID + time.Time without custom
  marshallers. Internal storage only — wire formats stay XML.
- Use internal/meta/tikv/buckets.go::prefixEnd helper for any
  "lex starts with prefix" range scan upper bound. Locally implemented
  (mirrors kv.PrefixNextKey) so test code does not pull the txnkv tree.
- Pessimistic txn boilerplate in tikv: defer rollbackOnError(txn, &err)
  immediately after Begin, then assign the named return / outer err on
  every step. Commit is the last statement; if it succeeds err stays
  nil and rollbackOnError is a no-op.
- Pessimistic txns with EARLY-RETURN paths (e.g. SetObjectStorage CAS
  reject returning applied=false, nil) MUST call txn.Rollback() before
  returning. rollbackOnError fires only when err != nil; without an
  explicit Rollback the LockKeys lease leaks for the txn lifetime, and
  in the in-process memBackend it deadlocks the next caller forever.
- Object row payload (codec.go::objectRow) is JSON {meta.Object fields,
  ManifestRaw []byte}. Manifest is encoded via data.EncodeManifest so
  STRATA_MANIFEST_FORMAT (proto vs JSON) flows through the same codec
  Cassandra uses. Reading uses data.DecodeManifest which sniffs the
  first non-whitespace byte → transparent format upgrade.
- Range-scanning all versions of a single user-space key:
  start = ObjectPrefixWithKey(bucketID, key) + 0x00 0x00 (the escaped
  terminator), end = prefixEnd(start). Captures every key whose
  escaped body equals `key` exactly, regardless of the 24-byte
  version-DESC suffix.
- Version-DESC ordering means: range scan with limit 1 over a key's
  prefix returns the lex-first encoded suffix = MaxUint64-ts8 with the
  SMALLEST inverted ts, i.e. the row with the BIGGEST timestamp. Real
  TimeUUIDs sort BEFORE the null sentinel (which has ts=0 → MaxUint64
  inverted). Same as Cassandra `version_id DESC` LIMIT 1 — null row
  surfaces only via explicit ?versionId=null lookup, never as latest
  while a TimeUUID-versioned row exists.
- Listing pagination on TiKV: batched Scan loop with cursor =
  lastKey + 0x00 between batches; exit when Scan returns fewer than
  the batch size or empty. Pass listScanBatchSize=1024. The encoded
  key for the "key marker" boundary is escaped(marker) + 0x00 0x00 (the
  terminator) — and prefixEnd of THAT captures every row of the marker
  key, so for ListObjects (exclusive marker) advance start to that
  prefixEnd; for ListObjectVersions (inclusive marker) advance start
  to escaped(marker) + 0x00 0x00 itself.
- Multipart Complete on TiKV is one pessimistic txn that holds TWO
  locks: the upload-row key (for the LWT-equivalent status flip
  uploading→completing) AND the final ObjectKey (so concurrent
  PutObject / Complete on the same final key cannot race). LockKeys
  is incremental — call it once at txn start with the upload key,
  then again post-status-flip when the final versionID is known.
  Status flip's Set is buffered until Commit, but LockKeys
  participates in the conflict graph immediately so concurrent
  Completers serialise pre-Commit; the loser observes
  ErrMultipartInProgress (or ErrMultipartNotFound if the winner
  already committed and removed the upload row).
- TiKV has no native TTL — for any "row expires after N" surface
  (RecordMultipartCompletion, audit log retention via US-009 sweeper,
  ...) persist ExpiresAt in the row payload and lazy-expire on Get.
  A periodic sweeper (US-009) handles eager deletion later. Mirror of
  Cassandra `USING TTL`.
- Per-bucket single-document config blobs (lifecycle/CORS/policy/
  public-access-block/ownership-controls/encryption/object-lock/
  notification/website/replication/logging/tagging) collapse on TiKV
  to one shared key prefix `s/B/<uuid16>/c/<kind>` where kind is a
  fixed 2-byte BlobX constant from keys.go. Per-config Set/Get/Delete
  are thin wrappers over setBucketBlob/getBucketBlob/deleteBucketBlob
  in blobs.go — never write fresh CRUD per endpoint, never pass a
  free-form kind string from a handler. Per-(bucket, configID)
  inventory configs use a separate sub-prefix `i/` with the configID
  byte-stuffed via appendEscaped so List can range-scan and decode
  configID via readEscaped after stripping the prefix.
- IAM access keys carry TWO rows (per-key record + per-user index).
  CreateIAMAccessKey writes both in one pessimistic txn so the index
  cannot desync (mirror of Cassandra's LoggedBatch atomicity);
  DeleteIAMAccessKey reads first to learn UserName, then deletes both
  keys. Cassandra schema does NOT use IF NOT EXISTS on access-key
  insert — TiKV matches that (no ErrAccessKeyAlreadyExists sentinel
  exists; access-key IDs are gateway-minted opaque tokens, collisions
  are not realistic). For ListIAMAccessKeys with a non-empty userName,
  range-scan the index prefix, readEscaped the accessKeyID out of the
  row body (after stripping idxPrefix), then Get each per-key row.
  Empty userName falls back to scanning prefixIAMAccessKey directly +
  in-process sort by AccessKeyID — matches memory backend semantics.
- Audit log on TiKV stamps each row with ExpiresAt at write time (TiKV
  has no native TTL). Readers (ListAudit, ListAuditFiltered,
  ListAuditPartitionsBefore, ReadAuditPartition) lazy-skip expired rows
  so a delayed sweeper tick never surfaces stale data; the leader-elected
  AuditSweeper goroutine in sweeper.go eager-deletes them in the
  background. Both halves are required — lazy filter alone leaks disk;
  sweeper alone leaves a window where reads see stale rows.
- Pattern for any "sweep N rows then more next batch" deletion on TiKV:
  open a fresh Begin per batch, Scan(cursor, end, batchSize), Delete each
  row that fails the keep predicate, Commit, advance cursor to lastKey +
  0x00. Bounded txn size keeps a partition with millions of rows from
  exceeding TiKV's per-txn limit and lets ctx cancellation bail between
  batches without leaking a long-held lease. See deleteAuditRange in
  audit.go.
- Worker queues (notify / notify-DLQ / replication / access-log / GC) on
  TiKV use one shared shape: key = `<prefix><partition><tsNano8-BE>
  <eventID-escaped>`. Big-endian ts8 makes ascending range scan claim
  oldest-first; escaped eventID breaks ts ties. Per-bucket queues use the
  16-byte bucket UUID as partition; the GC queue uses the byte-stuffed
  region string. Ack recomputes the same key from the entry's timestamp
  + EventID and Deletes the row — no per-row secondary index. Means the
  caller MUST round-trip the entry back through List → Ack so the
  Time/EnqueuedAt + EventID survive (JSON RFC3339Nano preserves
  nanoseconds; UnixNano matches across encode/decode). Memory backend's
  Ack-by-EventID linear scan is the divergent shape — both end up with
  the same observable behaviour. See queues.go.
- Cross-process leader-election Locker on TiKV (locker.go) is a
  pessimistic-txn RMW over LeaderLockKey(name): row payload is
  {Holder, ExpiresAt}. Acquire = LockKeys + Get + (CAS on holder
  match OR ExpiresAt < now) + Set; Renew = LockKeys + Get + assert
  Holder + Set; Release = LockKeys + Get + assert Holder + Delete.
  The pessimistic lock is held only for the RMW step — persistent
  ExpiresAt is what a new gateway uses to steal an abandoned lease.
  Plug into serverapp.buildLocker via NewLocker(store) when US-015
  wires STRATA_META_BACKEND=tikv.
- Bucket-by-ID lookup inside a TiKV txn: scan `s/b/` to find the row
  key, LockKey it, RE-Get to obtain the authoritative pre-image,
  decode, mutate, Set. The Scan alone is read-only and may race with
  concurrent SetBucketVersioning / SetBucketRegion etc. memBackend
  won't enforce the discipline, but real TiKV's pessimistic-txn
  for_update read does. See findBucketKeyByID in reshard.go.
- Cross-bucket list operations (ListReshardJobs, ListAccessPoints,
  sweepers) are cheaper off a global key prefix than a per-bucket
  scope + filter. When the surface needs a "list across all buckets"
  path, promote the row to a global prefix instead of writing a
  secondary index. Reshard job key got promoted from scoped
  `s/B/<uuid16>/R` to global `s/Rj/<bucket16>` in US-011 for this
  reason; cursor/lifecycle/inventory worker keys stayed scoped
  because those workers operate per-bucket.
- testutils.NewMockTiKV (client-go/v2/testutils + tikv.NewTestTiKVStore)
  is NOT a full transactional fake — pessimistic-txn Commit hangs on
  heartbeat indefinitely against the in-process mock, even though
  LockKeys/Get/Set succeed. Use real PD+TiKV containers for any
  contract-level exercise; reserve the mock for low-level RPC bench
  (which is what tikv/kv_test.go in upstream does — single-Get RPC
  shape, no commit). Memory backend (internal/meta/memory) is the
  parity oracle for surface contract.
- TiKV testcontainer networking: PD's advertise-client-url and TiKV's
  advertise-addr must be reachable from BOTH PD-the-container AND the
  test client. The pattern that works across macOS/Linux/Lima docker
  contexts is `host.docker.internal:<host-port>`. In containers add it
  via HostConfigModifier { ExtraHosts: "host.docker.internal:host-gateway" };
  on the host it resolves on Docker Desktop natively but on Lima docker
  setups it does NOT resolve from the macOS host — so the test must be
  prepared to t.Skipf when the gateway alias misses. CI workflow
  (US-017) sidesteps this by setting STRATA_TIKV_TEST_PD_ENDPOINTS
  directly against an existing PD it brought up via docker-compose
  (US-014), bypassing testcontainers entirely.
- docker-compose profiles cannot toggle env on a single service; mutually-
  exclusive shapes (cassandra-backed strata vs tikv-backed strata) get
  distinct service names sharing the same image. Compose's `profiles: [...]`
  gates whether a service is started, so profile-tagged services live
  alongside the default ones (no per-profile override of env on a default
  service). Pattern: bake the differing env onto each variant, publish on
  distinct host ports so both can coexist for comparison/debugging. See
  strata-tikv (profile `tikv`) vs strata (default) in
  deploy/docker/docker-compose.yml.
- TiKV's upstream container image (pingcap/tikv:v8.5.0) has no clean HTTP
  healthcheck contract — the status server returns plain text whose body
  is not a public API, and the alpine-glibc base ships no curl. For
  docker-compose the simplest portable wait shape is a host-side TCP probe
  on the published RPC port (`make wait-tikv` uses bash `/dev/tcp`). The
  PD service does have a stable HTTP health endpoint
  (`/pd/api/v1/health`), so its compose healthcheck is HTTP-shaped and
  downstream services can rely on `depends_on: pd: service_healthy`. TiKV
  downstream consumers use `service_started`; the TiKV client retries
  until PD assigns regions, so transient boot races absorb in the
  application layer.

- **Learnings for future iterations:**
  - The in-process MockTiKV (client-go/v2/testutils.NewMockTiKV +
    tikv.NewTestTiKVStore) is NOT a full transactional fake. It
    handles low-level RPC (Get / Put / PessimisticLock /
    TxnHeartBeat) but Commit() of a pessimistic txn never resolves —
    heartbeats fire forever while prewrite sits unanswered. We
    discovered this by writing the test against the mock first; the
    BucketLifecycle case stalled at the first CreateBucket Commit
    with `[INFO] [2pc.go:1187] ["send TxnHeartBeat"] [newTTL=30000]`
    and never progressed. Use real PD+TiKV for anything that exercises
    Commit; reserve the mock for tests that only touch the RPC layer.
  - testcontainers-go panics (rather than returns) when
    DOCKER_HOST is unset on macOS+Lima setups. Wrap the whole
    setUpTiKVCluster in `defer func() { if r := recover(); r != nil
    { err = fmt.Errorf(...) } }()` so the caller can t.Skipf cleanly.
  - host.docker.internal does NOT resolve from the macOS host on
    Lima docker setups (only inside containers via host-gateway).
    That breaks the "advertise-as-host.docker.internal" pattern for
    test clients running on macOS — they cannot reach the published
    TiKV port through that name. CI runners on Linux / Docker
    Desktop don't have this issue. The test handles this by t.Skipf
    when probe fails, and CI workflow (US-017) bypasses
    testcontainers via STRATA_TIKV_TEST_PD_ENDPOINTS.
  - Wait probes (wait.ForHTTP / wait.ForLog) need the actual
    container port the service is bound to, not the upstream
    default. PD here listens on a dynamically-allocated host port
    (so concurrent test runs do not collide), and the wait probe
    must reflect that — `WithPort(nat.Port(fmt.Sprintf("%d/tcp",
    pdPort)))`, not `"2379/tcp"`. The first iteration of the test
    used the default and timed out without diagnostic.
  - `freeTCPPort()` reservation pattern: net.Listen("tcp",
    "127.0.0.1:0") + Close + return port. Race-free for one-shot
    use because the OS won't immediately reuse the just-freed port,
    and the testcontainers ExposedPorts publish step takes ms. Used
    here so PD and TiKV claim distinct host ports without collision.
  - Each test run gets a fresh PD+TiKV cluster (via testcontainers
    Cleanup) so subsequent runs don't see stale buckets from the
    last invocation. The contract cases use unique bucket / object
    names per case so cross-case state leak inside a single run is
    bounded; no per-subtest cluster reset needed.

## 2026-04-29 - US-011
- Reshard state on TiKV: StartReshard / GetReshardJob /
  UpdateReshardJob / CompleteReshard / ListReshardJobs. Lives in
  internal/meta/tikv/reshard.go.
- Reshard job key promoted to a global prefix `s/Rj/<bucket16>` (was
  reserved as bucket-scoped `s/B/<uuid16>/R`; never used). Global
  prefix collapses ListReshardJobs to one ordered range scan instead
  of "scan every bucket-scoped row and filter" — mirrors the Cassandra
  reshard_jobs partitioning shape. The bucket-scoped `subReshardCursor`
  key shape stays reserved for a future per-shard cursor story.
- StartReshard runs a single pessimistic txn that LockKeys the
  bucket-by-name row + the global reshard job key, re-Gets bucket at
  the for_update timestamp, validates target > current ShardCount,
  asserts no job present, then writes BOTH the job row + the bucket
  row (TargetShardCount = target). Concurrent StartReshard for the
  same bucket waits at lock-acquire and observes the persisted job on
  the second Get → ErrReshardInProgress.
- CompleteReshard atomically flips bucket.ShardCount to job.Target,
  zeros TargetShardCount, deletes the job row.
- UpdateReshardJob is the resize-worker watermark setter — pessimistic
  RMW with LockKeys + Get + Set; only LastKey/Done/UpdatedAt move
  forward. ErrReshardNotFound when no job is in flight (Cassandra
  IF EXISTS mirror). Nil job is a no-op.
- New findBucketKeyByID helper scans `s/b/` inside the same txn and
  returns the row key — caller then LockKeys + re-Get for fresh
  pre-image. The Scan alone is read-only and may race with concurrent
  bucket-row mutators; the LockKey+re-Get pattern is the correctness
  bar for the real txnkv path (memBackend doesn't enforce it but the
  contract suite US-013 does).
- Cross-process leader-election Locker on TiKV: NewLocker(store)
  returns a leader.Locker implementation backed by the same kvBackend
  the store uses. Lives in internal/meta/tikv/locker.go.

  - The escape transform's prefix-preservation is what makes the
    one-shot range scan correct: user-space "prefix" of a key always
    yields a byte-prefix of the encoded form, because byte-stuffing is
    pointwise (each byte maps to itself or 0x00 → 0x00 0xFF). Means
