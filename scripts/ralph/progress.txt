- **Learnings for future iterations:**
  - **PoolStatus.cluster field flows JSON.omitempty → TS `cluster?: string`.** When the Go side uses `omitempty`, the matching TypeScript optional `?` is the right shape — not `string | null`. Memory-backend rows omit the key entirely; rados/s3 rows include it. The Storage table cell handles both via `p.cluster ? render : '—'`.
  - **The DataTab sort contract is explicit even when the server pre-sorts.** rados/health_helper.go::buildPendingPoolStatuses already returns rows sorted by (cluster, pool, ns), but the UI re-sorts ascending with empty-last anyway. This is intentional — it makes the table's order self-evident at the call site and tolerates a future backend that doesn't pre-sort. Cost: one memoised slice() + sort() per render of a tab the operator only opens occasionally.
  - **`ClustersSubsection` only mounts when backend !== 'memory'.** Memory backend has no registered clusters and `/admin/v1/clusters` returns `[]` — rendering the subsection would just show an empty-state. The branch hides it instead, matching the existing `MemoryExplainer` flow that already replaces the pools table for memory backends.
  - **TanStack Query v5 `meta.label` is consumed by the global QueryCache error toast.** When adding a new useQuery, pass `meta: { label: 'short label' }` so failures show "Failed to load short label" instead of generic copy. The throttle in `web/src/lib/query.ts` dedups by query key so the toast doesn't storm during polling outages.
  - **`queryKeys.clusters` is a bare tuple, not a function.** This is intentional — there are no query params, and a stable identity means `useQuery({ queryKey: queryKeys.clusters })` from two components (ClustersSubsection + future PlacementDrainBanner in US-004) shares one fetch per refetch window.
  - **For US-004 next iteration**: the shared `queryKeys.clusters` key is in place; PlacementDrainBanner should reuse it with `refetchInterval: 15_000` (US-002 picked 10s, US-004 picked 15s — TanStack will use the smaller cadence when both are mounted, which is fine).

- **Learnings for future iterations:**
  - **No `<Slider>` or `<Tooltip>` primitives in `web/src/components/ui/`.** PRD AC mentioned them but they don't exist. Use native `<input type="range">` with Tailwind classes (`h-2 w-full appearance-none rounded-full bg-muted accent-primary`) and the native `title` attribute for tooltip-like hints. Don't add Radix Slider / Tooltip deps just to satisfy the AC text — the slider element + accent color is sufficient and stays within the bundle budget.

All three reads short-circuit through the same TanStack cache entry — don't define per-component cluster query keys.
  - **Two-way slider + number input binding is pure state.** Bind both inputs to the same `rows[id].weight` field; the slider's `onChange` updates state, the number input's `onChange` updates state, both re-render from the same source. No `useEffect` sync needed.
