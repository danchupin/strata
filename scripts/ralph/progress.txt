## Codebase Patterns
- RADOS data backend lives in `internal/data/rados/backend.go`. `PutChunks` (line 162) is the sequential-chunk-write hot path; `GetChunks` (line 241) returns a sequential `radosReader`. Both target US-001 / US-002 of this cycle.

- Memory data backend (`internal/data/memory/backend.go`) is tests-only — no production relevance. NOT touched by this cycle.
- `writeChunk` (`backend.go:221`) is the single-OID RADOS write helper. It wraps `goceph.CreateWriteOp().WriteFull().Operate()`. Wrap this call site in the worker goroutine; the rest stays the same.
- `ObserveOp(ctx, b.logger, b.metrics, b.tracer, spec.Pool, op, oid, start, err)` is the per-op observability hook. Fires AFTER the RADOS op returns. Must continue to fire from each worker so per-chunk duration metrics + tracer spans stay accurate (start time captured per chunk inside the worker, not in the dispatcher).
- ETag is a running MD5 over chunk bytes IN SOURCE ORDER (`hash.Write(chunk)` in the current sequential loop). Parallel impl MUST preserve this: a single owner goroutine reads the input stream chunk-by-chunk in order, feeds the hasher in order, dispatches `(idx, []byte)` work to workers, and seals `m.ETag = hex.EncodeToString(hash.Sum(nil))` only after all workers report done. NOT a per-worker partial-hash — that breaks the byte-stream-MD5 invariant.
- Manifest chunk order MUST equal source byte order. Workers can complete out of order; assemble via `m.Chunks = make([]data.ChunkRef, totalChunks)` and `m.Chunks[idx] = ...` from each worker — pre-size by counting reads, OR use an ordered append into a `[]data.ChunkRef` keyed by index with a single seal pass after `wg.Wait()`.
- Error semantics: on first worker error, cancel the context to stop in-flight workers, drain the pool, call `cleanupManifest` over chunks that DID land in RADOS. The current sequential code does `cleanupManifest(m.Chunks)` on error — preserve that exactly. Use `errgroup.WithContext` from `golang.org/x/sync` (already a transitive dep).
- Reader cancel: respect `ctx.Err()` at chunk-read boundary (already done in current code) AND at worker dispatch (don't block on a backed-up worker pool channel when the client gave up). Use `select { case workCh <- work: case <-ctx.Done(): return ctx.Err() }`.
- `STRATA_RADOS_PUT_CONCURRENCY` env knob (range [1, 256], default 32). Read in `rados.New` / `rados.Backend` constructor — NOT per-call. Follows the project's "worker envs read at Build time" convention (similar to `STRATA_GC_CONCURRENCY` per CLAUDE.md).
- `STRATA_RADOS_GET_PREFETCH` env knob (range [1, 64], default 4). Same constructor-time read. 4 × 4 MiB chunks = 16 MiB inflight memory budget per request.
- `radosReader` (line 260) holds `chunks []data.ChunkRef, pos, end int64, buf []byte, bufPos int`. Prefetch implementation: spawn a single fetcher goroutine that walks chunks in order, fetches up to N ahead, pushes `(idx, []byte, err)` onto an ordered channel; `Read` pulls from the channel head. On `Close`, cancel the fetcher's context + drain the channel. Test for goroutine leak via count-running-goroutines or `go.uber.org/goleak`.
- Range GET: first chunk fetched is the one containing `offset`; prefetch window stops at the chunk containing `offset+length`. The current `radosReader` already filters by `pos, end` — preserve that filtering when computing which chunks to prefetch.
- Tracer / observer: span per fetch must reflect the OSD read duration, NOT the wait-on-channel time. Start the timer INSIDE the worker right before `op.Operate`, observe right after — same as the current `start := time.Now()` pattern in PutChunks.

Both paths fold per-chunk ioctx resolution into the worker — no special-case multi-cluster code path.
