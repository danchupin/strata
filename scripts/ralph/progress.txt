## Codebase Patterns
- Crypto packages live under `internal/crypto/<area>/`. SSE-S3 master key code is in `internal/crypto/master/`; AEAD chunk + DEK wrap primitives are in `internal/crypto/sse/`.
- Provider interfaces return `(key, keyID, error)`. Callers (gateway main) own fatal-log on error — log-free providers are still preferred. **Exception:** when a provider needs to log non-fatal warnings (e.g. Vault refresh-failure with cached fallback), inject a `*slog.Logger` via the config struct and default to `slog.Default()`. Tests capture WARN by handing in `slog.New(slog.NewJSONHandler(&buf, …))`.
- Sentinel errors (e.g. `master.ErrNoConfig`, `master.ErrInvalidKeyLength`, `sse.ErrInvalidKeySize`, `sse.ErrShortWrappedDEK`) are exported from the package so callers and tests can `errors.Is` them.
- Use `t.Setenv` / `t.TempDir` for env-var and file-backed tests. For mtime-based hot-reload tests, set the cached mtime to the past via `os.Chtimes` before rewriting so the rewrite is observable on coarse-mtime filesystems.
- For external-service providers (Vault, AWS-KMS), inject a `Now func() time.Time` so TTL-based refresh logic is testable without `time.Sleep`. Fake `now` advances by `+= duration` between Resolves.
- Every test that calls `master.FromEnv()` must `t.Setenv` ALL precedence vars (vault, file, env) to a known value — otherwise an ambient process env (e.g. on a developer machine) flips precedence and tests pass/fail intermittently.
- AEAD layout: chunk ciphertext stored as bare `gcm.Seal` output (no nonce prefix — IV recomputed via `sse.DeriveChunkIV(oid, chunkIndex)` on decrypt). Wrapped DEKs stored as `nonce(12) || gcm.Seal output` because the wrap nonce is random.
- For SSE-S3 chunk alignment: pChunkSize = data.DefaultChunkSize - 16 (= 4 MiB - tag). Encrypt up to pChunkSize plaintext per crypto chunk → ciphertext = exactly 4 MiB → backend produces one backend chunk per crypto chunk (index aligned). On decrypt, walk manifest chunks via `sum(chunk.Size for j<i)` to map crypto-chunk i to its ciphertext byte range; chunk plaintext size = chunk.Size - 16.
- For SSE objects, the ETag persisted on the object row is the plaintext MD5; the data manifest's ETag/Size stay as the backend wrote them (= ciphertext). The s3api PUT response uses `obj.ETag`, not `m.ETag`. GET range parsing uses `o.Size` (plaintext). Don't unify the two — manifest-internal arithmetic in `data.GetChunks` still expects ciphertext-space.
- s3api harness in `testutil_test.go` attaches a default `harnessMasterProvider` so existing AES256 tests keep working — opt out by building a server manually if you need to test the no-master path (see `TestSSEMissingMasterReturns500`).
- `internal/data/memory.Backend.CorruptFirstChunk()` is a test-only helper for tampering tests. Memory backend only — RADOS backend has no equivalent (and shouldn't).
- Don't import `internal/crypto/sse` as `sse` inside `internal/s3api/server.go`; the file already has a local `sse` variable for the SSE algorithm string. Use the alias `ssecrypto` instead. Same caution applies whenever a file mixes SSE-algorithm-string vars with crypto package calls.
- Don't run `go mod tidy` casually inside Ralph stories — it pulls in unrelated module drift (koanf direct/indirect flips, transitive grpc/protobuf bumps). If a new package adds a direct import that already lives in `go.sum` as indirect, Go builds fine without retidying; leave the `// indirect` annotation alone unless the story explicitly requires a deps refresh.
- `data.Manifest` is JSON-encoded inside the meta backend (cassandra/codec.go is `json.Marshal/Unmarshal`). Adding a new field is **schema-additive** and needs zero migration if you tag it `json:",omitempty"`. Use this lever (e.g. `Manifest.PartChunks`) before adding a new Cassandra column.
- For multipart SSE: each part gets oid = `key + ":part=" + partNumber`, chunkIndex local-to-part. After Complete, the merged manifest carries `PartChunks []int` (= chunk count per part, in part order) so the GET path can reconstruct (partNumber, chunkIndexInPart) for any flat manifest chunk index. Pattern lives in `internal/s3api/sse_io.go::multipartChunkLocator` + `multipartPartOID`.
- For multipart objects, two sizes diverge: per-part `Size` is plaintext (used for ListParts, composite ETag, obj.Size) and per-part `Manifest.Size` is ciphertext (used by data.Backend GetChunks bounds). When constructing `obj.Manifest` in `CompleteMultipartUpload`, sum chunk sizes — not part sizes — to get ciphertext total. Same lesson as US-004 single PUT (obj.Size plaintext / manifest.Size ciphertext) but harder to spot because both backends used to use a single `totalSize` for both.
- Per-part ETag persisted on `multipart_parts` is plaintext MD5 of the part body (AWS-compatible). Composite ETag in Complete is built by concatenating the part-ETag bytes and re-MD5-ing — works on plaintext-MD5 inputs, exactly matching the unencrypted multipart contract. Don't accidentally store ciphertext ETag on the part row.
- Per-part state needed AFTER Complete (when multipart_parts rows are gone) lives on the object: `o.PartSizes []int64` (Cassandra `objects.part_sizes list<bigint>`) for plaintext part sizes, `o.Manifest.PartChecksums []map[string]string` (json:omitempty) for per-part stored x-amz-checksum-<algo>. Populated in CompleteMultipartUpload (both backends in lockstep). Helpers `partOffsetLength` + `partChecksumsAt` in `internal/s3api/multipart.go` derive (offset,length) and per-part checksum from N. `?partNumber=N` GET overrides Range entirely (status=206, Content-Range=bytes off-end/total, x-amz-mp-parts-count=len(PartSizes)). Reuse for US-026 ObjectParts subtree.
- Multi-key SSE-S3 master rotation: `master.RotationProvider` (in `internal/crypto/master/rotation.go`) holds an ordered list `entries[0]=active, entries[1..]=unwrap-only`. Server code never calls `Master.Resolve` for unwrap directly — it uses `master.ResolveByID(ctx, p, keyID)` which type-asserts to `Resolver` (rotation provider) and otherwise falls back to single-key Resolve with id-equality / legacy-empty-id handling. Same helper covers PUT-side unwrap of the upload row in multipart. Single-key providers (env, file, vault) keep their previous semantics.
- Operator-tooling-only meta methods (`UpdateObjectSSEWrap`, `UpdateMultipartUploadSSEWrap`, `Set/GetRewrapProgress`) live on `meta.Store` so both backends + the contract suite (`storetest.caseSSEWrapRotation`) cover them. Background workers like `cmd/strata-rewrap` consume them via the regular `meta.Store` interface, not a separate type-asserted interface — keeps the type-system load consistent across stories.
- For background workers, factor logic into `internal/<feature>` and keep `cmd/<feature>/main.go` as the wiring shim (config load, signal handling, build store/provider, hand off to the lib). See `internal/rewrap/rewrap.go` + `cmd/strata-rewrap/main.go` for the canonical split. The lib gets unit tests against the memory backend; the cmd binary stays untested. Mirrors the existing `internal/gc` + `cmd/strata-gc` pattern.
- Single-document bucket-config endpoints (CORS, policy, encryption, public-access-block, ownership-controls, lifecycle blob, …) all reuse `setBucketBlob` / `getBucketBlob` / `deleteBucketBlob` in `internal/meta/cassandra/store.go` + the memory backend's per-config maps. Handler reads bucket → calls `Set/Get/DeleteBucketX`. ErrNoSuch<X> sentinel + matching `ErrNoSuch<X>` APIError. resolveSSE in `internal/s3api/encryption.go` is the canonical "feature falls back to bucket default" pattern (header > config > none); copy this shape for any future bucket-default-applied-on-write feature.
- Server-side hooks (notification, replication-status, audit log) live next to the handler emitting them — `s.replicationStatusFor(...)` in server.go::putObject is the prior pattern, `s.emitNotificationEvent(...)` (US-008) is the same shape: best-effort, swallow `meta.ErrNoSuch*` silently, never fail the underlying request. Hook calls go AFTER the meta write succeeds and BEFORE `WriteHeader` so headers from the hook can still be set; for completeMultipart the hook fires before status emission too. When the hook needs request-scoped facts (sourceIP, principal), use the `clientSourceIP` / `principalFromContext` helpers in `internal/s3api/notification.go` rather than re-deriving.
- Buffer-and-drain queues (gc_queue, notify_queue) on the meta interface follow a 3-method shape: `EnqueueX(ctx, *X) error`, `ListPendingX(ctx, partition, limit) ([]X, error)`, `AckX(ctx, X) error`. Cassandra partitions by `(scope, time-bucket)` with TimeUUID clustering — gc_queue uses `(region, enqueued_at)` (timestamp clustering), notify_queue uses `((bucket_id, hour), event_id)` (TimeUUID clustering). For Cassandra ListPending the scan walks N most-recent time-bucket partitions inside the method (e.g. last 48 hours for notify); the worker is unaware of the partitioning. Re-use this shape for US-011 replication_queue and US-013 access_log_buffer.
- DLQ tables follow the same `((bucket_id, time-bucket))` partition shape as their live queue but with a coarser bucket — notify_queue uses `hour`, notify_dlq uses `day` because dropped events stay around longer (operator-inspectable). Pair the table with a `meta.<X>DLQEntry` struct that **embeds** the original event struct + adds Attempts/Reason/EnqueuedAt. Two-method interface (`EnqueueXDLQ`, `ListXDLQ`) — no Ack since DLQ is a forensic/inspect surface, not a delivery queue. Worker moves a row out of the live queue + into the DLQ atomically from the worker's POV (Enqueue then Ack); a crash between Enqueue and Ack causes one extra delivery attempt on next tick — acceptable for at-least-once semantics. Re-use for US-010 (DLQ inspection HTTP endpoint), US-014 (access_log buffer drain failure), US-046 (audit_export DLQ).
- Background worker dispatch loops follow a `Run(ctx)` ticker + `RunOnce(ctx)` single-pass split. `RunOnce` is exported so unit tests drive deterministic ticks instead of sleeping on a real ticker; `Run` is the production entry. The retry/backoff scheduler uses `Now func() time.Time` + `Backoff func(attempt int) time.Duration` injection so tests advance time + flatten backoff to zero without sleeps. Pattern: `internal/notify.Worker` (US-009); reuse for US-012 replicator + US-014 access-log + US-046 audit-export. Don't bury the loop body inside `Run` — pull it out so tests aren't fighting a ticker.
- Sink-style fan-out (notify, replicator targets, access-log targets) uses an interface (`Sink { Type() string; Name() string; Send(ctx, evt) error }`) + a `Router { Resolve(evt) (Sink, bool) }`. `StaticRouter` is `map[string]Sink` keyed by `<type>:<arn>` with arn-only fallback. Env-driven router parses `STRATA_<FEATURE>_TARGETS=<type>:<arn>=<spec>,...` where the spec is sink-specific (`<url>|<secret>` for webhooks). Router resolution failure (no sink for the row's TargetARN) routes the row straight to the DLQ with `reason='no sink registered for target'` rather than retrying forever — operators see misconfiguration immediately.
- HMAC delivery shape: lowercase-hex HMAC-SHA256 of the JSON body in `X-Strata-Signature`. `notify.ComputeSignature(secret, body) string` + `notify.VerifySignature(secret, body, header) bool` — the verify side hex-decodes then constant-time-compares with `hmac.Equal`. Same shape will be the auth pattern for US-022 (audit hook) and US-046 (audit export upload signing) — keep these helpers in `internal/notify` for now, hoist to `internal/crypto/hmac` only if a third caller materialises.
- External SDK injection without dep bloat: when a sink/feature needs an external SDK (AWS, GCP, ...), define a narrow interface in the `internal/<feature>` pkg covering only the methods used (`SQSAPI` = SendMessage only) + add a compile-time check `var _ NarrowAPI = (*sdk.Client)(nil)` in a test file. Expose an `EnvOption` (e.g. `WithSQSClientFactory(func(region string) (NarrowAPI, error))`) the cmd layer provides — that way the SDK lives in `cmd/<binary>/main.go` only, internal pkg tests keep using fakes, and operators that don't use that sink never compile the SDK code. `router_env.go` rejects sink-spec without its factory at startup so misconfiguration surfaces immediately, not on first event. Same shape will work for SNS, Kinesis, GCP Pub/Sub (US-014/046).

Re-marshalling wraps the payload in a notify-internal envelope and breaks consumer parity (SQS subscriber sees a different shape than webhook subscriber). Same rule for replicator destination payloads in US-012 (the source PUT body is the contract, not a re-encoded mirror).

Don't shortcut by relying on signed-request presence — `IsAnonymous` distinguishes signed-but-non-root from unsigned, and both should 403. Bucket-not-found uses `mapMetaErr` (returns NoSuchBucket); do not custom-map. Response is JSON `{records, next_continuation_token}` with `Content-Type: application/json` — XML stays for S3-spec endpoints; admin surfaces are JSON.

- Workers that copy / migrate objects (replicator US-012, future inventory US-039, audit export US-046) read source bytes via `meta.Store.GetObject` → `data.Backend.GetChunks(manifest, 0, size)` — same path the gateway GET uses. For SSE-encrypted objects this returns ciphertext (peer stores verbatim; no decrypt-then-re-encrypt). Always `defer src.Body.Close()` on every dispatcher return path — including early-return on validation errors — so the data backend reader is freed even when the dispatch never reaches the network. `MetricsObserver` interface pattern (`internal/<feature>` defines it; cmd binary provides a tiny prometheus-wrapping shim) keeps the internal pkg free of `prometheus/client_golang` while still emitting per-label histograms + counters.
- Worker `Name` for `worker_locks` (gc/lifecycle/notify/rewrap/replicator) MUST be unique per binary — a shared cassandra cluster shares the lock table, and reusing a name silently makes only one of N workers run. Convention: lock name = binary basename without `strata-` prefix. New workers (US-014 access-log, US-046 audit-export) keep this rule.
- slog migration: every binary calls `logging.Setup()` first thing in main; that returns a JSON `*slog.Logger` driven by `STRATA_LOG_LEVEL` (DEBUG/INFO/WARN/ERROR; default INFO) and installs it as `slog.Default()`. Workers (`leader.Session.Logger`, `gc.Worker.Logger`, `lifecycle.Worker.Logger`, `notify.Config.Logger`, etc.) take `*slog.Logger`, never `*log.Logger`. Replace `log.Printf("foo: %v", err)` with `logger.WarnContext(ctx, "foo", "error", err.Error())` — pass `ctx` through every call so the per-request logger from `logging.LoggerFromContext` carries `request_id`. - Per-action counter shape `<feature>_tick_total{action, status}`: `status` vocab is `success | error | skipped` where `skipped` covers CAS-race / idempotent no-op outcomes. `action` vocab is the verb-set the worker performs (transition / expire / expire_noncurrent / abort_multipart for lifecycle). Reuse this shape for any future per-outcome worker dashboard.
- Path-rewriting middlewares (vhost, future host-aliasing, future X-Forwarded-Host honouring) MUST run AFTER auth.Middleware in the handler chain — SigV4 signs both `r.Host` and `r.URL.Path`, so any rewrite before validation breaks signatures. The wrap order in `cmd/strata-gateway/main.go` is `mw.Wrap(s3api.NewAccessLogMiddleware(metaStore, apiHandler), s3api.WriteAuthDenied)`: auth first, then s3api (which contains the vhost rewrite block). - TTL-bearing meta methods take `ttl time.Duration` as a first-class parameter (not a config struct field) — `RecordMultipartCompletion(ctx, rec, ttl)` is the prior shape and `EnqueueAudit(ctx, evt, ttl)` (US-022) follows it. Cassandra appends `USING TTL <sec>` when ttl > 0 (`max(int(ttl/time.Second), 1)` to avoid `USING TTL 0`); memory backend stores `expiresAt = now+ttl` per row and prunes lazily on each List call (no background sweeper). Per-binary env parsing lives in `cmd/<binary>/main.go`, not in the internal package — internal stays env-free. Convention: helper named `ParseXRetention(s)` in the internal pkg (accepts `''`, `<N>d`, or any `time.ParseDuration` value), `XRetentionFromEnv()` in main reads the env var and falls back on error with a WARN log.
- Audit middleware sits INSIDE the access-log middleware and INSIDE `mw.Wrap` so it (1) sees the inner handler's status via its own writer wrapper, and (2) still emits a row when auth denies the request. Wrap ordering in `cmd/strata-gateway/main.go`: `mw.Wrap(s3api.NewAccessLogMiddleware(metaStore, s3api.NewAuditMiddleware(metaStore, ttl, apiHandler)), s3api.WriteAuthDenied)`. Each wrapper uses its own `responseWriter` shim that captures status + remembers `wroteHeader`; do NOT try to share a single shim across middlewares, the indirection is cheap and the layered ownership is clearer. State-changing detection is method-based (skip GET/HEAD/OPTIONS/CONNECT/TRACE) — works because IAM admin endpoints arrive as POST. For IAM-scoped writes the row uses `BucketID=uuid.Nil`, `Bucket="-"`, `Resource="iam:<Action>"`; bucket-scoped writes resolve `bucketID` via `meta.GetBucket` (best-effort — falls back to uuid.Nil if the bucket doesn't exist yet, e.g. PUT /bkt itself before the row is committed). Action label is CamelCase AWS verb derived from method+scope+sub-resource (CreateBucket / PutObject / PutBucketLifecycle / DeleteBucketCors / etc.); reuse `deriveAuditAction` shape if you add admin verbs.
- Filtered-list meta methods that take a `Filter` struct + return `(rows, nextToken, error)` are the canonical shape for paginated admin queries (US-023 `ListAuditFiltered(meta.AuditFilter)`, future US-046 audit-export, future inventory). Filter struct fields: optional `Start`/`End time.Time`, optional `Principal string`, paired `BucketID uuid.UUID` + `BucketScoped bool` so uuid.Nil stays a valid scope (IAM rows use it) and a `Limit int` + `Continuation string`. Continuation token is the EventID of the last row on the previous page; both backends accept it as opaque. Memory backend sorts the full set in-memory then walks past `Continuation` until found. Cassandra walks day-partitions (bucket-scoped: iterate the days; not-scoped: `SELECT DISTINCT bucket_id, day FROM audit_log` to enumerate partitions, then per-partition scan, then in-memory sort). For a low-volume admin endpoint the DISTINCT scan is acceptable; do NOT add a denormalised mirror table unless the surface goes high-volume. Admin handler validates filters (RFC3339 `time.Parse` for ISO-8601, positive `limit`), resolves bucket name → ID via `GetBucket` (yields NoSuchBucket), passes Filter through, and JSON-encodes `{records, next_continuation_token}`.
- Admin endpoints reject presigned URLs even when the principal is `[iam root]`. Concrete check at handler entry: `if r.URL.Query().Get("X-Amz-Signature") != "" { 403 }`. Reason: presigned URLs can be replayed from logs/links and are not appropriate for security-audit/admin surface. Header-signed SigV4 is the only auth path. Reuse this guard for any new `/?<admin>` route (US-034 admin CLI etc.).
- AWS-parity HEAD/GET response headers can be **gated behind a request opt-in header**. The canonical pattern (US-025): `x-amz-checksum-mode: ENABLED` (case-insensitive) is the gate for `x-amz-checksum-*` + `x-amz-checksum-type` echo on object reads. Helper `checksumModeEnabled(headerVal) bool` lives in `internal/s3api/checksum.go`. When adding a new "client-asked-for-it" header (future: `x-amz-trace-id` echo, expanded metadata-only views), define a tiny `<thing>ModeEnabled` predicate next to the writer rather than inlining the check — it documents the AWS-parity requirement and gives one place to flip semantics. Tests must explicitly set the gate header where they expect echo; existing tests written before the gate landed will silently regress to checking empty headers if not updated.
- `o.PartSizes` (Cassandra `objects.part_sizes`) is the canonical post-Complete signal for "is this a multipart object" — `len(o.PartSizes) > 0` means yes, period. Don't infer from the ETag suffix (`<hex>-<N>`); the ETag-suffix shortcut existed pre-US-024 and was removed in US-026 (`buildObjectPartsAttrs`). Anything new that needs per-part info post-Complete (ObjectParts subtree, future ListParts-after-complete endpoints, audit/inventory exporters) reads `o.PartSizes` for sizes + `o.Manifest.PartChecksums` (`[]map[string]string`) for per-part checksums via `partChecksumsAt(o, i)`. Both backends populate these in lockstep inside `CompleteMultipartUpload`.
- Versioning "null" literal handling: `meta.NullVersionID` (zero UUID) is the stored sentinel for rows in Disabled / Suspended-null buckets; `meta.NullVersionLiteral = "null"` is the wire form. `meta.ResolveVersionID(v)` translates wire→stored at entry of every `meta.Store` method taking a versionID (PutObject, GetObject, DeleteObject, findLatest). New methods MUST call it. Memory + cassandra also force `o.VersionID = NullVersionID` + `o.IsNull = true` inside PutObject when `versioned=false` — callers don't set the sentinel themselves. Cassandra column `objects.is_null` (additive ALTER); the zero UUID stores cleanly into the `timeuuid` clustering column despite version bits being 0 (gocql doesn't validate). Don't drop the `DELETE FROM objects WHERE bucket_id=? AND shard=? AND key=?` that runs before INSERT in `versioned=false` PutObject — it's what makes sentinel reuse safe across overwrites.
- Wire-side null version translation: every place s3api emits a VersionID into XML / `x-amz-version-id` / `NextVersionIdMarker` MUST run it through `wireVersionID(o)` so IsNull rows surface as the literal "null" rather than the zero-UUID sentinel. Helper lives in `internal/s3api/versioning.go`. Cassandra's `versionCursor.advance()` scans `is_null` so cursor rows carry the flag end-to-end (memory backend already populates it on PutObject). When adding any new ListObjectVersions-shaped surface (e.g. s3select scan, audit-style version inspector), keep both ends of this contract aligned: cursor reads is_null + wireVersionID() at render time. NextVersionIdMarker has no direct flag, so the s3api uses `nextWireVersionID(res)` which compares the raw NextVersionID against `meta.NullVersionID`.
- Suspended-mode null replacement: PutObject takes a 3rd path when `versioned=true && o.IsNull=true` — the meta backend forces VersionID=NullVersionID, drops just the prior null-versioned row (cassandra LWT `IF EXISTS`; memory in-place filter), and inserts the new null row above any TimeUUID ancestors. Caller (s3api putObject/copyObject) sets `obj.IsNull=true` when `b.Versioning == VersioningSuspended` BEFORE PutObject — meta does not look at the bucket. For the unversioned DELETE path, `meta.Store.DeleteObjectNullReplacement(ctx, bucketID, key)` is the dedicated Suspended-mode entrypoint: same atomic-replace semantics, returns a fresh null delete marker (IsNull=true, IsDeleteMarker=true). s3api enqueues GC chunks for the replaced null row at the handler layer (best-effort GetObject before the call) — meta does not own chunk cleanup. Wire shape: `x-amz-version-id: null` via `wireVersionID(obj)` on Suspended PUT/DELETE responses (don't emit raw zero-UUID).
- Website-endpoint behaviour (RedirectAllRequestsTo, future RoutingRules, IndexDocument/ErrorDocument fallback) bypasses S3 SigV4 auth and ACL/policy gates — `serveWebsiteRoot` and `tryWebsiteRedirectAll` run BEFORE `requireObjectAccess`/`requireACL`. Matches AWS's static-site model where the website endpoint is anonymous. Subresource GETs (`?acl`, `?tagging`, `?attributes`, `?retention`, `?legal-hold`, `?uploadId`) are excluded because they short-circuit out of `handleObject` before the method switch — management ops stay auth-gated. When adding new website-side handlers, keep both invariants: no auth, and the redirect-or-serve fork lives only in the plain-GET method-switch arm.
- Tests asserting on 3xx responses must disable redirect-following. `internal/s3api/website_test.go` defines `noRedirectClient()` returning an `http.Client` whose `CheckRedirect` returns `http.ErrUseLastResponse`, plus `(*testHarness).doNoRedirect(method, path)` as the harness wrapper. - **Learnings for future iterations:**
  - HKDF input choice: `IKM = oid || uint64BE(chunkIndex)` and `info = "strata-sse-chunk-iv"` — salt left nil since neither side has an out-of-band entropy source. Different chunkIndex / oid produce different IVs (verified by tests). If US-049 ever needs IV diversity across rewrites, the DEK already differs per object so collisions are not a worry.
  - Chunk ciphertext is **not** prefixed with a nonce — the receiver derives the IV deterministically from oid+chunkIndex. Saves 12 bytes per chunk on disk. The same shortcut is **not** safe for WrapDEK (random nonce, must be persisted), so the layouts differ on purpose.
  - `gcm.Seal(nil, …)` for chunks vs `gcm.Seal(out, …)` with pre-built `out=nonce||cap` for the wrapped DEK. Standard idiom — see `WrapDEK` for the prefix pattern.
  - The package depends on `golang.org/x/crypto/hkdf`. That module was already in `go.sum` as indirect; `go build ./...` accepts the upgrade-to-direct without complaint, so we did NOT run `go mod tidy` (would have churned unrelated transitive bumps — koanf, grpc-gateway, protobuf). Pattern: when adding a direct import that already exists indirect, just import and build. Reserve `tidy` for when a new module needs adding to `go.sum`.
  - `sse.EncryptChunk` is deterministic in (dek, oid, chunkIndex) — same inputs yield same ciphertext byte-for-byte. This is fine for SSE-S3 (every object has a unique DEK so no cross-object reuse), and lets the GC + lifecycle stories assume bit-stable rewrites. Test `TestEncryptChunkSameInputsProduceSameCiphertext` pins this.
  - This story does NOT wire anything into PutObject/GetObject. US-004 is the wiring point — it will need to (a) call `WrapDEK` and persist `objects.sse_key` + `sse_key_id`, and (b) walk chunk indexes and call `EncryptChunk` per chunk. The OID we feed into `DeriveChunkIV` must be deterministic per (object, version) — likely use `bucket/key/versionID` as the canonical string. Decide that in US-004; the package is agnostic.

Re-use this shape for US-039 inventory walker + US-046 audit export — read meta first, then drive `data.Backend` for body bytes, never re-derive.
  - **Pattern (per-rule destination encoding):** US-012 piggybacks the AWS-spec `AccessControlTranslation.Owner` field as the peer URL (host:port), so each replication rule encodes its own destination cluster and the worker doesn't need a sidecar config table to map rule_id → peer. AWS uses Owner for cross-account-arn-style translation; we repurpose it because Strata's IAM model doesn't have AWS account translation. Document if a future story (US-040 access points or cross-region IAM) needs the original Owner semantics back; for now the field is overloaded.
  - **Pattern (metrics injection without dep bloat):** `MetricsObserver` interface lets `internal/replication` stay free of `prometheus/client_golang`; the cmd binary provides a tiny `promMetrics{}` shim that wraps the package globals. Same shape was used informally in US-009/010; here it's a named interface because the worker has multiple counter sites. Re-use for US-014 access-log + US-046 audit-export when they need per-rule histograms.
  - **Gotcha (closing body in dispatcher):** `Source.Body` MUST be closed on every Send return path including early-return on missing endpoint/bucket — the test `TestHTTPDispatcherMissingEndpointFails` pins this. Future dispatchers (S3-SDK PutObject for true cross-region) need the same `defer src.Body.Close()` discipline; resource leaks here would be invisible until producer count grows.
  - **Gotcha (lock name uniqueness across workers):** every worker on the same Cassandra cluster shares the `worker_locks` table; pick a unique `Name` per worker (gc/lifecycle/notify/rewrap/replicator) so an outage in one doesn't block the others. New workers (US-014, US-046) must continue this pattern. The convention is "worker binary basename without `strata-` prefix".
  - **Gotcha (replication-status header on GET/HEAD):** the PUT response sends `PENDING` from the live emit; GET/HEAD now read the persisted `o.ReplicationStatus`. Don't conflate the two paths — PUT response is immediate-and-stateless, GET/HEAD is the durable contract. Tests asserting on PUT-time PENDING (`TestPutObjectReplicationStatus*`) must not GET-then-assert; the worker hasn't run.
  - **Gotcha (SSE + replication interaction):** the worker streams ciphertext when the source uses SSE; the peer can only round-trip it if both clusters share the master key list (env STRATA_SSE_MASTER_KEYS or vault). Document in operator notes; the worker neither decrypts nor re-encrypts. If a future story needs decrypt-then-re-encrypt across clusters, plumb the master.Provider into `replication.Worker.loadSource` and decrypt before handoff — but that breaks SSE-S3's "data never leaves the gateway plaintext" claim, so default to the verbatim behaviour.
