- **Per-object metadata pattern (US-001)**: tags + grants piggyback on objectRow (the JSON-encoded `objects` row payload) instead of separate `object_meta:...:<kind>` keys. Reason: parity with Cassandra (UPDATE objects SET tags=? / grants=?) and Memory (mutates the in-struct field). One row Get, one Set per RMW. Helpers `mutateObjectRow` + `snapshotObjectRow` in `internal/meta/tikv/object_meta.go` decode `objectRow` directly (skipping the meta.Object roundtrip) so callers can touch fields that meta.Object does not surface (objectRow.Grants is a `[]byte` blob; meta.Object has no Grants field). US-002 retention/legal-hold/restore-status and US-005 SSE-rewrap + replication-status follow the same pattern — every "per-object scalar field" stub maps to `mutateObjectRow(... func(*objectRow) { row.<Field> = <value> })`.
- **keycodec.go reach criteria**: PackKey is the helper for new key shapes that don't fit one of the entity constructors in keys.go (BucketKey, ObjectKey, AccessPointKey, …). US-001..US-005 piggyback paths don't need PackKey — they reuse existing object keys. Use PackKey only when adding a NEW prefix (e.g. if a future story needs a sse_rewrap_progress_history table the entity constructor would belong in keys.go, not as ad-hoc PackKey calls).
- **Memory vs Cassandra divergence — empty grants list**: SetObjectGrants(nil) on Cassandra → blob NULL → Get returns ErrNoSuchGrants. Memory → stores nil slice in map (key present) → Get returns `(nil, nil)`. Contract test ObjectGrantsRoundTrip does NOT assert behavior after Set(nil) — divergence is pre-existing and out of scope for US-001. PRD says "empty != absent"; Memory matches, Cassandra doesn't. Future cycle can align Cassandra. **Same divergence on bucket-grants (US-003)** — TiKV matches Cassandra (Set(empty)≡clear via nil-bytes); contract case BucketGrantsRoundTrip's (e) leg accepts either path so memory keeps passing.
- **Optimistic vs pessimistic txn rule of thumb**: pessimistic (`Begin(ctx, true)`) ONLY when the same txn does a Read + a Set on the same key (RMW per CLAUDE.md "Plain Put on a key with prior LWT history" gotcha). Blind single-row Set/Delete on a config-doc key (BucketBlobKey, BucketGrantsKey, …) uses optimistic (`Begin(ctx, false)`) — same shape as the bucket-blob trio in blobs.go. PRD's carbon-copied "pessimistic txn" boilerplate is over-prescription for blind writes; reserve pessimistic for stories with a Read step (US-001/US-002 mutateObjectRow, US-004 access-point dual-index atomic Create, US-005 SSE-rewrap + replication-status RMW).
