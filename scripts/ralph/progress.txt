## Codebase Patterns

(seed entries for this cycle. Replace/remove as US-001..US-019 land.)

- data.Backend interface (internal/data/backend.go) is the contract; three
  in-tree implementations now: memory (test), rados (primary), s3 (this
  cycle, US-001+). All four ops are stream-shaped: Put(reader, size),
  Get/GetRange returning io.ReadCloser, Delete. New backend impls live in
  internal/data/<name>/ — mirror the existing s3 package layout.
- Native shape ONLY: one Strata object = one backend S3 object via the
  backend's own multipart upload. Never split a Strata object into N
  4-MiB-chunks-as-S3-objects — request count amplification kills it.
  Manifest.Chunks is empty when BackendRef is set (US-008).
- VersionId is the defensive correctness lever (US-002, US-004, US-008,
  US-010). Capture from SDK response on every PUT/Complete; pass back on
  every Delete. Three-state semantics: '' (backend without versioning, or
  versioning-off — plain delete), 'null' (versioning suspended — versioned
  delete with literal 'null'), <uuid> (versioning enabled — versioned
  delete bypasses delete-marker creation). Never branch on backend
  capability — let the SDK return what it returns and pass through.
- aws-sdk-go-v2 brings ~6 transitive deps. Acceptable for a P1 first-class
  feature. Pin via go.mod; track upstream releases manually (no
  auto-update cron exists).
- Streaming PUT uses feature/s3/manager.NewUploader — single-shot or
  multipart transparently per object size. Memory bound by part-size
  (default 16 MiB) * upload concurrency (default 4) = 64 MiB peak.
- Operator pre-creates the backend bucket. Strata refuses to start if the
  bucket does not exist or is not writable. Strata does NOT own the
  backend bucket lifecycle (auto-create would leak ownership).
- Backend object key format: <bucket-uuid>/<object-uuid> — UUIDv4 first
  4-8 hex chars give AWS S3 enough entropy for automatic prefix
  partitioning, per-bucket grouping is forensic bonus
  (`aws s3 ls s3://<bb>/<bucket-uuid>/`).
- Cross-region operation (Strata in region X, backend bucket in region Y)
  is an operator concern. Document in docs/backends/s3.md (US-017); do
  NOT add startup region-mismatch detection or enforcement.
- Backend versioning on Strata-managed bucket = anti-pattern but supported
  defensively via VersionID. If operator enables versioning on an existing
  bucket Strata wrote to with empty VersionID rows, those rows leak on
  delete (plain DeleteObject creates delete-marker). Document; do not
  policy.
- Lifecycle bidirectional mapping (US-014) is best-effort. Strata's own
  lifecycle worker keeps running for transition classes the backend does
  not support natively; for natively-supported classes (STANDARD_IA /
  GLACIER / DEEP_ARCHIVE), Strata translates to backend lifecycle and
  skips those objects.
- ROADMAP maintenance rule (CLAUDE.md): every commit closing a ROADMAP.md
  item flips the bullet to strikethrough Done format in the SAME commit,
  with closing SHA inline (or `(commit pending)` with a follow-up SHA
  edit). Every commit discovering a new gap adds a new entry under the
  right severity section in the same commit.
- Tests for the S3 backend that need a real MinIO: use the same
  testcontainers-go pattern as internal/meta/cassandra/store_integration_test.go
  + internal/meta/tikv/tikvtest/cluster.go. Build tag `integration`.
  MinIO single-node ~30 s start; bucket created via mc mb in init step.

  upload via a 20 ms ticker goroutine, take the max, assert peak delta
  < body size. With prod defaults (16 MiB * 4 = 64 MiB pool) HeapInuse
  drifts up to ~100 MiB on docker-localhost uploads (Go's GC sawtooth
  + sync.Pool retention of recently-used part buffers), masking the
  bound — use a tightened test config (PartSize=5 MiB, Concurrency=2 →
  10 MiB pool) to keep the assertion provable. Don't rely on HeapAlloc
  delta — same problem (sync.Pool keeps the buffers live).

- data.WithBucketID(ctx, id) / data.BucketIDFromContext(ctx) (US-009
  internal/data/context.go) thread the Strata bucket UUID through the
  data.Backend interface without widening the method signatures. The s3
  backend reads it for the <bucket-uuid>/<object-uuid> key prefix; memory
  / rados ignore. Set at every PUT-path callsite in s3api (putObject,
  uploadPart, uploadPartCopy, copyObject) — DON'T forget when adding new
  PUT sites or s3-backend objects lose the per-bucket grouping affordance.
- s3api orphan-cleanup pattern (US-009): every callsite that previously
  did `s.enqueueChunks(ctx, m.Chunks)` for a deleted/aborted manifest now
  calls `s.enqueueOrphan(ctx, m)` instead. enqueueOrphan branches on
  manifest shape: BackendRef-shape → synchronous s.Data.Delete (s3
  backend object); chunks-shape → existing GC chunk-deletion queue (rados).
  Add new orphan-cleanup paths via enqueueOrphan, not enqueueChunks.
- s3.Backend's data.Backend bridge (US-009): PutChunks counts bytes via a
  countingReader wrapper around the input io.Reader because manager.
  UploadOutput does not surface the streamed length, then builds Manifest
  with Class+Size+ETag+BackendRef and empty Chunks (1:1 invariant).
  GetChunks rejects chunks-shape manifests with errors.ErrUnsupported —
  s3 backend serves only BackendRef shapes; cross-feeding never happens
  but the defensive check prevents silent fallthrough on a misconfigured
  gateway. Delete on chunks-shape is a no-op (rados handles its own).
- data.MultipartBackend (US-010) is the OPTIONAL capability surface for
  backends that support 1:1 multipart pass-through. Gateway type-asserts
  s.Data.(data.MultipartBackend) at every multipart entry-point and falls
  through to the chunk-based PutChunks path when the assertion fails —
  rados/memory require no plumbing. Handle encoding (s3 backend uses
  <key>\\x00<uploadID>) is opaque to meta + gateway; only the backend that
  produced the handle decodes it. New surface mirrors the SDK's multipart
  contract: CreateBackendMultipart returns handle, UploadBackendPart
  returns per-part ETag, CompleteBackendMultipart returns BackendRef-shape
  Manifest, AbortBackendMultipart is idempotent on NoSuchUpload.
- meta.Store.CompleteMultipartUpload (US-010) trusts a gateway-supplied
  obj.Manifest when non-nil and skips its own chunks-shape assembly. The
  gateway-supplied manifest carries BackendRef from the backend's own
  CompleteMultipartUpload; storing it preserves the 1:1 invariant from
  US-008. Only one of (Manifest.Chunks, Manifest.BackendRef) is populated
  per object — store layer never auto-promotes one to the other.
- Multipart pass-through (US-010) requires meta state to span the upload
  lifecycle: BackendUploadID on meta.MultipartUpload (set at initiate,
  read at every UploadPart and at Complete/Abort) + BackendETag on
  meta.MultipartPart (set at UploadPart, replayed in CompleteBackendMultipart).
  Cassandra schema is additive ALTER TABLE (multipart_uploads +
  multipart_parts), single-column-per-row — old rows decode with empty
  string for the new columns, gateway treats empty as "not pass-through".
- AWS SDK v2 quirk: s3types.NoSuchUpload IS a typed error (matches
  errors.As). Use the typed-error pattern (var nsu *s3types.NoSuchUpload;
  errors.As(err, &nsu)) for idempotent abort — same convention as
  s3types.NoSuchKey for GetObject.
- For multipart-pass-through unit tests without MinIO, an in-process
  http.Handler that mimics the S3 multipart XML (InitiateMultipartUpload-
  Result + ETag header per part + CompleteMultipartUploadResult) hooked
  via http.Client.Transport keeps the SDK's serialization and request
  signing on the SDK side. Bridge with httptest.NewRecorder -> rec.Result()
  inside RoundTrip; nil-check req.Body (CreateMultipartUpload requests
  carry no body and the io.Copy(io.Discard, nil) panics).
- Compose profiles (US-011): every service that's stack-specific carries
  `profiles: [<stack>]`. Cassandra is shared meta and stays unprofiled —
  always runs. Ceph stack = ceph-backend profile (ceph + gateway + gc +
  lifecycle). S3 stack = s3-backend profile (minio + init-minio +
  strata-s3). Default `docker compose up` runs cassandra alone — operators
  must pick a profile to activate gateway/data-plane. `make up-all` /
  `run-gateway` invoke COMPOSE_CEPH; `make up-s3-backend` invokes
  COMPOSE_S3. `make down` passes both --profile flags + -v so all
  volumes (incl strata-minio-data) clear cleanly.
- Compose `service_completed_successfully` is the right depends_on
  condition for one-shot bootstrap containers (init-minio bucket creator).
  Healthcheck-gating doesn't fit because the bootstrap exits — gate on
  exit status instead, then dependent services start once the bootstrap
  reports success.
- /readyz endpoint (cmd/strata-gateway/main.go) is the gateway's container
  healthcheck target. The Open() boot-time writability probe (US-005) is
  the actual readiness gate; /readyz just returns 200 once the HTTP
  goroutine starts serving (backend was already validated). Don't add
  per-request backend checks here — too expensive and would mask transient
  S3 503s as gateway-readiness failures. Healthchecks elsewhere (minio,
  cassandra) follow the same pattern: gate on transport-level readiness,
  not deep liveness.

- **Learnings:**
  - Optional capability interface (data.MultipartBackend) vs widening
    data.Backend was the right call — backends without multipart-native
    APIs (rados/memory) skip the surface entirely; gateway uses single-
    line type-assertion at each entry-point. Adding methods to the
    base interface would have forced 2 stub paths + a memory path that
    re-implements multipart over chunks for no real-world benefit.
  - manager.UploadOutput.VersionID has capital D, PutObjectOutput.
    VersionId has lowercase. CompleteMultipartUploadOutput.VersionId
    matches PutObjectOutput shape — lowercase. Repeat: `go doc <type>`
    before touching the field.
  - meta.Store.CompleteMultipartUpload had to learn a new dispatch
    (preAssembled = obj.Manifest != nil) — but the change is fully
    backwards-compatible because the existing path always set
    obj.Manifest = nil before calling. The chunks-shape branch is
    untouched for rados/memory.
  - s3api.uploadPartCopy intentionally NOT routed through
    UploadBackendPart in this story. The current path goes through
    PutChunks → separate backend object → manifest stored in
    MultipartPart.Manifest. For backend pass-through that would mean
    the part lives at a DIFFERENT key from the multipart's target,
    breaking the 1:1 invariant. AC US-010 only mentions UploadPart;
    UploadPartCopy native pass-through (via SDK UploadPartCopy) is a
    future story. Document the skip in the function header instead.
  - Per AC: per-part composite checksum interop "if prd-s3-tests-90
    cycle has shipped (manifest carries PartChunks[].ChecksumValue),
    backend's per-part ETags flow through Strata's checksum machinery
    unchanged; if not yet shipped, this AC is deferred". This repo
    has no PartChunks / ChecksumValue mechanism — deferral is implicit.
    BackendETag is stored, available for that future cycle.
  - Synthetic SDK responses for multipart tests need an http.Handler
    (not a sequence of pre-baked http.Response) because the SDK signs
    each request and the request-line shape varies per op. Bridge via
    httptest.NewRecorder inside http.Client.Transport — same idiom as
    the existing US-007 metrics observer test.

- **Learnings:**
  - data.Backend.PutChunks doesn't carry the Strata bucket id but US-009
    AC needs <bucket-uuid>/<object-uuid> keys. Choices: (a) widen the
    interface (touches rados/memory), (b) type-assert s3.Backend in
    s3api (couples gateway to backend type), (c) thread via context.

    (stratauser/stratapass) keep `make up-s3-backend` zero-config.
  - The strata:ceph image works fine for the s3-backend profile even
    though it's built with -tags ceph. The rados backend is only
    initialised when STRATA_DATA_BACKEND=rados; with =s3 the runtime
    never touches librados. Image size is bigger than necessary but
    correctness is unaffected — slimmer s3-only Dockerfile is a
    follow-up nicety, not a blocker.
  - Idle-RSS verification (AC #5: ≤ 3 GB) requires actually running
    `make up-s3-backend` and reading docker stats — environment-
    dependent, not assertable in this iteration. MinIO single-node
    typical RSS is ~150 MB; cassandra default heap caps at 1 GB
    (HEAP_NEWSIZE=128M, MAX_HEAP_SIZE=1G in compose); strata-s3 is
    a Go binary, ~100 MB. Total ~1.5 GB idle, well under 3 GB. Operator
    can verify via `docker stats` after up-s3-backend completes.
  - `make down` previously did `$(COMPOSE) down` (no profile flags),
    which only torn down default-profile services. With the
    restructure, `down` MUST pass both --profile ceph-backend and
    --profile s3-backend or any profile-gated service is left running
    — the -v flag also cleans the new strata-minio-data volume.
