### CRC algorithms (US-003)
- AWS composite CRC formula details: https://docs.aws.amazon.com/AmazonS3/latest/userguide/checking-object-integrity.html
- AWS composite for CRC32/CRC32C/CRC64NVME = `crc(concat(raw_part_crcs))` — concat raw checksum bytes per part, then compute the CRC of the concat, encode base64. (NOT the SHA-style "concat then SHA-256".)
- Helper packages on main:
  - `hash/crc32` for CRC32 + Castagnoli (CRC32C — `crc32.MakeTable(crc32.Castagnoli)`)
  - `github.com/danchupin/strata/internal/checksum` (verify exists; if not, add) for CRC64NVME (`hash/crc64` + iso/ecma table OR vendored Go 1.21+ NVME polynomial)
- Algorithm dispatch: existing `manifest.PartChunks[N].ChecksumAlgorithm` field (US-001 of prior cycle) carries the per-part algo string.

## 2026-05-06 - US-001 Multipart copy edges
- Implemented:
  - `internal/s3api/copy_object.go::parseCopySource` now splits the header on `?` BEFORE `url.PathUnescape`. boto3 percent-encodes literal `?` in keys as `%3F`, so the legacy "decode-then-split" path corrupted `?versionId`-shaped keys. Bucket and key segments are now decoded independently after the split.
  - `internal/s3api/multipart.go::parseCopySourceRangeStrict` replaces the old `(start,end,bool)` parser. It returns a typed `*APIError`, distinguishing 400 InvalidArgument (syntax: missing `bytes=`, missing `-`, blank lo/hi, non-numeric, multi-range) from 416 InvalidRange (syntactically valid but `start >= size`, `end < start`, or `end >= size`). `uploadPartCopy` writes the returned error directly.
- Files changed:
  - `internal/s3api/copy_object.go`
  - `internal/s3api/multipart.go`
  - `internal/s3api/multipart_copy_test.go` — added `TestUploadPartCopyImproperRangeReturnsInvalidArgument`, `TestUploadPartCopyOutOfBoundsRangeReturnsInvalidRange`, `TestUploadPartCopySpecialSourceKeyNames`.
- Learnings for future iterations:
  - **boto3 wire shape for x-amz-copy-source**: `?` in source keys is `%3F`. Splitting on raw (encoded) form preserves the distinction between literal `?` and the `?versionId=…` query separator. PathUnescape order matters — decode AFTER the split, not before.
  - **AWS error-code split for copy-range**: the two s3-tests (`improper_range` vs `invalid_range`) test exactly this distinction. Bundling both as `InvalidRange` is what the prior cycle did and is exactly why `improper_range` failed. Always model the parser as a small state machine returning a typed error.
  - **Per-part ETag in `copyPartResult`** was already returning `"<32hex>"` (manifest.ETag of the per-part bytes, double-quoted). Don't refactor it — boto3 round-trips fine.
  - **s3-tests cache lives at `/tmp/strata-s3tests.MO5E/s3-tests/`** in this dev env; `s3tests/functional/test_s3.py` is the canonical source for the failing tests' wire-level assertions when the live suite isn't running.
