- **Learnings for future iterations:**
  - **Go vuln DB does NOT populate `osv.severity` field** — every record has `"severity": null`. The gate's "HIGH/CRITICAL" classification is structural (callgraph reachability) not CVSS-based; jq script treats any callgraph hit terminating in a Strata module as HIGH by default. If a CVSS hierarchy is needed later, cross-reference `osv.aliases` (CVE IDs) against an external CVSS feed — keep this out of the CI hot path.
  - **`go get` + `go mod tidy` switching `go.mod`'s `go` directive** triggers `GOTOOLCHAIN=auto` to fetch the matching toolchain. Locally I had go1.25.1; the Go toolchain auto-downloaded go1.25.10 on first `GOTOOLCHAIN=go1.25.10+auto go ...` invocation. CI's `actions/setup-go@v5` with `go-version: '1.25'` resolves to latest 1.25.X (currently 1.25.10) so no manual toolchain pin is needed in the workflow.
  - **`go.work` IS git-tracked in this repo** (despite CLAUDE.md hinting otherwise) — pinned at 58aeb82. Bumping `go.mod`'s `go` directive past `go.work`'s directive errors `go.work lists go X.Y`; bump both in lockstep. `go.work` carries `toolchain go1.25.10` line; `go.mod` does not need it explicitly because the `go X.Y.Z` requirement already implies it.
  - **govulncheck callgraph trace shape**: `finding.trace[]` arrays — element 0 is the vuln site (`stdlib` / module package), last element is the calling Strata function. `trace.length == 1` means imported-but-not-called (no callgraph hit). `trace.length > 1` AND last frame's `module == github.com/danchupin/strata{,/cephimpl}` = real callgraph hit. The jq filter in `scripts/lib/severity.sh::severity_jq_callgraph_filter` enforces this with `--argjson strata_modules` for the module-name allowlist.
  - **govulncheck exit codes**: `0` = no callgraph hits, `3` = at least one callgraph hit, anything else = scan error (binary missing, build failure). The wrapper script accepts {0,3} and lets the jq filter make the PASS/FAIL decision — relying on the binary's exit alone would over-trigger (it doesn't know about our WARN-only allowlist).
  - **cephimpl module needs librados-dev** (cgo) to run govulncheck. CI installs via `sudo apt-get install -y librados-dev` before the cephimpl matrix axis runs. Local macOS dev hosts trip `librados.h not found` — the wrapper degrades to WARN+exit 0 in that case (matches `make vet`'s degradation pattern for the same reason). The CI matrix is the source of truth for cephimpl coverage.
  - **`scripts/lib/severity.sh` is shared infrastructure** — US-002 trivy, US-003 gosec, US-009 go-licenses are expected to source it for severity-classification helpers + ANSI color + Strata module allowlist. Adding helpers there is preferred over duplicating jq filters into each scanner wrapper.
  - **`actions/checkout@v4` + `actions/setup-go@v5` are first-party GitHub** — current ci.yml convention is `@vN` tag pin, not SHA. SHA-pinning at impl time per US-001 AC applies to **third-party** actions (`aquasecurity/trivy-action`, `securego/gosec/v2`, `github/codeql-action`, `slsa-framework/slsa-github-generator`, `anchore/sbom-action`, `sigstore/cosign-installer`, `softprops/action-gh-release`, `dependabot/fetch-metadata` — none used by US-001 itself). - **Learnings for future iterations:**
  - **`gh api repos/<owner>/<repo>/git/refs/tags/<tag>` returns the tag-ref object SHA, NOT the commit SHA when the tag is annotated.** Two-hop resolution required: `gh api .../git/refs/tags/<tag> → .object.sha` then `gh api .../git/tags/<that-sha> → .object.sha` yields the commit SHA. Annotated tags are the upstream convention for both `trivy-action` and `codeql-action`. Lightweight tags would resolve in one hop (`.object.type == "commit"` directly). Always check `.object.type` first — `tag` = annotated, `commit` = lightweight. SHA-pinning to the tag-ref SHA instead of the commit SHA technically works but is brittle: a tag-rewrite changes the tag-ref SHA, breaking the pin. Pin to the commit SHA underneath.
  - **`aquasecurity/trivy-action` v0.36.0** (latest stable as of 2026-05-28) supports `trivy-config: <path>` input which loads a `trivy.yaml` shape. Action's other knobs (`severity`, `exit-code`, `format`, `output`) layer on top — the config file is for cycle-wide policy (scanners, timeout), the action inputs are for per-pass gates. Two passes (CRITICAL + HIGH) is the standard split — single pass with `severity: 'CRITICAL,HIGH'` would conflate gate semantics.
  - **`github/codeql-action/upload-sarif` `category` input** distinguishes multiple SARIF uploads in the same job — GitHub's Code Scanning UI merges entries by category. Setting `category: 'trivy-critical'` keeps this gate distinct from future SARIF emissions (US-003 gosec will use `category: 'gosec'`).
  - **`if: always()`** on the SARIF upload + HIGH WARN-only pass + artifact upload steps ensures the workflow continues past the CRITICAL gate failure — operators see the full HIGH-class findings + Code Scanning entry even when the gate hard-fails. Without `if: always()`, a CRITICAL failure short-circuits the job and Code Scanning never receives the SARIF.
  - **`trivy-image` rebuilds the image even though `docker-build` already built it** — buildx cache layers are job-local on GitHub Actions runners. Cross-job image artifact passing via `docker save | docker load` would transfer multi-GB tarballs over the Actions artifact backend, dominating job wall-clock. Rebuilding under `docker/setup-buildx-action@v3` warm cache is faster + simpler. Same trick as `e2e` / `e2e-full`. The PR review-time wall-clock delta is ~3-5 min image build + ~30s trivy scan — fits the +30-60s budget per S4 from PRD review.
  - **`scripts/check-trivy.sh` does NOT mirror the `check-govulncheck.sh` matrix axis pattern** — trivy scans the published image (`strata:ceph`), not Go source modules. The cephimpl module shape is irrelevant at the image layer — the final image already linked cephimpl via the `-tags ceph` build inside the Dockerfile. One image, one scan, one gate. The matrix axis is only relevant for Go-source-level scanners (govulncheck, gosec, go-licenses) where the cephimpl module is a separate Go module per CLAUDE.md.
  - **trivy 0.36 image-scan `format: 'sarif'`** writes SARIF v2.1.0 with `runs[0].tool.driver.name = 'Trivy'` — codeql-action SARIF upload validator accepts this out-of-the-box. No conversion step needed.
  - **trivy CRITICAL+HIGH dual pass duplicates DB download** — each invocation pulls the vuln DB fresh unless TRIVY_CACHE_DIR is set + warm. Future optimization (not blocking US-002): pin `cache-dir: /tmp/trivy-cache` + actions/cache the directory across the two steps. Skipping for now — DB download is ~5-10s, dominated by image-scan wall-clock.
