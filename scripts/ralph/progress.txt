- **Learnings for future iterations:**
  - Schema-additive proto field on existing message: bump the next free field number in manifest.proto, regenerate with `protoc --go_out=. --go_opt=paths=source_relative manifest.proto` (note: paths=source_relative emits next to .proto, so the resulting .pb.go must be `mv`d into manifestpb/). Then mirror the field on the Go struct with `json:",omitempty"` and add it to BOTH manifestToProto and manifestFromProto in manifest_codec.go. Old rows decode with zero-value Cluster; no migration needed.
  - The S3 backend's bucketOnCluster pattern lets one storage class span multiple clusters with different bucket names (e.g. STANDARD->c1/bkt-c1, STANDARD-B->c2/bkt-c2). The rebalance.S3Mover.BucketBy resolver wraps this so the mover can find the destination bucket on the picked cluster without re-querying the s3.Backend's class map directly.
  - GC enqueue shape for S3 rebalance: chunk-shape ChunkRef where Cluster=cluster_id, Pool=bucket_name, OID=backend_key. The s3.Backend.Delete dispatches this shape via deleteFromCluster. Pre-existing rados.Backend code handles chunk-shape natively; the s3 backend handler was a small additive branch that preserves the existing no-op semantics for legacy entries (chunk.Cluster=="").
  - Build-tag layout for movers: build-tag-free helpers (`rebalance_movers_s3.go`) call into `internal/data/s3` (which doesn't need ceph) directly. Build-tag-conditional files (`rebalance_movers_ceph.go` / `rebalance_movers.go`) compose them into the per-binary mover slice. This matches the convention that the s3 backend is always available but rados is only present under `-tags ceph`.
  - Pre-emptive CAS check via buildUpdatedBackendManifest is the same hazard avoidance as the RadosMover's buildUpdatedManifest: a class-match LWT predicate doesn't verify the BackendRef.Key, so a concurrent client write could rewrite the key (within the same class) and the rebalance LWT would still apply, clobbering the live locator. Checking live.BackendRef.Key vs planned SrcRef.OID before SetObjectStorage routes the conflict to GC for the new copy instead.
  - The fakeS3Cluster pattern (in-memory map per (bucket, key), Endpoint/Region as struct fields driving same-endpoint detect) is the testing oracle for S3Mover. Mirrors fakeRadosCluster's shape so the test toolkit is uniform.
  - Mismatch between PRD verbiage and Go method names persists: PRD says "EnqueueGCEntry" but the meta.Store method is `EnqueueChunkDeletion`. Use the real name (US-004 caveat).
  - Racetest TestRunCleanRunNoInconsistencies remains the pre-existing flake noted in US-002/US-004; not chasing.
