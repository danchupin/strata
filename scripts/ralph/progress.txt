## 2026-05-04 â€” US-005 â€” OTel trace ring buffer + endpoint
- New `internal/otel/ringbuf` package â€” `RingBuffer` implements `sdktrace.SpanProcessor` (no-op `OnStart`/`Shutdown`/`ForceFlush`; `OnEnd` records every span). Bytes-budgeted LRU eviction (default 4 MiB; `STRATA_OTEL_RINGBUF_BYTES` overrides). Per-trace span cap 256 (`PerTraceSpanCap`); excess spans dropped with one WARN per trace via injected `*slog.Logger`. Two indices: `traces` keyed by `trace.TraceID` and `byReq` keyed by `request_id` span attribute. Public `GetByRequestID` / `GetByTraceID` return the wire-shape `Trace` doc.
- Wired into `otel.Init` via `STRATA_OTEL_RINGBUF` (default `on`). When OTLP endpoint is set we install BOTH the existing tail-sampler+OTLP pipeline AND the ring buffer in parallel so failing spans still export and the in-process retention always has data. - **Learnings for future iterations:**
  - Span processors in OTel SDK 1.x register as a list; the SDK fans `OnEnd` to each in order. Installing the tail-sampler-wrapped BSP and the ring buffer as separate processors means the ring buffer sees ALL spans regardless of sample ratio (which is the point â€” operators want the last 10 minutes retained even when ratio=0.01).
