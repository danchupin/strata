### CRC algorithms (US-003)
- AWS composite CRC formula details: https://docs.aws.amazon.com/AmazonS3/latest/userguide/checking-object-integrity.html
- AWS composite for CRC32/CRC32C/CRC64NVME = `crc(concat(raw_part_crcs))` — concat raw checksum bytes per part, then compute the CRC of the concat, encode base64. (NOT the SHA-style "concat then SHA-256".)
- Helper packages on main:
  - `hash/crc32` for CRC32 + Castagnoli (CRC32C — `crc32.MakeTable(crc32.Castagnoli)`)
  - `github.com/danchupin/strata/internal/checksum` (verify exists; if not, add) for CRC64NVME (`hash/crc64` + iso/ecma table OR vendored Go 1.21+ NVME polynomial)
- Algorithm dispatch: existing `manifest.PartChunks[N].ChecksumAlgorithm` field (US-001 of prior cycle) carries the per-part algo string.

## 2026-05-06 - US-001 Multipart copy edges
- Implemented:
  - `internal/s3api/copy_object.go::parseCopySource` now splits the header on `?` BEFORE `url.PathUnescape`. boto3 percent-encodes literal `?` in keys as `%3F`, so the legacy "decode-then-split" path corrupted `?versionId`-shaped keys. Bucket and key segments are now decoded independently after the split.
  - `internal/s3api/multipart.go::parseCopySourceRangeStrict` replaces the old `(start,end,bool)` parser. It returns a typed `*APIError`, distinguishing 400 InvalidArgument (syntax: missing `bytes=`, missing `-`, blank lo/hi, non-numeric, multi-range) from 416 InvalidRange (syntactically valid but `start >= size`, `end < start`, or `end >= size`). `uploadPartCopy` writes the returned error directly.
- Files changed:
  - `internal/s3api/copy_object.go`
  - `internal/s3api/multipart.go`
  - `internal/s3api/multipart_copy_test.go` — added `TestUploadPartCopyImproperRangeReturnsInvalidArgument`, `TestUploadPartCopyOutOfBoundsRangeReturnsInvalidRange`, `TestUploadPartCopySpecialSourceKeyNames`.
- Learnings for future iterations:
  - **boto3 wire shape for x-amz-copy-source**: `?` in source keys is `%3F`. Splitting on raw (encoded) form preserves the distinction between literal `?` and the `?versionId=…` query separator. PathUnescape order matters — decode AFTER the split, not before.
  - **AWS error-code split for copy-range**: the two s3-tests (`improper_range` vs `invalid_range`) test exactly this distinction. Bundling both as `InvalidRange` is what the prior cycle did and is exactly why `improper_range` failed. Always model the parser as a small state machine returning a typed error.
  - **Per-part ETag in `copyPartResult`** was already returning `"<32hex>"` (manifest.ETag of the per-part bytes, double-quoted). Don't refactor it — boto3 round-trips fine.
  - **s3-tests cache lives at `/tmp/strata-s3tests.MO5E/s3-tests/`** in this dev env; `s3tests/functional/test_s3.py` is the canonical source for the failing tests' wire-level assertions when the live suite isn't running.

## 2026-05-06 - US-002 ?partNumber=N GET ETag wire shape
- Implemented:
  - `internal/s3api/server.go::getObject` partNumber path: dropped the `partRange.ETag` override. The response ETag header now always echoes `o.ETag` (the whole-object multipart composite `"<32hex>-<count>"`), matching what boto3 sees in the CompleteMultipartUpload response. Prior code overrode with the per-part raw ETag, which diverged from AWS.
  - Out-of-range partNumber (`n < 1`, `n > partsTotal`, non-numeric, or partNumber on a non-multipart object with `n != 1`) now returns `400 InvalidPart` instead of `416 InvalidRange`. AWS-parity confirmed against s3-tests test_multipart_get_part / _single_get_part.
  - Non-multipart object `?partNumber=1` now returns the whole object (`200 OK`, single-PUT ETag) instead of `416 InvalidRange`. Mirrors s3-tests test_non_multipart_get_part.
- Files changed:
  - `internal/s3api/server.go`
  - `internal/s3api/multipart_partnumber_test.go` — replaced `TestGetByPartNumberNonMultipartReturns416` with `TestGetByPartNumberNonMultipart` (asserts 200 + ETag for n=1, 400 InvalidPart for n=2). Flipped `TestGetByPartNumberOutOfRange` to assert 400. Added `TestMultipartGetByPartNumberEchoesWholeObjectETag` with single-part + multi-part subtests asserting GET/HEAD partNumber=N ETag equals the CompleteMultipartUpload XML `<ETag>` value.
- Learnings for future iterations:
  - **AWS `?partNumber=N` ETag is the WHOLE-OBJECT multipart ETag, not the per-part ETag.** boto3's `response['ETag']` on a `client.get_object(...PartNumber=N)` call must equal `client.complete_multipart_upload(...)['ETag']`. The PRD note in this story said the opposite — trust the s3-test source over the PRD when they conflict. Verified by reading test_multipart_get_part / _sse_c_get_part / _single_get_part directly.
  - **AWS error code for partNumber out-of-range is `400 InvalidPart`, NOT `416 InvalidRange`.** Range-shape errors (e.g. `Range: bytes=99999999-` inside a valid partNumber) stay 416 — that's the inner range parser, not the part selector. Two error codes for two different failure modes.
  - **Non-multipart object `?partNumber=1` returns the whole object** (200 OK, normal ETag). Only `?partNumber=2+` on non-multipart yields InvalidPart. Implementation: when `partsTotal == 0 && n == 1`, leave `partNumber=0` so the rest of the GET path treats the request as plain (no Content-Range, no 206 status).
  - **XML test fixture parsing**: `<ETag>"abc"</ETag>` in the response body becomes `<ETag>&#34;abc&#34;</ETag>` in raw bytes (Go's `encoding/xml` escapes the inner quotes). Decode via `xml.Unmarshal` not raw `strings.Index` — the round-trip is `"<32hex>-<count>"` (with literal quote bytes) once decoded, matching what boto3 receives over the wire.

## 2026-05-06 - US-003 Multipart composite checksum CRC32 / CRC32C / CRC64NVME
- Implemented:
  - `internal/s3api/checksum.go`: extended `composeMultipartChecksums` to take a `checksumType` arg. SHA1/SHA256 stay on the COMPOSITE shape (`BASE64(HASH(concat(raw_i)))-N`). CRC32 / CRC32C / CRC64NVME branch on `checksumType="FULL_OBJECT"` and combine per-part CRCs into the whole-stream CRC via standard zlib-style `crc32_combine` / `crc64_combine` math (no `-N` suffix). When `checksumType` is empty or `COMPOSITE`, CRC family keeps the legacy concat-then-hash shape (still passes today's tests + boto3 default).
  - New helpers in the same file: `combineCRCParts(algo, raws, parts)`, `crc32Combine(poly, crc1, crc2, len2)`, `crc64Combine(poly, crc1, crc2, len2)`, plus their `gf2MatrixTimes/Square` workhorses. Polynomials use the reversed/right-shift convention (`0xedb88320` IEEE, `0x82f63b78` Castagnoli, `crc64NVMEPoly` for NVME) so they line up with `hash/crc32` and our `crc64NVMEHash` Sum values without an extra inversion.
  - `checksumPart` grew a `Size int64` field — required to feed `len2` into the combine math. Multipart Complete site already has this on `meta.MultipartPart`, so the wire is `requested = append(requested, &checksumPart{Checksums: sp.Checksums, Size: sp.Size})`.
  - `multipart.go::completeMultipartUpload` now passes `mu.ChecksumType` into `composeMultipartChecksums`. The XML response and HEAD/GET (with `ChecksumMode=ENABLED`) echo the FULL_OBJECT shape unchanged via the existing `o.Checksums` + `o.ChecksumType` plumbing.
  - Added `internal/s3api.CombineCRCPartsForTest` (test-only export) to drive the FULL_OBJECT combine math against the AWS-published s3-tests vectors without spinning up the HTTP harness.
- Files changed:
  - `internal/s3api/checksum.go`
  - `internal/s3api/multipart.go`
  - `internal/s3api/multipart_checksum_test.go` — added `TestFullObjectCRCS3TestVectors` (locks the published `test_multipart_use_cksum_helper_crc{32,32c,64nvme}` per-part + composite values for the 5 MiB / 5 MiB / 5 MiB shape) and `TestMultipartFullObjectCRCCombine` (HTTP round-trip for all 3 CRC algos with non-uniform part sizes, asserts no `-N` suffix and ChecksumType=FULL_OBJECT on Complete + HEAD).
- Learnings for future iterations:
  - **AWS FULL_OBJECT for CRC family = `crc(concat(bytes))`, NOT `crc(concat(raw_per_part_crcs))`.** PRD AC #2 said "may be concat-then-CRC; verify against AWS docs at story start" — that's the COMPOSITE shape, not what s3-tests assert. The s3-test calls explicitly set `ChecksumType="FULL_OBJECT"` for CRC* and the published composite values have no `-N` suffix; reproducing them required the standard `crc32_combine` formula (zlib's gf2 matrix square+times pattern) and per-part sizes. This is the third PRD-vs-s3-tests divergence in this cycle (US-002 was the same pattern — trust the s3-test source).
  - **Per-part Size is required for combine math.** Don't drop `Size` from `checksumPart`. Even though the existing COMPOSITE path doesn't read it, the FULL_OBJECT branch needs `len2` for every i>0 step. Sourced from `meta.MultipartPart.Size`, already plaintext (SSE-aware in `uploadPart`).
  - **Reversed polynomial convention.** Go's `hash/crc32` and our CRC-64/NVME implementation operate on the right-shift / reflected polynomial. zlib's `crc32_combine` does too — feed `0xedb88320` (IEEE), `0x82f63b78` (Castagnoli), `crc64NVMEPoly` (NVME) into the `odd[0]` slot. Don't flip with the forward-poly form (`0x04c11db7` etc.) — combine results will be off by the reflection.
  - **SHA + FULL_OBJECT is unreachable in normal flow.** AWS rejects `ChecksumType=FULL_OBJECT` + SHA on Initiate. We fall back to COMPOSITE shape for SHA in the helper anyway, so a misbehaving client still gets a well-formed (if technically wrong-typed) response.
  - **s3-tests vector locking pattern**: when a story closes a wire-shape gap, lock the published vectors as a unit test. The 3-line `cases := []struct{...}` table with per-part b64 + sizes + expected composite is cheap; it catches polynomial / reversal / size-handling regressions without needing the live s3-tests run.
