## Codebase Patterns
- `data.Manifest` field rename + JSON migration: rename old Go field, drop the old JSON tag, and write a `func (m *Manifest) UnmarshalJSON(b []byte) error` shim that sniffs the legacy shape (`json.RawMessage` → try new struct, fall back to old type). Proto wire compat is "free" if the field number is preserved — only the label changes.
- Per-backend manifest construction lives inside `Store.CompleteMultipartUpload` in each meta backend (memory/cassandra/tikv). Shared shape goes through an exported helper in `internal/meta/store.go` (e.g. `meta.BuildPartRange`) so all three backends stay lockstep.
- After editing `internal/data/manifest.proto`, regenerate via `protoc -I internal/data --go_out=. --go_opt=module=github.com/danchupin/strata internal/data/manifest.proto` (NOT `--go_opt=paths=source_relative` — that drops the file alongside the .proto, ignoring the `manifestpb` package directory).
- Streaming-checksum on copy/transform paths: single `hash.Hash` allocated by `newChecksumHasher(algo)`, body wrapped via `io.TeeReader(reader, hasher)`, finalized to base64 after the downstream consumer (PutChunks) returns. Compare against the client-supplied `x-amz-checksum-<algo>` header only when non-empty; otherwise the server-computed digest is authoritative. Persist into `MultipartPart.Checksums[algo]` so US-001 PartRange + composite Complete carry it forward.

## 2026-05-05 - US-004 Multipart copy FlexibleChecksum path (UploadPartCopy)
- `internal/s3api/multipart.go::uploadPartCopy` now reads `x-amz-checksum-algorithm`. When set, it allocates the per-algo hasher via `newChecksumHasher`, wraps the GetChunks reader in `io.TeeReader(rc, hasher)` so PutChunks streams unchanged while the hasher fans out, then base64-encodes `hasher.Sum(nil)` once PutChunks returns.
- If the client ALSO supplied `x-amz-checksum-<algo>` (precomputed digest), the computed value is compared; mismatch deletes the freshly written manifest via `s.Data.Delete` and returns `ErrBadDigest`. Without a precomputed digest the server-computed value is used unconditionally (the boto SDK's `FlexibleChecksum` path treats this as authoritative on the response).
- Computed digest stored on `MultipartPart.Checksums[algo]` — flows through US-001's `meta.BuildPartRange` → `Manifest.PartChunks[i].ChecksumValue` (per-part response on `?partNumber=N` GET with ChecksumMode=ENABLED) and through `composeMultipartChecksums` on `CompleteMultipartUpload` (composite response shape).
- `copyPartResult` XML grew `<ChecksumCRC32>`/`<ChecksumCRC32C>`/`<ChecksumSHA1>`/`<ChecksumSHA256>`/`<ChecksumCRC64NVME>` omitempty fields. Response also gained an explicit `ETag` HTTP header — previously only present in the XML body, but `multipart_test.go::TestUploadPartCopyChecksumPersistsToCompositeOnComplete` showed Complete fails with InvalidPart when the test reads `resp.Header.Get("Etag")` and feeds an empty value into `<Part><ETag>...`. The non-copy uploadPart path already sets the header; copy just hadn't.
- New tests in `internal/s3api/multipart_copy_test.go`:
  - `TestUploadPartCopyWithChecksumAlgorithm`: SHA256 round-trip; verifies `x-amz-checksum-sha256` response header AND `<ChecksumSHA256>` XML body; expected base64 derived via the existing `partChecksum` helper.
  - `TestUploadPartCopyChecksumMismatchReturnsBadDigest`: client supplies a deliberately-wrong precomputed SHA256 → expects 400 + `<Code>BadDigest</Code>`.
  - `TestUploadPartCopyChecksumPersistsToCompositeOnComplete`: Initiate (with `x-amz-checksum-algorithm: SHA256`) → upload-part-copy → Complete; verifies the composite `<ChecksumSHA256>val-1</ChecksumSHA256>` body is present and matches the per-part digest from the upload response.
- Files changed: `internal/s3api/multipart.go`, `internal/s3api/multipart_copy_test.go` (new), `scripts/ralph/prd.json`.
- Tests: `go test ./internal/s3api/ -count=1` green. Full `go test ./... -short -count=1` green.
- **Learnings for future iterations:**
  - `uploadPartCopy` previously wrote the ETag only into the `CopyPartResult` XML body — the response HTTP `ETag` header was missing. Boto / `aws-cli` callers tolerate this for non-multipart copies but a Complete that pulls the header for the part list breaks. Set `w.Header().Set("ETag", ...)` alongside the XML body, mirroring the regular `uploadPart` path. (Pre-existing latent bug unmasked by US-004's test.)
  - Pattern for "compute server-side, optionally validate client-side": single hasher, `io.TeeReader`, finalize after PutChunks, ONLY compare-against-expected if the client header is non-empty. Reusable for any future copy/transform endpoint where the SDK auto-attaches FlexibleChecksum.
  - For copy-style endpoints the body comes from `s.Data.GetChunks(...)` — the `io.ReadCloser` must be wrapped (not replaced) so the original `defer rc.Close()` still fires. `io.TeeReader(rc, hasher)` returns an `io.Reader`; close path stays on `rc`.
  - boto SDK `FlexibleChecksumError` fires when the SDK computes its own checksum on the SOURCE data (not the copy body), expects the server to return a matching digest, and detects divergence. With server-side recomputation matching what the SDK computes from the source bytes, the integrity check passes — even though no precomputed digest was sent over the wire.

## 2026-05-05 - US-002 ?partNumber=N GET serves the exact part body
- `internal/s3api/server.go::getObject`: ?partNumber=N now resolves Manifest.PartChunks first, falls back to PartSizes for legacy rows.
- Bad N (out-of-range, non-numeric, ≤0, or ?partNumber on non-multipart) → 416 InvalidRange (matches AWS shape; PRD AC).
- ETag header echoes per-part `PartChunks[N-1].ETag` when present (whole-object ETag previously leaked through).
- x-amz-mp-parts-count drives off len(PartChunks) when present (else len(PartSizes)).
- `Range:` + `?partNumber=` is part-relative: subOff/subLen resolved against partLen via parseRange, then offset = partOff + subOff. Out-of-range subRange yields 416.
- ChecksumMode=ENABLED + partNumber merges PartChunks[i].ChecksumValue (US-001 carry-forward) into the response checksum-headers.
- Files changed: `internal/s3api/server.go`, `internal/s3api/multipart_partnumber_test.go`, `scripts/ralph/prd.json`.
- Tests: existing `TestMultipartGetByPartNumber` + `TestMultipartGetByPartNumberPerPartChecksum` still green; `TestGetByPartNumberNonMultipartReturns400`/`TestGetByPartNumberOutOfRange` retargeted to 416 (renamed `…Returns416`); new `TestMultipartGetByPartNumberRange` exercises the Range+partNumber part-relative path and the OOB-subrange 416.
- **Learnings for future iterations:**
  - AWS S3 returns `416 InvalidRange` (not `400 InvalidArgument`) for bad ?partNumber=N — including the non-multipart-object case. Match that exactly; the s3-tests upstream relies on it.
  - The Range header parser (`parseRange`) is reusable: for Range+partNumber, call it with `partLen` as the size and re-base the resulting offset onto `partOff`. Avoids re-implementing the byte-range grammar.
  - When PartChunks exists, prefer it for ETag/Offset/Size lookups over PartSizes — the latter only carries sizes, so per-part ETag would otherwise leak the whole-object ETag.
  - Single-PUT objects keep `PartChunks=nil` (US-001 contract), so `partsTotal == 0` for them — that's the signal to reject `?partNumber` with 416.
