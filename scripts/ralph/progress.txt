- **Learnings for future iterations:**
  - **Cookie-jar auth pattern for admin endpoints**: helper functions
    that create+cleanup IAM users via the Strata admin API need a
    short-lived cookie jar to retain the session JWT across the
    login → create → mint-key sequence. Pattern: `jar=$(mktemp); curl
    -c "$jar" -X POST .../auth/login -d {...}; curl -b "$jar" -X POST
    .../iam/users -d {...}; curl -b "$jar" -X POST .../iam/users/<u>/
    access-keys; rm "$jar"`. Cleanup helper must replay the login
    (cookies do not persist between bash subshells).
  - **TiKV requires non-empty values**: `txn.Set(key, nil)` and
    `txn.Set(key, []byte{})` both fail at Commit with `can not set nil
    value`. The check is on the wire, in TiKV's `txnkv/transaction/txn.go
    Set`. Use `[]byte{1}` (or any single byte) as the canonical
    "presence-only" marker for secondary-index rows. The `mockTiKV`
    used in store_test.go silently accepts nil/empty so tests don't
    catch this — pessimistic-txn semantics are a known gap (already
    documented in CLAUDE.md TiKV gotchas, but the nil-value gap was
    not yet noted; consider augmenting that section).
  - **Per-backend CredentialStore wiring is the gating contract for
    IAM**: every meta backend that wants IAM users to authenticate
    via SigV4 must (a) implement `meta.Store.GetIAMAccessKey`, (b)
    expose a `*CredentialStore{ s: *Store }` type satisfying
    `auth.CredentialsStore`, (c) be registered in
    `internal/serverapp/serverapp.go::Run` via a type-assert branch
    against `auth.NewMultiStore`. The first two existed for TiKV; the
    third was missing. Future backends (Polestar, etc.) must replicate
    all three to keep IAM functional.
  - **Strata bucket ownership is a hard ACL gate, even for the admin
    cred**: the bench's static admin AK has `Owner=owner` (from
    STRATA_STATIC_CREDENTIALS=admin:adminpass:owner); IAM users have
    `Owner=<userName>`. A bucket created by an IAM user CANNOT be
    deleted by the admin AK (Strata DeleteBucket gates on owner
    match). For bench cycles that create per-invocation IAM users,
    use a per-invocation suffix on the bucket name (`bench-iam-auth-
    <target>-<unix-ts>`) so an interrupted run does not leave an
    orphan bucket the next attempt cannot use.
  - **warp `delete` `--batch` floor**: warp aborts with
    `Too few objects: With current --batch and --concurrent settings,
    at least N objects should be used for a valid benchmark` when
    `objects < concurrent * batch * 4`. Defaults are batch=100,
    concurrent=20 → floor 8000. The PRD's 1000-object spec at conc=8
    requires batch ≤ 31. Auto-pick is the cleanest fix; operator
    override via DELETE_BATCH env if a specific shape is required.
  - **warp `get --range-size=N` op-label is `GET`** (not RANGE_GET or
    similar) — the range mode is a per-request flag on the GET path
    inside warp, not a separate op category. Existing op-anchored
    parser works as-is by passing `"GET"`.
  - **Strata's TiKV bucket_stats write-conflict storm under concurrent
    DELETE / PUT prepare** repeatedly surfaced across US-008 smoke
    iterations. The per-bucket `s/B/<bid>/bs` LWT key serialises every
    object-count + byte-count update; at concurrency ≥ 8 the
    pessimistic-retry tornado either fails outright (warp aborts) or
    backs the OSD into slow ops (ceph HEALTH_WARN). PRD's 100k seed
    @ c=8 (list) and the multipart workloads pushed the lab into
    HEALTH_WARN twice during the US-008 cycle. US-012 should surface
    as **P1** ROADMAP entry: `bucket_stats RMW under concurrent
    write load saturates the per-bucket TiKV key — every concurrent
    PUT / DELETE serialises through one LWT key; on the bench lab
    this manifested as catastrophic write-conflict floods at conc≥8.
    Move bucket_stats to additive counters or per-shard fan-out.`
  - **Lab restart-fragility under bench load**: the bench-rgw lab
    (single-OSD memstore ceph) is sized for unit-test traffic, not
    bench traffic. After multiple workload runs the ceph OSD can
    enter slow-op or down state without recovering on
    `docker compose restart ceph`. Volume wipe between US-004..US-012
    sweeps is the canonical reset; document in US-010 methodology
    page as a lab caveat (production ceph clusters would not exhibit
    this pattern with proper OSD sizing).
